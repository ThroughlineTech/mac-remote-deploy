## Description

Today the only way to pair a new phone/browser is from the Mac: the menu bar
mints a one-time token and shows a QR code on the Mac's screen. If you don't have
physical access to the Mac, you cannot enroll a second device -- even if you
already have one phone paired and working.

Add a "Pair another device" flow to the iOS companion so an **already-paired**
phone (Phone A) can mint a fresh one-time pairing token and display it as a QR
code on its own screen. A second device (Phone B) scans it with the existing,
unmodified scanner and pairs normally, getting its **own** device record and
token. This is the "one phone shares Wi-Fi credentials with another" pattern,
done so each device keeps a separate identity (independently revocable).

Crucially, this needs **no server change**: `POST /api/v1/pair/pending` is already
gated on "any authenticated paired device," not loopback-only (see Investigation),
so Phone A's existing bearer token is sufficient to mint a pending token.

### Security posture (decided with user)

Risk accepted as low for now. This means any paired device can enroll further
devices without Mac-side approval, which is a deliberate relaxation of the
"QR only shows on the Mac" trust model. We are NOT adding an approval step or
PIN in this ticket; a PIN/confirmation gate can be layered on later if we decide
it's warranted. Do surface enrolled devices clearly so the Mac user can see and
revoke them (already supported via the Devices list + `DELETE /api/v1/devices/:id`).

## Acceptance Criteria
- [ ] The companion has a "Pair another device" action (e.g. in settings /
      connection screen) available only when currently paired/connected.
- [ ] Tapping it calls the existing `APIClient.mintPairingToken()` and renders the
      returned one-time token as a scannable QR code on Phone A's screen.
- [ ] The QR encodes the exact `PairingPayload` JSON shape the existing scanner
      decodes: `{ url, token, serverName, localURL? }`, using Phone A's own server
      `url` + `serverName` and the freshly minted `token` (NOT Phone A's own token).
- [ ] A second device running the stock companion scanner pairs successfully and
      receives its own `PairedDevice` record (distinct id + token hash), visible in
      the Mac's Devices list and independently revocable.
- [ ] The on-screen QR communicates the 10-minute TTL / single-use nature (e.g. a
      "expires in ~10 min, regenerate" affordance), matching `pendingTokenTTL`.
- [ ] Graceful handling of mint failures: not-connected, 401, and the pairing
      rate-limit/lockout (429) all show a clear message rather than a blank QR.
- [ ] No server-side code changes required (confirm `/api/v1/pair/pending` still
      authorizes a normal companion bearer token end to end).
- [ ] Phone B must reach the server over HTTPS for the claim; the displayed payload
      uses the Tailscale HTTPS `url` so the claim isn't blocked (see Investigation).

## Investigation
<!-- Pre-filled from the design discussion that created this ticket. Re-verify line
     numbers during /ticket-investigate; they were accurate at creation. -->

Pairing is two steps: **mint** a one-time token, then **claim** it.

1. Mint -- `POST /api/v1/pair/pending` -> `PairingRouteHandler.mintPending`
   (`RemoteDeploy/API/Routes/PairingRouteHandler.swift:88`). It generates a fresh
   token, registers its hash as pending (10-min TTL, single-use), and returns the
   raw token in `PendingPairingResponse`.

2. The route is gated by the GENERIC auth path, not loopback:
   `RemoteDeploy/API/APIRouter.swift:150` runs `authenticate(request)` for every
   non-`/pair` route, and `AuthMiddleware.authenticate`
   (`RemoteDeploy/API/AuthMiddleware.swift:27`) accepts ANY valid bearer token by
   hashing it and looking it up in the device store -- no loopback/IP check, no
   device-name check. The handler doc comment says the caller "is the menu bar's
   loopback token," but that is descriptive, not enforced. => Phone A's existing
   token is enough to mint.

3. Client method already exists and is already linked by the companion:
   `APIClient.mintPairingToken()`
   (`Packages/RemoteDeployShared/Sources/RemoteDeployShared/API/APIClient.swift:169`,
   POSTs `/api/v1/pair/pending` with an empty body; endpoint mapping at
   `APIEndpoint.swift:58`). The menu bar uses it via
   `MenuBarClient.swift:315`.

4. Phone A already holds everything needed for the payload: Keychain stores
   `{ url, token, serverName }` (`RemoteDeployCompanion/Services/KeychainStore.swift:52`),
   surfaced on `ConnectionManager` (`serverName`, the `APIClient.baseURL`).

5. The scanner on the receiving device decodes exactly
   `PairingPayload { url, token, serverName, localURL? }`
   (`RemoteDeployCompanion/Views/Pairing/QRScannerView.swift:81`) and then calls
   `connectionManager.pair(url:token:serverName:)`
   (`RemoteDeployCompanion/Services/ConnectionManager.swift:117`). So Phone A must
   emit that exact JSON; Phone B needs NO changes.

6. HTTP/HTTPS gate: only `path == "/api/v1/pair"` (the claim) is rejected over
   plain HTTP (`RemoteDeploy/Services/HTTPHandler.swift:123`). The mint is not
   gated, but Phone B's claim must go over HTTPS -- so the payload's `url` should be
   the Tailscale HTTPS URL (Phone B on the tailnet). `localURL` (LAN HTTP) would be
   rejected for the claim and should be omitted or clearly secondary.

The only genuinely new code is on Phone A: a "show QR" screen + a QR *generator*
(the companion currently only has a scanner). QR generation is ~5 lines of
CoreImage (`CIFilter.qrCodeGenerator` / `CIQRCodeGenerator`), mirroring the host's
`RemoteDeploy/Services/QRCodeGenerator.swift`.

## Proposed Solution

Companion-only change (no server, no Phone B change):

1. Add a QR generator helper to the companion (CoreImage `CIQRCodeGenerator`),
   analogous to the host's `QRCodeGenerator`, producing a `UIImage` from a
   `PairingPayload` encoded as JSON.
2. Add a "Pair another device" view, reachable only while connected. On appear (or
   on a "Generate" tap) it calls `connectionManager`'s client `mintPairingToken()`,
   builds `PairingPayload(url: <own server url>, token: <minted>, serverName:
   <own serverName>, localURL: nil)`, and renders the QR with a TTL hint +
   "Regenerate" button.
3. Reuse the existing `PairingPayload` shape. Consider promoting `PairingPayload`
   into `RemoteDeployShared` so the host generator, companion generator, and
   companion scanner share one definition instead of three copies -- decide during
   investigate (keep scope tight if it balloons).
4. Error states: surface not-connected / 401 / 429-lockout from the mint call.

Explicitly out of scope (per user): PIN / Mac-side approval gate. Leave a short
note in code/ticket that this is an intentional deferral.

## Implementation Plan
- [x] Add a companion QR *generator* (CoreImage `CIQRCodeGenerator` -> `UIImage`),
      mirroring the host's `QRCodeGenerator`. It owns the companion's single
      `PairingPayload` definition.
- [x] Add a "Pair Another Device" view that, on appear / on Regenerate, mints a
      fresh token via `APIClient.mintPairingToken()`, builds a `PairingPayload`
      with this phone's own server URL + name and the minted token, and renders
      the QR with a TTL hint + Regenerate button.
- [x] Wire a "Pair Another Device" row into `RemoteSettingsView`, gated on
      `connectionManager.isConnected`.
- [x] Refactor the scanner (`QRScannerView`) to decode the shared
      `QRCodeGenerator.PairingPayload` instead of its own local copy, so the two
      companion sites cannot drift (2 copies -> 1 within the companion).
- [x] Surface mint failures clearly: not-connected, 401 (revoked), 429
      (rate-limit/lockout), other HTTP, and unreachable.

### Decisions
- **`PairingPayload` kept companion-local, not promoted to `RemoteDeployShared`.**
  Promoting it would pull `Packages/` into the diff and trigger a full notarized
  host release build on ship (per the `scripts/ship-deploy.sh` allowlist) for a
  deliberately companion-only feature. Instead it is defined once in the companion
  and shared by both the generator and the scanner. A code comment records this so
  promotion can be revisited if a third companion copy ever appears.
- **URL is HTTPS-only.** The payload uses the canonical Tailscale HTTPS URL
  (`https://<hostname>:<port>` from `serverStatus`, falling back to the phone's own
  `apiClient.baseURL`) and sets `localURL: nil`, because the receiving device's
  claim (`POST /api/v1/pair`) is rejected over plain HTTP. If the resolved URL is
  not HTTPS, an inline heads-up caption is shown.
- **TTL from the server, not hardcoded.** The "expires in ~N min" copy is derived
  from `PendingPairingResponse.expiresInSeconds` (currently 600s / 10 min).
- **PIN / Mac-side approval gate intentionally out of scope** (decided with user);
  noted in `PairAnotherDeviceView`'s header comment.

## Files Changed
- `RemoteDeployCompanion/Services/QRCodeGenerator.swift` (new) -- CoreImage QR
  generator producing `UIImage`; owns the companion's `PairingPayload`.
- `RemoteDeployCompanion/Views/Pairing/PairAnotherDeviceView.swift` (new) -- the
  mint-and-show-QR screen with TTL hint, Regenerate, and error handling.
- `RemoteDeployCompanion/Views/Settings/RemoteSettingsView.swift` -- adds the
  connected-only "Pair Another Device" navigation row.
- `RemoteDeployCompanion/Views/Pairing/QRScannerView.swift` -- decodes the shared
  `QRCodeGenerator.PairingPayload` instead of a local struct.
- `RemoteDeploy.xcodeproj/project.pbxproj` -- regenerated by `xcodegen generate`
  to include the two new files.

No server, shared-package, or host (`RemoteDeploy/`, `RemoteDeployServer/`) source
changes -- satisfies AC "no server-side code changes required."
