## Description
Route handlers in `RemoteDeploy/API/Routes/` are constructed inline in `configureAPIRouter()` ([RemoteDeployApp.swift:332-372](RemoteDeploy/RemoteDeployApp.swift#L332-L372)) with hardcoded closures (e.g. `schemeDetector`, `buildCanceler: { false }`, `buildHistoryProvider: { [] }`). This blocks unit testing of individual handlers. Extract closures into protocols, build a factory, allow mock injection in tests. Prerequisite/companion to TKT-003.

## Acceptance Criteria
- [x] New protocols: `SchemeDetecting`, `BuildTriggering`, `BuildCanceling`, `BuildHistoryProviding`, `BuildStatusProviding`, `StatusProviding`, `SettingsProviding`, `SettingsUpdating` (8 total — scope expanded slightly to cover all closure-based seams)
- [x] Real implementations live alongside existing services under `RemoteDeploy/Services/API/`
- [x] `configureAPIRouter()` delegates to a new `APIRouterFactory` that accepts protocol-typed dependencies
- [x] Each route handler accepts only protocol-typed dependencies (no more `@Sendable` closure parameters)
- [x] Mocks added under `RemoteDeployTests/Mocks/` including the missing `MockPairedDeviceStore`
- [x] Smoke test `APIRouterFactoryTests` builds a router from mocks and dispatches at least one request on each of: status, pairing, projects, build, installs, devices, settings, filesystem
- [x] Existing tests still pass (`xcodebuild test` green)
- [x] `xcodebuild build` clean, no new warnings

## Investigation

### Current state — what's already injectable
Four of the nine route handlers **already take protocol types** and are unit-testable today:
- [PairingRouteHandler.swift:22](RemoteDeploy/API/Routes/PairingRouteHandler.swift#L22) — `init(deviceStore: PairedDeviceStoring, ...)`
- [ProjectsRouteHandler.swift:14](RemoteDeploy/API/Routes/ProjectsRouteHandler.swift#L14) — `init(projectStore: ProjectStoring)`
- [InstallsRouteHandler.swift:14](RemoteDeploy/API/Routes/InstallsRouteHandler.swift#L14) — `init(installTracker: InstallTracking)`
- [DevicesRouteHandler.swift:15](RemoteDeploy/API/Routes/DevicesRouteHandler.swift#L15) — `init(deviceStore: PairedDeviceStoring)`
- [AuthMiddleware.swift:16](RemoteDeploy/API/AuthMiddleware.swift#L16) — `init(deviceStore: PairedDeviceStoring)`

### Current state — what's NOT injectable (the real scope)
The other four route handlers take `@Sendable` **closures**, not protocols. That's technically injectable (tests can pass stub closures) but comes with three real problems:

1. The closures are built inline in [RemoteDeployApp.swift:280-389](RemoteDeploy/RemoteDeployApp.swift#L280-L389) — 110 lines of wiring. No seam for tests to reuse.
2. The `schemeDetector` closure executes `Process()` → `/usr/bin/xcodebuild` inline ([RemoteDeployApp.swift:332-357](RemoteDeploy/RemoteDeployApp.swift#L332-L357)). Any test that touches `FilesystemRouteHandler.detectSchemes` either shells out for real or has to reproduce that closure setup.
3. Stub closures can't easily record calls or mutate state (no class identity, Sendable capture limits). Class-based mocks are the codebase pattern — see [MockProjectStore.swift](RemoteDeployTests/Mocks/MockProjectStore.swift) with its `saveCallCount`, `lastSavedProject`, `saveShouldThrow`.

The four closure-based handlers:
- [BuildRouteHandler.swift:27-37](RemoteDeploy/API/Routes/BuildRouteHandler.swift#L27-L37) — 4 closures: `buildTrigger`, `buildStatusProvider`, `buildCanceler`, `buildHistoryProvider`
- [StatusRouteHandler.swift:17](RemoteDeploy/API/Routes/StatusRouteHandler.swift#L17) — 1 closure: `statusProvider`
- [SettingsRouteHandler.swift:19-22](RemoteDeploy/API/Routes/SettingsRouteHandler.swift#L19-L22) — 2 closures: `settingsProvider`, `settingsUpdater`
- [FilesystemRouteHandler.swift:17](RemoteDeploy/API/Routes/FilesystemRouteHandler.swift#L17) — 1 closure: `schemeDetector`

### Current state — the wiring
[RemoteDeployApp.configureAPIRouter()](RemoteDeploy/RemoteDeployApp.swift#L280-L389) is `@MainActor`, captures `AppStateBridge` ([line 523](RemoteDeploy/RemoteDeployApp.swift#L523)) for thread-safe AppState reads from NIO's event loop, and hand-builds every handler in a single function. Notable quirks it must preserve:
- `buildTrigger` posts `.apiBuildRequested` notification on main queue (the MenuBarView listens and runs the same flow as the UI button)
- `buildCanceler` is a no-op stub returning `false` (real impl is TKT-016)
- `buildHistoryProvider` returns `[]` (real impl is TKT-008)
- `settingsUpdater` is a no-op stub returning `nil` (real impl is TKT-009)
- `nioServer.apiRouter = router` side-effect at [line 385](RemoteDeploy/RemoteDeployApp.swift#L385)
- `services.pairingHandler = pairingHandler` side-effect at [line 388](RemoteDeploy/RemoteDeployApp.swift#L388) — needed so [PairDeviceView](RemoteDeploy/Views/PairDeviceView.swift) can call `registerPendingToken()`

### Existing protocol/mock conventions to follow
- Protocols live in [RemoteDeploy/Protocols/](RemoteDeploy/Protocols/) with one protocol per file, no `API/` subdirectory yet
- Mocks live in [RemoteDeployTests/Mocks/](RemoteDeployTests/Mocks/): class-based, `@unchecked Sendable`, `callCount` + `last…` + `…ShouldThrow` pattern (see [MockProjectStore](RemoteDeployTests/Mocks/MockProjectStore.swift))
- `MockPairedDeviceStore` does **not** exist yet — will be needed by this ticket and TKT-003

### Regression Risk — **MEDIUM**
The refactor touches the construction path of every API request. Behavior must be byte-identical. Main risks:

1. **Sendable/actor isolation drift.** Protocols need `Sendable` conformance; adapters capturing `AppStateBridge` must remain safely callable from NIO threads. Mitigation: make every new protocol require `: Sendable`, mirror the existing `@unchecked Sendable` pattern on adapter classes.
2. **MainActor capture.** `configureAPIRouter()` is `@MainActor` because it reads AppState live. The factory must also be `@MainActor` at construction time, even though the router it produces runs on NIO threads.
3. **Notification flow for build trigger.** If the new `NotificationBuildTrigger` drops the `DispatchQueue.main.async` hop, UI-thread listeners could miss the event. Mitigation: verbatim copy of the current closure body.
4. **AppStateBridge timing.** The bridge captures `appState` via `nonisolated(unsafe)` at [line 530](RemoteDeploy/RemoteDeployApp.swift#L530). Adapters should consume the existing bridge rather than re-capturing AppState.
5. **No existing test coverage for the API surface** ([HTTPServerIntegrationTests](RemoteDeployIntegrationTests/HTTPServerIntegrationTests.swift) exercises OTA serving but never hits `/api/`). A regression in the factory would silently compile and fail at runtime. Mitigation: ship a minimal `APIRouterFactoryTests` in this same ticket (5-10 smoke tests covering each handler's happiest path) so TKT-004 lands with its own safety net. TKT-003 then builds out comprehensive coverage.

## Proposed Solution

### Approach
Introduce **eight tiny one-method protocols**, each with a real adapter implementation and a test mock. Extract `configureAPIRouter()` into an `APIRouterFactory` that accepts protocol-typed dependencies and returns a constructed `APIRouter`. `RemoteDeployApp` shrinks its wiring function to ~20 lines that build real adapters and call the factory.

Each protocol is trivial (one method, `Sendable`), so the file count is high but the cognitive load per file is near zero. This matches the codebase's existing "one protocol per file" convention.

### Why protocols over keeping closures
The ticket acceptance criteria already specify protocols, but the concrete wins:
1. **Class-based mocks.** Matches [MockProjectStore](RemoteDeployTests/Mocks/MockProjectStore.swift) pattern — `callCount`, `lastCalledWith`, `shouldThrow` — which closures make awkward.
2. **Named semantics at the use-site.** `BuildTriggering` reads at a glance; `@Sendable (UUID, String?) -> String?` does not.
3. **Future-proofing.** TKT-008 (build history), TKT-009 (settings update), TKT-016 (build cancel) each need to replace a stub with a real implementation. Swapping the conforming type at the factory call-site is cleaner than replacing a closure body in a god function.

### Tradeoffs
- **File count.** ~16 new production files (8 protocols, 8 adapters). Small files, but many.
- **Adapter boilerplate.** Each adapter is 5-10 lines wrapping a closure or calling into `AppStateBridge`. Accepted — it's the cost of the test seam.
- **No behavior change this ticket.** The stubs (`NoopBuildCanceler`, `NoopBuildHistoryProvider`, `DeferredSettingsUpdater`) preserve today's no-op semantics. Real implementations are explicitly deferred to TKT-008/009/016.
