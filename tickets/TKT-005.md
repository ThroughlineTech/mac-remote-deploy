### Existing Logger usage

Three other files import `os` but they're using `OSAllocatedUnfairLock`, not `Logger`:

- `RemoteDeploy/Services/XcodeBuildEngine.swift`
- `RemoteDeploy/Services/NIODeployServer.swift`

So the audit's claim "iOS companion already uses os.Logger in some places" is technically true but is one file with one private logger instance.

### What's NOT being logged that should be (per acceptance criteria)
- **HTTPHandler request logging.** [HTTPHandler.swift](RemoteDeploy/Services/HTTPHandler.swift) processes every API and OTA request but logs nothing. AC: log method, path, status, duration. Need to capture start time at the head of `handleRequest`/`channelRead` and log on response. Care needed because responses come from multiple `send*` paths.
- **AuthMiddleware failed-auth logging.** [AuthMiddleware.swift](RemoteDeploy/API/AuthMiddleware.swift) silently returns nil on bad bearer tokens. AC: log failed attempts. Source IP isn't available at this layer (it has only the headers list); log the path instead.

### Existing tests
- Existing tests don't capture stdout, so converting `print()` → `Logger` doesn't break any test.
- The 249-test baseline includes 11 integration tests that exercise HTTPHandler. They don't assert on logs but will catch regressions if my logging additions break the request path (e.g. timing capture racing with the response writers).

### Privacy annotations
`os.Logger` interpolation supports `\(value, privacy: .private|.public|.sensitive)`. By default, custom values are `.private` in release builds (auto-redacted in Console). Per the security audit, what's sensitive vs public:

| Type | Privacy |
|---|---|
| User home paths, project paths, cert paths, hostnames, IPs, push tokens | `.private` |
| HTTP method, response status code, milliseconds, slug, error category name | `.public` |
| Project names, device names | `.private` (could leak personal info) |
| Bundle IDs | `.public` (typically `com.example.app`, not sensitive) |

### Regression Risk — **LOW**
Pure observability swap:
- Log calls cannot affect program control flow.
- HTTPHandler timing capture is the only non-trivial change. If I implement it wrong (e.g. forget to log on one of the early-return paths), I miss logs but don't break the request.
- No test changes expected — zero tests assert on stdout.
- Both targets must build clean. iOS gets its own logging file in `RemoteDeployCompanion/`.

Risk vectors:
1. **Forgotten print() in a code path I didn't see.** Mitigation: final `grep` before commit confirms zero `print(` matches.
2. **Privacy annotation slip-up.** Logging a path or hostname as `.public` would leak it in Console for users with debug-level logging on. Mitigation: default to `.private` and only mark public when sure.
3. **HTTPHandler request logging double-firing.** If I log inside `handleRequest` *and* inside the response writers, requests get logged twice. Mitigation: log only at the response-writer layer with the start time captured at request entry, OR log only at request entry and have the writers update no state. I'll choose the cleanest: capture start time on request entry, log a single line right before each write call's `flush`.
4. **Performance.** `os.Logger` is significantly faster than `print()` and is async at the kernel boundary. No performance regression expected.

## Proposed Solution

### Approach: subsystem-keyed Logger extensions, mechanical print() replacement

**Two new logging files** — one per target — define `Logger` extensions with named static categories. Every existing `print()` call becomes `Logger.<category>.<level>("...")` with appropriate privacy annotations. HTTPHandler gets new request-logging code; AuthMiddleware gets failed-auth logging; everything else is a 1:1 swap.

### Subsystems

**macOS host** — `com.remotedeploy.host`:
| Category | Used by |
|---|---|
| `server` | NIODeployServer lifecycle, BonjourAdvertiser |
| `api` | HTTPHandler request logging, APIRouter |
| `pairing` | PairingRouteHandler, AuthMiddleware |
| `build` | XcodeBuildEngine, IPAImporter, build flow in MenuBarView |
| `tailscale` | CLITailscaleProvider, Tailscale-related code in RemoteDeployApp/SettingsView |

| `notifications` | NotificationManager, push notifier failures |

|---|---|

| `ui` | (future use) |

### Logger extension pattern

```swift
// RemoteDeploy/Logging.swift
import os

extension Logger {
    private static let hostSubsystem = "com.remotedeploy.host"
    static let server = Logger(subsystem: hostSubsystem, category: "server")
    static let api = Logger(subsystem: hostSubsystem, category: "api")
    static let pairing = Logger(subsystem: hostSubsystem, category: "pairing")
    // ...
}
```

Then call sites use `Logger.api.info("Request \(method, privacy: .public) \(path, privacy: .private)")`.

### HTTPHandler request logging

In `handleRequest`, capture `let start = ContinuousClock.now` at the top. Wrap each `sendDataResponse`/`sendResponse`/`sendFileResponse`/`sendNotFound` call with a logging side-effect. Cleanest: a single helper method `logRequest(method:path:status:start:)` called at the end of each terminal `send*` block.

Even cleaner: wrap the response writers in `HTTPHandler.swift` to log automatically. But that breaks the boundary between HTTPHandler.swift (which has the writers post-TKT-015) and NIOResponseGenerator.swift (which has the response generators that call them). Simpler: inject the logger call just before each `send*` site in `handleRequest`. ~5 sites.

Even simpler: log once in `handleRequest` at entry (incoming), once at exit (outgoing) using a `defer` block that captures the resolved status. Problem: status isn't resolved until the response is built. Defer can't read the response.

**Final decision:** instrument the response writers. Add a `responseStatusForLogging` private var on HTTPHandler that the writers set just before flushing. A `defer` in `handleRequest` reads it after the switch arms have run. ~10 lines of new code in HTTPHandler.swift.

### AuthMiddleware failed-auth logging

Single line addition: log when `authenticate(headers:)` returns nil. Don't log the token (obviously). Log the failure reason: missing header / malformed prefix / unknown token. Caller passes the path? It doesn't have the path — the router has it. **Move the failed-auth log to APIRouter.handle()** where we have both the path and the auth result. Cleaner.

### README update

Append a "Logging" section to README.md showing:
```
log stream --subsystem com.remotedeploy.host --level debug
log stream --subsystem com.remotedeploy.host --predicate 'category == "api"'
```
Plus a note about Console.app's "Process > RemoteDeploy" filter.

### Tradeoffs
- **One log per category, not per file.** Some files will use multiple categories (RemoteDeployApp uses storage, tailscale, server, notifications). Acceptable — categories are by domain, not by source location.
- **No logging in tests.** Tests don't assert on log output. If a future ticket needs to verify logging, OSLogStore queries are possible but heavy. Out of scope.
- **HTTPHandler timing precision.** ContinuousClock gives nanosecond precision. We'll log milliseconds which is more than enough.
