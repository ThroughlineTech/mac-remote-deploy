## Description

After a macOS project build succeeds (TKT-051), RemoteDeploy currently just serves the `.app.zip` for download — the user has to manually download, extract, and launch. For local-machine deploys, this should be automatic: build succeeds → quit the running target app → replace it with the new build → launch it.

The infrastructure already exists:
- TKT-051 added macOS build support (archive → zip `.app`)
- TKT-050 added `scripts/graceful-relaunch.sh` (graceful quit → port wait → relaunch)
- `applicationWillTerminate` in AppDelegate cleans up NIO server and Bonjour before exit

What's needed is a post-build hook for macOS projects that wires these together. When a macOS project has "local deploy" enabled, after a successful build:
1. Extract the `.app` from the archive (or the zip)
2. Run the graceful-relaunch flow on the target app (quit existing, wait for port release)
3. Copy the new `.app` to the target location (e.g., `/Applications/` or a configured path)
4. Launch the new `.app`

This should work for:
- **ClaudeDash**: build on this Mac, deploy to this Mac
- **RemoteDeploy self-deploy**: build itself, quit itself, launch the new version
- **Any macOS app**: generic pattern for local iteration


## Acceptance Criteria

- [ ] macOS projects have an optional "local deploy" toggle (or it's the default behavior for macOS builds on the same machine)
- [ ] After a successful macOS build, the target app is gracefully quit, replaced, and relaunched automatically
- [ ] Uses the graceful-relaunch.sh flow (AppleScript quit → timeout → force-kill → port wait)
- [ ] Configurable target path for the deployed `.app` (default: alongside the build output, or `/Applications/`)
- [ ] Works for RemoteDeploy deploying itself (self-deploy scenario)
- [ ] Push notification says "deployed" not just "ready to download" for local deploys
- [ ] Falls back gracefully if the target app isn't running (just launch the new one)

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### macOS build pipeline end-to-end

1. **BuildManager.triggerBuild()** (`RemoteDeploy/Managers/BuildManager.swift:107-234`) is the orchestrator. It calls `buildEngine.build(project:)`, then on success: registers the project with the deploy server, starts the server if needed, fires local notifications (`notifyBuildSuccess`) and push notifications (`sendPushNotification`).

2. **BuildEngineRouter** (`RemoteDeploy/Services/BuildEngineRouter.swift`) dispatches to `XcodeBuildEngine` or `ExpoBuildEngine` based on `project.projectType`.

3. **XcodeBuildEngine.build()** (`RemoteDeploy/Services/XcodeBuildEngine.swift:79-218`) branches on platform at line 168:
   - **iOS path**: archive → exportArchive → copy IPA to `~/Library/Application Support/RemoteDeploy/serve/<slug>/app.ipa`
   - **macOS path** (line 168-176): archive → `zipAppBundle()` → writes `app.zip` to `~/Library/Application Support/RemoteDeploy/serve/<slug>/app.zip`. No exportArchive step (macOS apps don't use IPA).

4. **zipAppBundle()** (`XcodeBuildEngine.swift:573-609`): locates the `.app` bundle inside the archive at `<archive>/Products/Applications/`, uses `ditto -ck --keepParent` to create `app.zip`, returns the zip path. The `.app` bundle path is `<archivePath>/Products/Applications/<AppName>.app`.

5. **Serve flow**: `NIOResponseGenerator.swift` serves `/<slug>/` as a download page for macOS (calls `installPageGenerator.generateDownloadPage()`), and `/<slug>/app.zip` as the zip file. The `HTTPHandler` routes `app.zip` requests at line 185-186.

6. **After build succeeds** in BuildManager (line 154-201): the returned path (`ipaPath` — poorly named for macOS, it's actually the zip path) is stored in `BuildStatus.success(ipaPath:)`. Then notifications fire with "ready to install" messaging. This is where the local deploy hook needs to go.

### graceful-relaunch.sh analysis

`scripts/graceful-relaunch.sh` (102 lines):
- Takes `<AppName>`, optional `--port PORT`, `--timeout SECS`, `--no-relaunch`
- Flow: pgrep for app → AppleScript `quit` → poll 0.5s intervals up to timeout → force-kill if needed → wait for port release → `open -a <AppName>` to relaunch
- Already handles "app not running" gracefully (exits 0 with message)
- The `--no-relaunch` flag is key: we want to quit the old app, copy the new one, THEN launch — so we'd use `--no-relaunch` and handle launch ourselves after the copy

### Where the local deploy hook integrates

The hook goes in **BuildManager.triggerBuild()** between the successful `build()` return (line 154) and the notification block (line 190). Specifically, after `let ipaPath = try await buildEngine.build(project: project)` succeeds and before the notifications fire.

For macOS + localDeploy projects, the flow becomes:
1. `build()` returns the zip path
2. **NEW**: Extract `.app` from archive (or unzip the zip) → copy to target path → graceful quit old app → launch new app
3. Notifications say "deployed" instead of "ready to install"

### Shell script vs native Swift

**Shell script approach**: Call `graceful-relaunch.sh` via `Process`. Pros: already works, tested. Cons: requires script to be at a known path (bundled in the app or at the repo root); fragile for distributed builds; another process spawn.

**Native Swift approach**: Use `NSRunningApplication` APIs + `Process` for `open`. Pros: no external dependency, cleaner for a macOS app, better error handling. Cons: more code to write.

**Recommendation: Native Swift.** The graceful-relaunch logic is ~30 lines of Swift:
- `NSRunningApplication.runningApplications(withBundleIdentifier:)` to find the target
- `.terminate()` (sends Apple Event quit, same as AppleScript)
- Poll `.isTerminated` with a timer, `.forceTerminate()` on timeout
- `FileManager.copyItem` / `replaceItemAt` to deploy the `.app`
- `NSWorkspace.shared.openApplication(at:configuration:)` to launch

This avoids bundling the shell script and is more robust for the self-deploy scenario where RemoteDeploy quits itself (the Swift code can set up a dispatch_after or trampoline process before calling terminate on itself).

### Self-deploy scenario

When RemoteDeploy deploys itself, it must:
1. Copy the new `.app` to the target path
2. Spawn a trampoline process (a tiny shell `Process` that waits for the PID to die, then runs `open`)
3. Call `NSApplication.shared.terminate(nil)` on itself
4. The trampoline detects process exit and launches the new app

`AppDelegate.applicationWillTerminate` (line 146-157) already stops NIO and Bonjour synchronously, so port release is handled.

### ProjectConfig changes needed

`ProjectConfig` (`Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/ProjectConfig.swift`) needs:
- `localDeploy: Bool` — whether to auto-deploy after build (default `false`)
- `localDeployPath: String?` — target directory for the `.app` (default: `/Applications/`). Nil means use the default.

These go in the shared package since the companion/PWA might want to display deploy status. The backward-compatible decoder already handles new optional fields gracefully (line 78-95 pattern: `decodeIfPresent` with defaults).

### Regression risks

- **Low risk**: ProjectConfig changes are additive (new optional fields with defaults). Existing saved configs decode fine via `decodeIfPresent`.
- **Medium risk**: BuildManager changes are in the success path. A bug in the deploy logic could prevent notifications from firing or cause the build to appear failed when it succeeded. Mitigation: wrap the deploy step in do/catch so deploy failures don't mask build success.
- **Medium risk**: Self-deploy scenario. If the trampoline fails, RemoteDeploy quits and doesn't relaunch. Mitigation: only use the self-deploy path when the target app's bundle ID matches RemoteDeploy's own bundle ID; test thoroughly.
- **Low risk**: The zip/app extraction is straightforward FileManager work.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

### Approach

Add a `LocalDeployManager` service that handles the post-build deploy flow for macOS projects. Wire it into `BuildManager.triggerBuild()` as a post-build hook. Add `localDeploy` and `localDeployPath` fields to `ProjectConfig`.

**Key design decisions:**

1. **Native Swift implementation** — no dependency on the shell script at runtime. The shell script remains useful for preview mode and external tooling, but the in-app deploy uses `NSRunningApplication` + `NSWorkspace` for cleaner integration and better error reporting.

2. **Extract `.app` from the archive**, not from the zip. The archive at `/tmp/RemoteDeploy/<name>.xcarchive/Products/Applications/<name>.app` is already present after the build. No need to unzip the zip we just created. This is both faster and avoids a round-trip through ditto.

3. **Self-deploy via trampoline process.** When the target app's bundle identifier matches `Bundle.main.bundleIdentifier`, spawn a background shell process (`/bin/sh -c "while kill -0 $PID 2>/dev/null; do sleep 0.2; done; open <appPath>"`) before calling `NSApp.terminate(nil)`. The trampoline outlives the parent process.

4. **Deploy failures are non-fatal to the build.** If the deploy step fails (copy error, app won't quit), the build is still marked successful and served for download. The deploy error is logged and included in the notification. This prevents a flaky deploy from hiding a successful build.

5. **Notification messaging changes.** For local-deploy projects, push notification says "deployed to /Applications/" instead of "ready to install". The install URL is still included for remote access.


## Implementation Plan
<!-- Agent fills this during /ticket-investigate -->
- [ ] 1. **Add `localDeploy` fields to ProjectConfig.** In `Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/ProjectConfig.swift`: add `public var localDeploy: Bool` (default `false`) and `public var localDeployPath: String?` (default `nil`, meaning `/Applications/`). Update `init(name:projectPath:)` to set defaults. Update `init(from:)` to decode with `decodeIfPresent` fallbacks.
- [ ] 2. **Create `LocalDeployManager` service.** New file `RemoteDeploy/Services/LocalDeployManager.swift`. A `@MainActor` class with method `func deploy(appBundleName: String, fromArchive archivePath: String, toDirectory targetDir: String, port: Int?) async throws`. Implementation:
    - Find target app via `NSRunningApplication.runningApplications(withBundleIdentifier:)` or fall back to `NSWorkspace` process list by name.
    - If running: call `.terminate()`, poll `.isTerminated` at 0.5s intervals up to 5s, then `.forceTerminate()` if needed.
    - If `port` is provided, poll `lsof -i :<port>` (or `Darwin.bind` probe) until free, up to 5s.
    - Copy `.app` from `<archivePath>/Products/Applications/<name>.app` to `<targetDir>/<name>.app` using `FileManager.default.replaceItemAt(_:withItemAt:)` for atomic replacement.
    - Detect self-deploy: if the target bundle ID matches `Bundle.main.bundleIdentifier`, use the trampoline pattern (spawn shell process that waits for PID death then calls `open`), then call `NSApp.terminate(nil)`. Return early — the trampoline handles launch.
    - Normal case: launch via `NSWorkspace.shared.openApplication(at: URL(fileURLWithPath: "<targetDir>/<name>.app"), configuration: NSWorkspace.OpenConfiguration())`.
- [ ] 3. **Create `LocalDeployManagerProtocol`.** New file `RemoteDeploy/Protocols/LocalDeployManagerProtocol.swift` with the `deploy(...)` method signature. `LocalDeployManager` conforms to it. This enables testing with a mock.
- [ ] 4. **Wire into BuildManager.** In `RemoteDeploy/Managers/BuildManager.swift`:
    - Add `private var localDeployManager: (any LocalDeployManagerProtocol)?` dependency.
    - Add it to `configure(...)` parameter list.
    - In `triggerBuild()`, after the successful `build()` call (line ~154) and before notifications (line ~190): if `project.platform.lowercased() == "macos" && project.localDeploy`, call `localDeployManager.deploy(...)` wrapped in a do/catch. On success, change the notification message to "deployed". On failure, log the error and append it to the build log, but keep the build status as success.
    - Compute `archivePath` from the same pattern as `XcodeBuildEngine`: `/tmp/RemoteDeploy/<sanitized_name>.xcarchive`.
    - Compute `targetDir` from `project.localDeployPath ?? "/Applications"`.
- [ ] 5. **Update AppDelegate to wire LocalDeployManager.** In `RemoteDeploy/AppDelegate.swift`, `performStartup()` (line ~186): pass the new `LocalDeployManager()` instance to `buildManager.configure(...)`.

- [ ] 7. **Update notification messaging.** In `BuildManager.triggerBuild()`, after a successful local deploy:
    - Local notification: `notifyBuildSuccess` with message "deployed to <path>" instead of install URL.
    - Push notification: title "Build Deployed", message "<project.name> deployed to <targetDir>", priority normal, URL still the install URL as fallback.
