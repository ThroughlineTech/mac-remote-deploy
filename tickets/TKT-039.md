## Description

Clicking "+ Add Project..." in the menu bar popover opens the Settings window, but it lands on the **Server** tab (the default tab) — not the **Projects** tab where you'd actually add a project. The user has to manually click the "Projects" tab to get where they intended to go. Sloppy.

Screenshot confirms: "Add Project..." was clicked in the popover → Settings window opened to "Server" tab with port/hostname/cert fields showing.

### Root cause

[RemoteDeploy/Views/MenuBar/ProjectsListSection.swift:33-40](RemoteDeploy/Views/MenuBar/ProjectsListSection.swift#L33-L40):

```swift
SettingsLink {
    Label("Add Project...", systemImage: "plus")
    ...
}
.buttonStyle(.plain)
.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })
```

`SettingsLink` is SwiftUI's built-in control for opening the Settings window. It opens Settings to **whatever tab it was last on** (or the first tab if freshly opened). There's no way to tell `SettingsLink` which tab to select — it's a limitation of SwiftUI's `Settings { }` scene API.

### Fix options

**A) Replace `SettingsLink` with a custom button** that opens the Settings window AND programmatically selects the Projects tab. This requires either:
- An `@AppStorage` or `@Binding` for the selected tab in the Settings `TabView`, which the "Add Project" button sets before opening the window.
- Or `openWindow(id:)` on a separate "Add Project" window (the Setup Assistant's project step, or a standalone add-project sheet).

**B) Navigate to the Setup Assistant's project step instead.** The Setup Assistant already has a project-add flow (Step 3: ProjectSetupStep). Clicking "Add Project..." could open the Setup Assistant directly to the project step, which is arguably a better UX than the Settings Projects tab anyway.

**C) Use `NSApp.sendAction(Selector(("showSettingsWindow:")), to: nil, from: nil)`** + programmatically switch tabs via the TabView selection binding. This is the most direct fix but requires exposing the Settings tab selection as shared state.

**Recommendation:** Option A with a `@SceneStorage` or `@AppStorage("selectedSettingsTab")` binding. The Settings TabView binds its selection to this value. The "Add Project" button sets it to `"projects"` before triggering the Settings window open.

## Reproduction Steps
1. Open the menu bar popover.
2. Click "+ Add Project..."
3. The Settings window opens — to the Server tab, not the Projects tab.
4. Expected: Settings opens directly to the Projects tab.

## Investigation

### Root cause

The `SettingsLink` in `RemoteDeploy/Views/MenuBar/ProjectsListSection.swift:33-40` opens the Settings window but has no way to control which tab is selected. SwiftUI's `SettingsLink` simply opens the `Settings {}` scene — it always lands on whichever tab was last active, or the first tab (Server) if the window hasn't been opened yet.

The `TabView` in `RemoteDeploy/Views/SettingsView.swift:16-52` uses an uncontrolled `TabView` with no `selection` binding. Tabs are: Server, Projects, Notifications, Devices, General, About (in that order). Since Server is first and there's no selection state, it's always the default.

The same `SettingsLink` pattern is used in `RemoteDeploy/Views/MenuBar/UtilitiesSection.swift:30-37` for the generic "Settings..." button — that one should continue to open to whatever tab was last viewed, which is the current (correct) behavior.

### Key files examined

- `RemoteDeploy/Views/MenuBar/ProjectsListSection.swift:33-40` — "Add Project..." uses `SettingsLink` with no tab targeting.
- `RemoteDeploy/Views/MenuBar/UtilitiesSection.swift:30-37` — "Settings..." also uses `SettingsLink`; should keep default behavior.
- `RemoteDeploy/Views/SettingsView.swift:16-52` — `TabView` has no `selection` binding. Defines 6 tabs.
- `RemoteDeploy/Models/AppState.swift` — No existing tab selection property.
- `RemoteDeploy/RemoteDeployApp.swift:56-61` — `Settings { SettingsView() }` scene, no tab control.

### Window activation

TKT-033 adds `orderFrontRegardless()` to the Settings `simultaneousGesture` in `UtilitiesSection.swift:37`. The same activation pattern should be applied to the "Add Project..." button in `ProjectsListSection.swift`. However, replacing `SettingsLink` with a `Button` means we won't use `simultaneousGesture` — we'll call `NSApp.activate()` and `orderFrontRegardless()` directly in the button action.

### Regression risk: **Low**

The change adds a `@Published` property to `AppState` and a `selection` binding to the `TabView`. The "Settings..." button in `UtilitiesSection` is untouched — it still uses `SettingsLink` which opens to the last-viewed tab. Only the "Add Project..." button changes behavior.

## Proposed Solution

**Option A from the ticket description** — replace `SettingsLink` with a `Button` that:

1. Sets `appState.selectedSettingsTab` to `"projects"`.
2. Opens the Settings window via `NSApp.sendAction(Selector(("showSettingsWindow:")), to: nil, from: nil)`.
3. Activates the app with `NSApp.activate()` (and `orderFrontRegardless()` per TKT-033 pattern).

The `SettingsView` `TabView` gains a `selection` binding to `appState.selectedSettingsTab`. Each `.tabItem` gets a `.tag()` modifier. This is the standard SwiftUI pattern for programmatic tab control.

The "Settings..." button in `UtilitiesSection` stays as `SettingsLink` — it opens to whatever tab was last active (the existing correct behavior), because the `selection` binding preserves the last-viewed tab.
