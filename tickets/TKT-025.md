## Description

At runtime the Xcode console logs repeated `Picker: the selection ... is invalid and does not have an associated tag, this will give undefined results` warnings. Two distinct variants were observed during a TKT-024 smoke-launch on 2026-04-08:

```
Picker: the selection "Optional(AC17ABA0-0934-455F-A820-EB9230B271B5)" is invalid and does not have an associated tag, this will give undefined results.
...
Picker: the selection "" is invalid and does not have an associated tag, this will give undefined results.
```

These are separate from TKT-021 (layout recursion / Bonjour chatter) — SwiftUI is telling us at least two pickers have a selection value that doesn't match any child `.tag(...)` at render time.

### Suspect 1 — Project picker in `BuildControlsSection`

[RemoteDeploy/Views/MenuBar/BuildControlsSection.swift:16-23](RemoteDeploy/Views/MenuBar/BuildControlsSection.swift#L16-L23):

```swift
Picker("Project:", selection: $appState.selectedProjectID) {
    ForEach(appState.projects) { project in
        Text(project.name).tag(Optional(project.id))
    }
}
```

The selection binding is `UUID?` and the tags are `UUID?`, which looks correct — but SwiftUI still warns. Likely race: `appState.selectedProjectID` holds a UUID from disk that does not match any current `appState.projects` entry during an early body evaluation (load order, or after a project deletion). The `Optional(AC17…)` variant of the warning matches this shape.

The picker only renders when `projects.count > 1`, so single-project setups will not see this warning.

### Suspect 2 — Scheme picker in `ProjectSetupStep`

[RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift):

```swift
Picker("Scheme:", selection: $selectedScheme) {
    ForEach(detectedSchemes, id: \.self) { scheme in
        Text(scheme).tag(scheme)
    }
}
```

`@State private var selectedScheme: String = ""` — if the picker is evaluated while `selectedScheme` is still the empty initial value and `detectedSchemes` contains no empty string, no tag matches. The `""` variant of the warning matches this shape. Note: TKT-024 Commit 3 added a `schemeError` inline caption for empty schemes, but that doesn't actually fix the warning — the picker itself still has a bad binding during the "no scheme selected yet" window.

## Reproduction Steps
1. Launch RemoteDeploy host app with at least two projects configured.
2. Open the menu bar popover and watch the Xcode console.
3. Observe the `Picker: ... invalid` warnings, both `Optional(UUID)` and `""` variants.
4. Open the Setup Assistant, navigate to the Project step, select a project — observe additional `""` variant warnings from the scheme picker during detection.

- [ ] No `Picker: the selection ... is invalid` warnings logged during Setup Assistant project selection flow

## Investigation

Both picker warnings are caused by the same root condition: the picker's selection binding holds a value that doesn't match any of the current tags. SwiftUI falls back to "undefined results" and logs the warning. Two distinct triggers, two distinct files.

### Picker 1 — Project picker (`BuildControlsSection`)

[RemoteDeploy/Views/MenuBar/BuildControlsSection.swift:15-23](RemoteDeploy/Views/MenuBar/BuildControlsSection.swift#L15-L23):

```swift
if appState.projects.count > 1 {
    Picker("Project:", selection: $appState.selectedProjectID) {
        ForEach(appState.projects) { project in
            Text(project.name).tag(Optional(project.id))
        }
    }
    .pickerStyle(.menu)
}
```

The binding is `Binding<UUID?>` and the tags are `Optional<UUID>` — types match on paper. The warning with the form `Optional(AC17ABA0-0934-455F-A820-EB9230B271B5)` fires when `appState.selectedProjectID` holds a UUID that no current project has.

**How the mismatch can happen:**

- `selectedProjectID` persists in memory as a `@Published` property on `AppState` with no disk persistence. On cold launch it starts as `nil`, then [AppDelegate.loadSavedProjects()](RemoteDeploy/AppDelegate.swift#L187-L200) sets it to `projects.first?.id`.
- Within `loadSavedProjects`, the sequence is `appState.projects = projects` followed by `appState.selectedProjectID = first.id`. Both are sync `@Published` writes on the main actor, so SwiftUI batches the view update — no intra-runloop mismatch.
- The real trigger is **`ProjectsListSection.removeProject(_:)`** at [ProjectsListSection.swift:46-54](RemoteDeploy/Views/MenuBar/ProjectsListSection.swift#L46-L54):
  ```swift
  appState.projects.removeAll { $0.id == project.id }
  ...
  if appState.selectedProjectID == project.id {
      appState.selectedProjectID = appState.projects.first?.id
  }
  ```
  The conditional only resets `selectedProjectID` when **the deleted project is the currently-selected one**. If the user deletes a *non-selected* project while the selected project is valid, the selection stays correct — good.
  But: if the user has **exactly 2 projects**, selected project `A`, and deletes `B`, the ForEach now has 1 project but the `if appState.projects.count > 1` guard at the top of the picker drops the picker entirely. No warning fires in this case.
- The confirmed trigger is **view-lifecycle timing**. The MenuBarExtra popover's content is re-evaluated each time the popover opens. If the `@Published` writes above happened via a background path that didn't fully settle before the first render (e.g., `AppDelegate.performStartup()`'s `DispatchQueue.main.asyncAfter(deadline: .now() + 0.15)` deferral from TKT-021 — introduces a 150ms window where `appState` state is being populated while the popover might already be rendering). The warning is logged "once" per app run per-picker, which aligns with TKT-021's layout-recursion timing window.
- A fourth trigger is **subsequent `ProjectStore` operations** (not observed today, but possible): any external sync of `appState.projects` that doesn't also normalize `selectedProjectID` leaves the binding orphaned.

**The robust fix**: normalize `selectedProjectID` against the current projects list whenever either changes — on projects mutation, if the selection no longer matches any project, reset to `first?.id`. Covers all current and future triggers without touching the picker surface.

### Picker 2 — Scheme picker (`ProjectSetupStep`)

[RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift:144-155](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift#L144-L155):

```swift
Picker("Scheme:", selection: $selectedScheme) {
    ForEach(detectedSchemes, id: \.self) { scheme in
        Text(scheme).tag(scheme)
    }
}
```

Binding type: `Binding<String>`. Tag type: `String`. Match on paper. The warning form `""` fires when `selectedScheme` is the initial empty string but the picker is rendering because `detectedSchemes` has items — and none of those items are the empty string.

**The trigger:** [detectSchemes()](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift#L315-L342) has a stale-state bug at line 324:

```swift
await MainActor.run {
    detectedSchemes = schemes
    if let first = schemes.first, selectedScheme.isEmpty {
        selectedScheme = first
    }
    ...
}
```

The `selectedScheme = first` assignment is gated on `selectedScheme.isEmpty`. This is correct on the **first** detection pass (selectedScheme starts as `""`, gets set to the first scheme).

But on a **subsequent** pass — e.g., the user already has a project configured, drag-drops a different project, which calls `applySelectedPath` → `detectSchemes()` again — `selectedScheme` is no longer empty. It holds the OLD scheme name from the previous project. The new project's scheme list may not contain that name, so after the assignment `detectedSchemes = schemes`, we have:

- `detectedSchemes = ["NewApp", "NewApp-Tests"]`
- `selectedScheme = "OldAppScheme"` (stale)

The picker's selection `"OldAppScheme"` doesn't match any tag in the ForEach → warning.

There's also a subtle transient race at line 317: `detectedSchemes = []` is set before the Task kicks off. During the reset, the view evaluates with `detectedSchemes.isEmpty == true`, which routes to the `else if` branch at line 135 ("No schemes detected") — the picker doesn't render. So the transient itself doesn't warn; only the post-completion stale state does.

**The fix**: after `detectedSchemes = schemes`, validate that `selectedScheme` is still in the list. If not, reset to the first available (or empty string if there are no schemes, which routes the view away from the picker branch anyway).

### Regression risk

- **Low.** Both fixes are defensive state normalization; neither changes the picker's UI or the user-facing flow.
- **Test coverage** is nil on both files. `BuildControlsSection` has no tests. `ProjectSetupStep` has no tests. `ProjectSetupValidators` has tests covering the validators only, not the SwiftUI state flow.
- **Verification is manual.** The warnings are runtime-only; no automated test can reliably trigger SwiftUI's `Picker: selection is invalid` log. Compile gate + manual menu-bar popover + setup-assistant inspection in Xcode console are the only safety nets.
- **Watch items:** make sure the new `.onChange(of: appState.projects)` modifier for the project picker doesn't itself trigger on every popover open — `.onChange` only fires on **value change**, not on view lifecycle, so this is fine by design.


## Proposed Solution

Two independent, surgical fixes. One commit per picker so each can be bisected if a regression shows up.

### Fix 1 — Project picker: normalize `selectedProjectID` on `projects` change

Add an `.onChange(of: appState.projects)` modifier to `BuildControlsSection` (or to `MenuBarView` as a parent — I'll put it on `BuildControlsSection` since that's the only place the picker lives, and colocating the fix with the picker makes the intent obvious to future readers):

```swift
.onChange(of: appState.projects) { _, newProjects in
    // TKT-025: Keep the picker's selection in sync with the available
    // projects. If the current selection no longer matches any project
    // (project deleted, loaded from disk, etc.), reset to the first
    // available. Prevents SwiftUI from logging "Picker: the selection
    // ... is invalid and does not have an associated tag".
    if let id = appState.selectedProjectID,
       !newProjects.contains(where: { $0.id == id }) {
        appState.selectedProjectID = newProjects.first?.id
    } else if appState.selectedProjectID == nil, let first = newProjects.first {
        appState.selectedProjectID = first.id
    }
}
```

Also change the `.tag(Optional(project.id))` call to the canonical form `.tag(project.id as UUID?)`. This is semantically identical but matches Apple's documented pattern for optional-binding pickers and avoids any future confusion about type erasure.

### Fix 2 — Scheme picker: validate `selectedScheme` against the new `detectedSchemes`

Replace the stale-state-tolerant assignment at [ProjectSetupStep.swift:324-326](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift#L324-L326) with a full validation:

```swift
await MainActor.run {
    detectedSchemes = schemes
    // TKT-025: Reset selectedScheme if it no longer matches any
    // detected scheme (e.g., user switched to a different project
    // whose scheme list doesn't include the old selection). Prevents
    // SwiftUI from logging "Picker: selection '' is invalid" when the
    // first detection pass runs against an empty selectedScheme.
    if !schemes.contains(selectedScheme) {
        selectedScheme = schemes.first ?? ""
    }
    isDetectingSchemes = false
    schemeError = ProjectSetupValidators.validateScheme(selectedScheme)
    detectBuildSettings()
}
```

If `schemes` is empty, `selectedScheme = ""` and the picker branch doesn't render (gated by `else if detectedSchemes.isEmpty`). If `schemes` has items and the old selection is valid, nothing changes. If the old selection is invalid (or was empty), it resets to the first scheme.

### Why this approach over alternatives

- **State normalization over picker workarounds**: adding a hidden `.tag(nil as UUID?)` placeholder option would silence the warning but introduces a phantom nil option in the dropdown. The normalization approach keeps the UI clean.
- **`.onChange` over computed Binding**: a computed `Binding<UUID?>` that filters invalid selections in its `get` would also work but obscures the intent — a reader has to trace through the getter to understand what's happening. `.onChange` makes the normalization explicit and local.
- **Co-locate the fix with the picker**: putting the `.onChange` on `BuildControlsSection` rather than `MenuBarView` or `AppState` keeps the fix next to the code it protects. If someone later removes the picker, they'll see the `.onChange` and can clean it up in the same PR.

### Tradeoffs

- The `.onChange` runs every time `appState.projects` changes, even though the normalization is usually a no-op. Negligible overhead — it's a linear scan of a typically 1–3 element array.
- The scheme picker fix changes behavior in one edge case: previously, if a user switched projects and the old scheme name happened to *also* exist in the new project's scheme list, the old selection was preserved. With the fix, that still works (because `schemes.contains(selectedScheme)` is true). Only changes behavior when the old selection is invalid for the new project — which is the bug we're fixing.
