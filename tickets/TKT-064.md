## Description

The PWA visual overhaul (TKT-061) restyled the real web app, which the server
serves at `/app/`. But the server root `/` -- the URL a user actually opens when
they navigate to the Tailscale hostname -- still serves `buildIndexPage()`, an
old, minimal server-rendered HTML page that just lists projects with install
links. So opening the URL shows the old design; the fancy new PWA is hidden one
path deeper at `/app/`. From the user's point of view the new design "isn't
working" because nothing routes them to it.

Goal:
1. Use the new web app (the `/app/` PWA from TKT-061) as the canonical client.
2. Retire the old one (`buildIndexPage` + its root route).
3. Make the new web app the default at the root URL, so opening the bare
   hostname lands on the PWA.

## Reproduction Steps (bugs only)
1. Open the server URL root (e.g. `https://<host>.<tailnet>.ts.net:8443/`).
2. Observe the old plain project-list page, not the TKT-061 PWA.
3. The new design only appears if you manually navigate to `/app/`.

## Acceptance Criteria
- [ ] Opening the server root URL `/` loads the new TKT-061 PWA.
- [ ] The old `buildIndexPage` server-rendered index is removed.
- [ ] Existing entry points keep working: installed PWAs / bookmarks pointing at
      `/app/` still load (serve there or 301-redirect, decided in investigate).
- [ ] Per-project OTA install pages at `/<slug>/` continue to work unchanged
      (these are NOT the old index and must stay).
- [ ] `manifest.json` (`start_url` / `scope`), `sw.js` scope, and `index.html`
      asset paths are consistent with wherever the app is served, so PWA install
      + offline fallback still function.
- [ ] `HTTPServerIntegrationTests.testRootReturnsIndexPage` is updated to assert
      the new root behavior (it currently asserts the old HTML).
- [ ] Confirm the new PWA actually renders/functions end to end (pair, list
      projects, trigger build, see live log) -- not just that it loads.

## Investigation

Current routing (`RemoteDeploy/Services/HTTPHandler.swift`):
- `/api/...` -> API router.
- `/app/` and `/app/<file>` -> `servePWAFile` -> bundle `Resources/pwa/*` (the
  new PWA). (`HTTPHandler.swift:139-143`)
- `GET /` -> `buildIndexPage(projects:)` -> old server-rendered list.
  (`HTTPHandler.swift:153-156`, builder at
  `RemoteDeploy/Services/NIOResponseGenerator.swift:17`)
- `GET /<slug>/...` -> per-project OTA install page / manifest / ipa / zip.
  (still needed)

The PWA is bound to the `/app/` prefix:
- `Resources/pwa/manifest.json`: `start_url` and `scope` are both `/app/`.
- `Resources/pwa/sw.js`: caches `/app/`, `/app/style.css`, `/app/app.js`,
  `/app/manifest.json`; runs under the `/app/` SW scope.
- `Resources/pwa/index.html`: links assets via absolute `/app/...` paths.

Only one caller of `buildIndexPage` (`HTTPHandler.swift:154`) and one test
asserts the old root (`HTTPServerIntegrationTests.testRootReturnsIndexPage`).

## Proposed Solution

Two viable approaches -- pick during investigate:

A. Redirect `/` -> `/app/` (smallest change). Keep the PWA at `/app/`; root
   issues a 301/302. Pros: SW scope, manifest, asset paths unchanged; installed
   PWAs unaffected. Cons: the canonical URL stays `/app/`; bare-host bookmarks
   carry a redirect hop.

B. Serve the PWA at root `/` and drop the `/app/` prefix (cleaner end state).
   Move `manifest.json` start_url/scope to `/`, `sw.js` shell list + scope to
   `/`, and `index.html` asset paths to root-relative; keep a `/app/` ->  `/`
   redirect for back-compat with installed PWAs. Cons: more moving parts; must
   re-verify SW update + offline; bump `CACHE_NAME`.

Recommendation: B for the long term (root is the natural home), with a `/app/`
redirect so existing installs don't break -- but confirm during investigate that
nothing else hardcodes `/app/`. In both approaches, remove `buildIndexPage` and
update the root test. Watch out: project OTA install pages live at `/<slug>/`;
the PWA owns `/` only, and slug routing must still win for real slugs.

## Chosen approach

Approach B (serve the PWA at the root `/`, redirect `/app/` -> `/` for back-compat),
confirmed with the user.

Also fixes a latent rendering bug found during implementation: the server's CSP
sent `style-src 'unsafe-inline'` with no `'self'`, which blocks same-origin
`<link rel=stylesheet>`. That is why the PWA rendered completely unstyled even at
`/app/` (the stylesheet was CSP-blocked all along; it only looked styled in the
Claude Design preview, which has no CSP). New CSP:
`default-src 'self'; img-src 'self' data:; style-src 'self' 'unsafe-inline'; connect-src 'self' wss: ws:`
(`'self'` so the stylesheet loads; `img-src data:` so the stylesheet's inline
data: SVG icons/masks render).

The server lives in the `RemoteDeployServer` target (Phase 6 split); it -- not the
menu bar app -- serves `:8443`. Redeploy the server to make this live.

## Regression Report
- Slug routing preserved: PWA assets are single-segment dotted paths; project slugs
  never contain ".", so `/<slug>/` and `/<slug>/app.ipa|app.zip|manifest.plist`
  still route to OTA (covered by existing passing integration tests). `/app` was
  already PWA-owned (never a usable slug), so the redirect introduces no conflict.
- SW scope widened from `/app/` to `/`; the fetch handler now ignores everything
  but shell assets, so OTA pages and `.ipa`/`.zip` downloads are never intercepted
  or cached.
