## Description

When deploying a macOS app to the same machine it's running on, the deploy process should gracefully quit the existing instance before launching the newly built one. Currently the preview profile does `pkill -x RemoteDeploy` before building and launching, but this needs to be a first-class, reliable pattern that works for:

1. **ClaudeDash** (immediate need) — iterative build-deploy cycles during active development where the app must be replaced cleanly each time.
2. **RemoteDeploy self-deploy** (stretch goal) — the menu bar app deploying a new version of itself. This is trickier because the running process is the one orchestrating the deploy, but worth investigating.

The core behavior should be: build succeeds → quit existing instance → launch new instance. If the existing app doesn't quit cleanly within a timeout, force-kill it.


## Acceptance Criteria

- [ ] Deploy pipeline gracefully terminates the running instance of the target app before launching the new build
- [ ] If graceful quit fails within a configurable timeout, force-kill the process
- [ ] Works reliably for ClaudeDash iterative development cycles
- [ ] Investigation covers whether RemoteDeploy can self-deploy (and documents the approach or explains why not)
- [ ] No orphaned processes or duplicate instances after a deploy cycle

## Investigation

### Current state of deploy / preview

The **preview profile** in [ticket-config.md](.claude/ticket-config.md:39-42) already does a basic kill-and-replace:

```
pkill -x RemoteDeploy 2>/dev/null; sleep 1; xcodegen generate && xcodebuild build … && open …/RemoteDeploy.app
```

This works but has problems:
1. **`pkill -x` sends SIGTERM** — no graceful cleanup. NIODeployServer's `stop()` ([NIODeployServer.swift:306-325](RemoteDeploy/Services/NIODeployServer.swift#L306-L325)) shuts down channels and the event loop group, but SIGTERM bypasses it entirely since there's no signal handler wired up. The port may not be released immediately.
2. **`sleep 1` is a fixed guess** — too short if the app is mid-build, too long for a clean quit.
3. **No verification** the app actually quit before launching the new one — `open` could launch a second instance.
4. **No AppleScript / graceful NSApplication.terminate path** — the app has a Quit button ([UtilitiesSection.swift:48](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L48)) that calls `NSApplication.shared.terminate(nil)`, which would allow AppKit lifecycle cleanup.

### ClaudeDash (immediate need)

ClaudeDash is a separate project the user is actively developing. The need is a **generic shell script / pattern** that any macOS app's deploy pipeline can use: quit existing → build → launch new. This should be reusable across both projects.

### RemoteDeploy self-deploy (stretch goal)

This is the "ship that replaces itself mid-voyage" problem. Key observations:

- **`build-release.sh`** ([scripts/build-release.sh](scripts/build-release.sh)) builds to `/tmp/RemoteDeployRelease/`, signs, notarizes, and creates a DMG. It does NOT launch the result.
- **`ship-deploy.sh`** ([scripts/ship-deploy.sh](scripts/ship-deploy.sh)) gates whether to run build-release.sh at all. Neither script touches the running app.
- The preview profile builds to `/tmp/RemoteDeployPreview/` and uses `open` to launch.
- **Self-deploy IS feasible** on macOS: unlike Linux, macOS doesn't lock the app bundle on disk. The running process has its Mach-O image loaded in memory. You can overwrite/replace the `.app` bundle and then relaunch. The standard pattern is:
  1. Build the new version to a staging path
  2. Use `osascript` to tell the running app to quit (or send SIGTERM + wait)
  3. Replace the `.app` bundle (or just `open` the new one from the build path)
  4. The new process launches
  
  The **only wrinkle** for self-deploy: the deploy agent (RemoteDeploy itself) must arrange for the new version to launch *after* it exits. This is solved with a tiny "trampoline" — a shell one-liner launched via `Process` that waits for the PID to die, then runs `open`.

### Regression risks

- **Port binding**: if the old app doesn't release port 8443/8080 before the new one starts, the new server will fail to bind. Graceful quit via AppleScript/terminate is critical.
- **Settings files**: both old and new versions read/write `~/Library/Application Support/RemoteDeploy/settings.json`. No risk of corruption since the old process will be dead before the new one reads.
- **Bonjour**: the old app's mDNS advertisement needs to be torn down before the new one advertises. SIGTERM won't do this; `NSApplication.terminate` would trigger `applicationWillTerminate` if we add cleanup there.

## Proposed Solution

Create a reusable **`scripts/graceful-relaunch.sh`** script that any macOS app's deploy pipeline can invoke. The script:

1. **Graceful quit via `osascript`**: sends `quit` to the app by name, which triggers `NSApplication.terminate` and gives AppKit + SwiftNIO time to clean up (close channels, release ports, tear down Bonjour).
2. **Wait with timeout**: polls for the process to exit (up to N seconds, default 5).
3. **Force-kill fallback**: if the process is still alive after the timeout, sends `SIGKILL`.
4. **Port-wait** (optional): waits for the app's port to be free before proceeding, preventing "address already in use" on relaunch.
5. Returns success, so the caller can then `open` the newly built `.app`.

The **preview profile** in ticket-config.md gets updated to call this script instead of bare `pkill`.

For **RemoteDeploy self-deploy**: add an `applicationWillTerminate` handler to [AppDelegate.swift](RemoteDeploy/AppDelegate.swift) that stops the NIO server synchronously (or as close to it as AppKit allows), ensuring port release. The graceful-relaunch script handles the rest.

**Why this approach**: A reusable shell script is the simplest thing that works across projects. AppleScript `quit` is the macOS-native way to gracefully terminate an app — it's what Cmd+Q does. No need for IPC, XPC, or custom protocols.

**Tradeoffs**: AppleScript requires the app to be a proper `.app` bundle with an Info.plist (both RemoteDeploy and ClaudeDash are). The 5-second timeout is a judgment call — too short and force-kill fires unnecessarily, too long and iteration feels slow. Making it configurable (env var) costs nothing.

## Implementation Plan

- [ ] 1. Create `scripts/graceful-relaunch.sh` — accepts app name (and optional port, timeout) as args. Uses `osascript -e 'tell application "AppName" to quit'`, polls with `pgrep`, falls back to `kill -9`, optionally waits for port release via `lsof`.
- [ ] 2. Add `applicationWillTerminate` to [AppDelegate.swift](RemoteDeploy/AppDelegate.swift) — calls `serviceContainer.deployServer.stop()` and `serviceContainer.bonjourAdvertiser.stop()` to release ports and mDNS before exit. Use a synchronous bridge (DispatchSemaphore) since applicationWillTerminate is sync.
- [ ] 3. Update the **macos preview profile** in [.claude/ticket-config.md](.claude/ticket-config.md) — replace `pkill -x RemoteDeploy 2>/dev/null; sleep 1` with `scripts/graceful-relaunch.sh RemoteDeploy --port 8443`.
- [ ] 4. Add a usage comment at the top of `graceful-relaunch.sh` showing how ClaudeDash (or any app) can use it: `scripts/graceful-relaunch.sh ClaudeDash --port 8787`.
- [ ] 5. Test: run RemoteDeploy via preview, then run preview again — verify clean quit, no port conflict, no duplicate instances.
- [ ] 6. Test: verify `open RemoteDeploy.app` after graceful quit launches successfully with working server.
- [ ] 7. Run existing unit and integration tests to check for regressions.
