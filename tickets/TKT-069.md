## Description

Using "Pair Another Device" (TKT-065) to enroll a second phone immediately
invalidates the FIRST phone: the first device starts getting
`HTTP 401: Invalid or missing bearer token`, and the Mac's Devices list only ever
shows a single paired device. You can never have two phones paired at once -- the
exact opposite of what TKT-065 was built to do.

Root cause is a collision between two changes that landed ~10 minutes apart:

1. **TKT-066 name-based dedup (commit 33aada9).** `PairingRouteHandler.pair()`
   calls `revokeStaleRecords(matching:)`, which deletes every non-loopback paired
   device whose `name` equals the just-paired device's `name`. The intent was to
   collapse repeated re-pairs of the *same* phone into one record.

2. **Generic device name (the enabler).** The companion sends
   `UIDevice.current.name`, which on iOS 16+ is the generic model string "iPhone"
   for *every* device unless the app holds the `user-assigned-device-name`
   entitlement (the companion has no entitlements file -- the keychain one was
   tried and reverted in 78ecbbb). The pair request carried no other stable
   identifier, so the server could not tell two different phones apart.

=> Pairing Phone B (name "iPhone") runs the dedup and deletes Phone A
   ("iPhone"). Phone A's token is gone -> 401. The Devices list collapses to one
   "iPhone". The `revokeStaleRecords` doc comment even acknowledged this exact
   failure mode but accepted it; TKT-065's whole purpose makes it unacceptable.

Two further defects compounded the confusion:

3. **Tests locked the bug in.** `test_pair_revokesOlderRecordsWithSameName`
   asserted that same-name records ARE revoked, so the green suite protected the
   regression.

4. **Completion-detection masked the damage (commit 6c8a470).** The Mac's pairing
   sheets were changed from "device count increased" to "a new device id appeared"
   *because* the dedup kept the count flat. Side effect: the Mac shows
   "Device Paired!" success at the very moment the dedup silently revoked the
   previous device, so the destructive eviction looked like a successful pairing.

## Reproduction Steps (bugs only)
1. Pair Phone A with the Mac (works; Devices list shows one "iPhone").
2. On Phone A: Settings -> Pair Another Device -> show the QR.
3. Scan it with Phone B (stock companion scanner). Phone B pairs.
4. Observe: the Mac's Devices list still shows ONE "iPhone" (Phone B); Phone A now
   gets `HTTP 401: Invalid or missing bearer token` on its next request.

## Fix

Dedup on a stable per-install identity, never on the device name.

- **Shared:** add `installID: String?` to `PairRequest` and `PairedDevice`
  (optional, Codable -- old on-disk records decode to nil). Add an
  `installID:` parameter to `APIClient.completePairing(deviceName:installID:)`.
- **Companion:** `KeychainStore.installID()` returns a UUID generated once and
  persisted in the Keychain (deliberately NOT cleared on unpair -- identity is
  per-install, not per-pairing). `ConnectionManager.pair()` sends it.
- **Server:** `revokeStaleRecords(matching:)` now revokes only records whose
  `installID` equals the freshly paired device's `installID`. A device with no
  `installID` (loopback record, browser/PWA, or an older companion build) is
  never a dedup source or target -- we leave any orphan in place rather than risk
  evicting a live device. The name check and the loopback name special-case are
  gone (loopback has no installID, so it is naturally excluded).

Result:
- Two distinct phones (both named "iPhone") have different installIDs ->
  pairing the second never evicts the first. TKT-065 works.
- A reinstall of the SAME phone sends the same installID -> its prior orphan is
  collapsed, satisfying TKT-066's dedup goal.

### Tradeoff / known limitation
`installID` is stored in the same plain Keychain item as the credential blob, so
it survives a reinstall exactly as well as the saved credential does. If a
re-signed / OTA reinstall changes the keychain access group (the scenario
TKT-066 raised), neither the credential nor the installID carries over: the app
re-pairs and gets a fresh installID, leaving the old record as a hand-revocable
orphan. That is strictly safer than the previous behavior, which silently revoked
a live device. Reliable cross-reinstall identity needs the stable
keychain-access-group entitlement that was reverted in 78ecbbb; out of scope here.
This change supersedes TKT-066's name-based dedup mechanism.

## Files Changed
- `Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/PairedDevice.swift`
  -- add optional `installID`.
- `Packages/RemoteDeployShared/Sources/RemoteDeployShared/API/APITypes.swift`
  -- `PairRequest` gains optional `installID`.
- `Packages/RemoteDeployShared/Sources/RemoteDeployShared/API/APIClient.swift`
  -- `completePairing(deviceName:installID:)`.
- `RemoteDeploy/API/Routes/PairingRouteHandler.swift`
  -- set `installID` on the new record; `revokeStaleRecords` matches on installID.
- `RemoteDeployCompanion/Services/KeychainStore.swift`
  -- add `installID()` (Keychain-persisted, not cleared on unpair).
- `RemoteDeployCompanion/Services/ConnectionManager.swift`
  -- send `installID` on pair.
- `RemoteDeployTests/API/PairingRouteHandlerTests.swift`
  -- replace the name-based dedup tests with installID-based ones plus a TKT-065
     regression guard (two same-name devices with distinct installIDs coexist).
