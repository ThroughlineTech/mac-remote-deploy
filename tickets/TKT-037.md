## Description

The Server tab in Settings has the right fields but they're floating in a large empty window with no visual structure. Screenshot shows: Port, Hostname, Certificate, and Private Key fields stacked vertically in the bottom half of the window with massive dead space above them. No section headers, no grouping, no status indicators.

### Issues

1. **Fields sink to the bottom** — the content sits low in the window with ~200px of blank space above it. Looks like a layout bug.
2. **No visual grouping** — Port and Hostname are conceptually "Server Identity" while Certificate and Private Key are "TLS Configuration." They should be in separate `GroupBox` or `Form` sections.
3. **No server status at the top** — the user has to look at the menu bar popover to know if the server is running. The Settings tab should show current status (running/stopped, port, URL) prominently at the top with a start/stop/restart button.
4. **Certificate paths are raw filesystem paths** — `/Users/plympton/Library/Application Support/RemoteDeploy/...` is ugly and truncated. Should show the filename only with a tooltip or secondary line for the full path. Or show a "Generated via Tailscale" badge when the cert was auto-generated.
5. **"Auto-Detect" button for hostname** — good, but no visual feedback when it succeeds. A brief checkmark or "Detected: fubars-mac-mini-2.tailf3787.ts.net" confirmation would help.
6. **No "Save" or "Apply" action** — unclear when changes take effect. Do they apply immediately? On next server restart? A "Restart Server" button after changing port/hostname would make the flow explicit.

### Design direction

Match macOS System Settings (Ventura+) style:
- Top section: server status card (green/red dot, port, URL, start/restart button)
- Middle section: "Server Identity" GroupBox (Port, Hostname with Auto-Detect)
- Bottom section: "TLS Certificate" GroupBox (Cert path, Key path, Browse buttons, "Generate via Tailscale" button)
- Tighter vertical rhythm, fields aligned at the top of the window not sinking to the bottom

This can be bundled with TKT-035 (notifications polish) and TKT-036 (General tab) as a "Settings UI polish pass" if preferred — they're all the same genre of work.

## Acceptance Criteria
- [ ] Server settings fields are grouped into labeled sections (e.g., "Server Identity" and "TLS Certificate")
- [ ] Content is anchored at the top of the window, not floating at the bottom
- [ ] Server running status is visible in the Server tab (not only in the popover)
- [ ] Certificate paths show the filename (not the full raw path) with the full path accessible via tooltip or secondary text
- [ ] No functional regression — port, hostname, cert, and key settings still persist and apply correctly
- [ ] `xcodebuild test` passes

## Investigation

### Current layout (ServerSettingsTab in SettingsView.swift)

**File:** `RemoteDeploy/Views/SettingsView.swift`, lines 59-185.

The `ServerSettingsTab` is a flat `Form` with four fields and no grouping:
1. `TextField("Port:", ...)` — bare, 80px wide (line 72-80)
2. `HStack` with `TextField("Hostname:", ...)` + "Auto-Detect" button (lines 83-100)
3. `Divider()` (line 102)
4. `HStack` with `TextField("Certificate:", ...)` + "Browse..." button (lines 105-118)
5. `HStack` with `TextField("Private Key:", ...)` + "Browse..." button (lines 120-134)

The `Form` has `.padding()` but no `.frame(maxHeight: .infinity, alignment: .top)`, so SwiftUI's default `Form` layout centers content vertically in the 450px-tall settings window, pushing the fields toward the bottom half.

### Server status availability

`AppState` (lines 12-13 of `AppState.swift`) exposes:
- `@Published var serverRunning = false`
- `@Published var tailscaleConnected = false`
- `@Published var serverURL = ""`
- `@Published var serverPort: Int = 8443`

`ServerStatusSection.swift` (lines 1-53) already renders a compact server/Tailscale status view with green/red dots, port, and URL for the menu bar popover. This can be reused or adapted for the Settings tab.

### Server restart capability

`DeployServerProtocol` exposes `start(port:certPath:keyPath:)` and `stop()`. `AppDelegate.startServer()` (line 318) already handles the full start sequence. There is no `restart()` method, but a restart is just `stop()` then `startServer()`. The `startServer()` method already guards against double-start (`guard !appState.serverRunning`), so the flow would be: stop -> set serverRunning=false -> startServer().

Currently there is no `restartServerRequested` notification; a new one would need to be added, or the "Restart Server" button can post `startServerRequested` after stopping.

### Certificate paths

Cert paths are stored as full filesystem paths (e.g., `/Users/plympton/Library/Application Support/RemoteDeploy/cert.pem`). The `TextField` displays the raw string. There is no logic to extract just the filename or detect whether the cert was auto-generated via Tailscale.

### Auto-Detect feedback

The "Auto-Detect" button (line 90-99) toggles `isDetectingHostname` to show a `ProgressView` spinner during detection. On success, it silently sets the hostname. No checkmark or confirmation text is shown.

### TKT-036 overlap

TKT-036 proposes merging the General tab's "Launch at Login" toggle into the Server tab (Option B). If TKT-036 is implemented first with Option B, the Server tab would gain a "Launch at Login" toggle. TKT-037 should account for this by placing it in a sensible section (e.g., a "General" GroupBox at the bottom, or within the server status card). However, TKT-037 does not depend on TKT-036 — if TKT-036 hasn't landed, we simply don't include the toggle.

### Regression risk: LOW

All changes are purely UI/layout. The underlying data flow (AppState properties, onChange handlers, requestSaveSettings, openFilePanel) remains untouched. The `Form` is replaced with a `ScrollView > VStack` with `GroupBox` sections, but every field binding and callback stays identical.

## Proposed Solution

Restructure `ServerSettingsTab` into three visual sections, anchored at the top of the tab:

1. **Server Status card** (top) — Reuse the status dot pattern from `ServerStatusSection.swift`. Show running/stopped state, port, URL, and a "Restart Server" button. This gives the user immediate visibility without opening the popover.

2. **"Server Identity" GroupBox** (middle) — Port and Hostname fields with the Auto-Detect button. Add brief success feedback (checkmark + detected hostname text) after auto-detect completes.

3. **"TLS Certificate" GroupBox** (bottom) — Certificate and Private Key fields. Display just the filename (via `URL(fileURLWithPath:).lastPathComponent`) with a `.help()` tooltip showing the full path.

Wrap everything in a `ScrollView > VStack(alignment: .leading, spacing: 16)` with `.frame(maxHeight: .infinity, alignment: .top)` so content sits at the top instead of centering.

Add a `restartServerRequested` notification (or reuse `startServerRequested` after calling stop) so the "Restart Server" button in Settings actually cycles the server.
