## Description

Phase 2 of the backend-decoupling plan (docs/backend-decoupling-plan.md). Today
there are two sources of truth for state:

1. **Projects.** The menu bar observes/mutates an in-memory `appState.projects`
   array while the API writes the on-disk `projectStore`. A project created via
   the API is not visible in the menu bar until relaunch, and its `/<slug>/`
   install route does not serve until the next server start (the
   `lockedProjectsBySlug` registry is only synced at `startServer`).
2. **Settings/status.** The API reads server/TLS/push settings and live status
   through an `AppStateBridge` snapshot of `AppState`, so `AppState` is the truth
   and the API mirrors it.

Make the stores authoritative: the menu bar observes a store-backed project
source, the deploy-server registry stays in sync on every project change (not
just startup), settings flow through one thread-safe writer, and the API reads
the stores directly. After this `AppStateBridge` is deletable.

## Acceptance Criteria
- [x] Projects have one source of truth: the `projectStore`. Every writer (API,
      menu bar settings, menu bar popover, setup assistant) writes only the store;
      `appState.projects` is a read-only projection refreshed from the store on
      change. A project created via the API appears in the menu bar with no
      relaunch.
- [x] The deploy-server `lockedProjectsBySlug` registry stays in sync on every
      project create/edit/delete via any path (new `syncProjects`), so a newly
      created project's `/<slug>/` install route serves immediately.
- [x] Settings have one writer: a thread-safe `SettingsStore` persisted to
      settings.json. The API reads/writes the store directly; the menu bar's
      saves route through it; `AppState`'s settings fields are refreshed from it.
- [x] The API status/settings providers read the stores (and BuildManager build
      status) directly -- not an `AppState` snapshot.
- [x] `AppStateBridge` and the `AppState`-snapshot providers
      (`AppStateStatusProvider`, `AppStateBridgeSettingsProvider`) are deleted.
- [x] All existing tests stay green; new tests cover `SettingsStore`, the
      store-backed settings updater, the project change notification, and the
      registry replace-all. 402 passed, 0 failed.

## Proposed Solution

Keep `AppState` as the menu bar's in-process UI model, but make the stores the
single source of truth that both the UI and the API read/write through.

### Projects
- `UserDefaultsProjectStore.save/delete` post `.projectsDidChange` after a
  successful write (fires for the API on a NIO thread and the menu bar on main).
- `AppDelegate` observes `.projectsDidChange`, hops to the main actor, reloads
  `appState.projects` from the store, normalizes `selectedProjectID`, and calls
  `deployServer.syncProjects(projects)` to rebuild the slug registry.
- `DeployServerProtocol.syncProjects(_:)` replaces the whole `lockedProjectsBySlug`
  map (real impl in `NIODeployServer`; a register-only default covers mocks).
- Menu bar write paths (`ProjectsSettingsTab`, `ProjectSetupStep.saveProject`,
  `ProjectsListSection.removeProject`) write only the store; the observer
  refreshes `appState.projects`. Removes the dual write.

### Settings
- New `SettingsStore` (thread-safe, `OSAllocatedUnfairLock`-backed, persisted to
  settings.json). Single writer. Conforms to `SettingsProviding` so the API reads
  it directly. Owned by `ServiceContainer`.
- `DeferredSettingsUpdater` writes validated settings through `SettingsStore`
  (drops the `AppStateBridge` dependency); the store posts `.settingsDidChange`.
- `AppDelegate.loadSettings/saveSettings` become thin wrappers over the store;
  a `.settingsDidChange` observer refreshes `AppState`'s settings fields and
  recomputes `serverURL`. Detected Tailscale hostname is written to the store
  (guarded on change) so the status endpoint reports the live hostname.

### Status + bridge deletion
- `ServerStatusProvider` (replaces `AppStateStatusProvider`) reads hostname/port
  from `SettingsStore`, serverRunning/port from the deploy server, build status
  from `BuildManagerBuildStatusProvider`, and `tailscaleConnected` from a small
  thread-safe `RuntimeStatusStore` updated by the Tailscale poll.
- Delete `AppStateBridge`, `AppStateStatusProvider`, `AppStateBridgeSettingsProvider`.

## Review Notes

- **[fixed] Lock-held notification.** `UserDefaultsProjectStore` originally posted
  `.projectsDidChange` while the `NSLock` was still held (before the `defer`
  unlock). Safe with today's only observer (it hops to a Task and never re-enters
  the store synchronously), but a future synchronous observer reading the store
  back would deadlock since `NSLock` is not reentrant. Hardened: the lock is now
  scoped to a `withLock { ... }` helper and the post happens after it returns, so
  notifications always fire outside the lock. `SettingsStore.update` already posts
  outside its `OSAllocatedUnfairLock`.
- **[accepted, harmless] Redundant recompute/sync.** The Tailscale poll sets
  `appState.serverURL` directly and the guarded store write recomputes it via
  `.settingsDidChange`; `syncProjects` runs once in `refreshProjectsFromStore`
  and again in `startServer` at boot. Both are idempotent and defensive; left as-is.
- **[accepted] Behavioral shift.** Local menu bar project edits now update the UI
  via the `.projectsDidChange` observer (one run-loop later) rather than by
  synchronous in-memory mutation. Imperceptible, and `BuildControlsSection`'s
  `.onChange(of: appState.projects)` still guards the picker selection. This is
  the intended single-source-of-truth trade-off; the full client inversion is
  Phase 3.
