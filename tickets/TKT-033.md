## Description

Two related window-management issues in the menu bar app:

### 1. "Setup Guide" vs "Setup Assistant" — pick one, be consistent

The same window is called different things depending on where you look:

| Location | Name used |
|---|---|
| Menu bar popover button label | "Setup Guide" ([UtilitiesSection.swift:23](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L23)) |
| Window title bar | "Setup Assistant" ([RemoteDeployApp.swift:63](RemoteDeploy/RemoteDeployApp.swift#L63)) |
| Code comments, struct names, source files | "Setup Assistant" (SetupAssistantView.swift, ProjectSetupValidators comments, etc.) |
| AppDelegate log/launch message | "setup assistant" ([AppDelegate.swift:183](RemoteDeploy/AppDelegate.swift#L183)) |

The button says "Setup Guide" but clicking it opens a window titled "Setup Assistant." This is jarring — the user wonders if they opened the right thing. The codebase overwhelmingly uses "Setup Assistant" (it's the struct name, the window ID, the AppDelegate message). The button label is the outlier.

**Fix:** rename the button label from "Setup Guide" to "Setup Assistant" in [UtilitiesSection.swift:23](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L23). One line.

### 2. Windows open behind VS Code (and other apps)

When clicking "Setup Guide" or "Settings..." from the menu bar popover, the window opens but does not come to the front. It appears behind the frontmost app (VS Code, Safari, etc.). The user has to manually click on the RemoteDeploy window in the dock-less app switcher or use Mission Control to find it.

**Root cause:** `LSUIElement = true` apps (no dock icon) don't get automatic window-ordering treatment from AppKit. When a window opens, AppKit doesn't call `NSApp.orderFrontRegardless()` or `NSApp.activate(ignoringOtherApps:)` on the new window — the app stays in the background and the window renders behind whatever's currently focused.

The "Settings..." button has a partial fix: `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })` at [UtilitiesSection.swift:37](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L37). But `NSApp.activate()` alone doesn't always bring windows to front in modern macOS (14+) — Apple deprecated the `ignoringOtherApps:` variant, and the default behavior respects the current activation policy.

The "Setup Guide" button has NO `NSApp.activate()` at all — which is why TKT-032 exists (it needs Settings opened first to "warm up" the activation state).

**Fix options:**
- **A)** Add `NSApp.activate()` to both button actions. Use `NSApp.activate()` (no deprecated flags) + `NSApp.windows.first(where: { $0.title == "Setup Assistant" })?.orderFrontRegardless()` after the window opens.
- **B)** Use `NSApp.setActivationPolicy(.regular)` temporarily when opening a window, then switch back to `.accessory` after the window closes. This makes the app behave like a normal app (dock icon appears) while a window is open, which gets AppKit's full window-ordering treatment.
- **C)** For the Setup Assistant specifically: instead of `openWindow(id:)`, use `NSApp.sendAction(Selector(("showSettingsWindow:")), to: nil, from: nil)` — no wait, that's for Settings only. For custom windows, force-ordering is the right call.

## Reproduction Steps

### Naming inconsistency
1. Open the menu bar popover.
2. Read the button: "Setup Guide"
3. Click it.
4. Read the window title bar: "Setup Assistant"
5. Notice the mismatch.

### Window behind other apps
1. Have VS Code (or any app) in the foreground, fullscreen or covering the screen.
2. Click the RemoteDeploy menu bar icon → popover opens.
3. Click "Settings..." or "Setup Guide."
4. The popover dismisses. The window is open but behind VS Code.
5. Switch to RemoteDeploy via Mission Control or ⌘Tab → the window is there, just not in front.
