## Description
On app startup, the following warnings appear in the console:

```
It's not legal to call -layoutSubtreeIfNeeded on a view which is already being laid out.
If you are implementing the view's -layout method, you can call -[super layout] instead.
Break on void _NSDetectedLayoutRecursion(void) to debug. This will be logged only once.
This may break in the future.
send failed: Invalid argument
send failed: Invalid argument
```

These shouldn't be appearing at startup. Investigate the root cause of both:
1. AppKit layout recursion warning — likely a SwiftUI/AppKit view triggering layout while already laying out.
2. "send failed: Invalid argument" — possibly from the NIO server, Bonjour advertiser, or a socket send during startup.

Fix both if possible.

## Reproduction Steps (bugs only)
1. Launch the RemoteDeploy macOS host app.
2. Observe console output on startup.
3. The warnings above appear.

- [ ] "send failed: Invalid argument" messages no longer logged at startup

## Investigation

### Warning 1 — `_NSDetectedLayoutRecursion`

AppKit fires this when something triggers a layout pass on a view already inside its own `-layout`. In a pure-SwiftUI menu-bar app with no custom `NSView`/`NSViewRepresentable`, the usual culprit is state mutated during SwiftUI `body` evaluation, which causes AppKit to re-enter layout on the `MenuBarExtra`'s hosting view.

[RemoteDeployApp.swift:26-33](RemoteDeploy/RemoteDeployApp.swift#L26-L33) calls `appDelegate.register(...)` directly inside `body`. On the first invocation ([AppDelegate.swift:59-77](RemoteDeploy/AppDelegate.swift#L59-L77)) it spawns a `Task { @MainActor in await self.performStartup() }` if `applicationDidFinishLaunching` has already fired. `performStartup()` then mutates `appState` extensively (`loadSettings`, `loadSavedProjects`, `checkTailscaleStatus`, `startServer`) — all `@Published` writes. When that Task lands while SwiftUI is still mid-`body`/layout cycle, AppKit sees recursive layout on the MenuBarExtra hosting window and logs once (matching the "logged only once" wording).

Even when `applicationDidFinishLaunching` fires first (the normal TKT-019 ordering), its Task ([AppDelegate.swift:101-104](RemoteDeploy/AppDelegate.swift#L101-L104)) is enqueued immediately and can still interleave with the first `body` eval, producing the same symptom.

Secondary smell: `let _ = appDelegate.register(...)` inside `body` is discouraged — SwiftUI may re-evaluate `body` at any time, and side-effectful work there is exactly what triggers "mutation during layout" warnings.

**Root cause:** State-mutating startup work is kicked off during SwiftUI `body` evaluation and/or synchronously from `applicationDidFinishLaunching`, overlapping AppKit's first layout pass on the `MenuBarExtra` hosting view.

### Warning 2 — `send failed: Invalid argument` (×2)

This is a Darwin log from the Bonjour/`dnssd` client socket, emitted when the low-level send to `mdnsresponder` returns `EINVAL`. Common causes: TXT record with an empty key/value, `port = 0`, no run loop scheduled (rare on main thread), or rapid repeat publishes on the same object.

[BonjourAdvertiser.swift:44-67](RemoteDeploy/Services/BonjourAdvertiser.swift#L44-L67) guards empty `hostname` and uses `if let` for `localIP`, so no empty values are sent. `httpPort` is hardcoded `8080` ([AppDelegate.swift:322-327](RemoteDeploy/AppDelegate.swift#L322-L327)). So the TXT content is valid.

The "×2" pattern is characteristic of the DNS-SD client retrying over IPv4 and IPv6 control sockets during the initial `publish()`. On macOS 14/15 this shows as two identical `send failed: Invalid argument` lines when the service is registered while the resolver is still initializing, or when a stale advertisement from a prior run is cached in `mdnsresponder`.

Contributing factor: `bonjourAdvertiser.start(...)` is called inside the same `Task` as `server.start(...)` ([AppDelegate.swift:313-327](RemoteDeploy/AppDelegate.swift#L313-L327)), racing NIO's bind on :8080 against mDNSResponder learning about the socket.

**Root cause:** Benign-but-noisy Apple framework chatter from the legacy `NetService.publish()` API, aggravated by publishing immediately after bind. Apple's guidance on modern macOS is to prefer `NWListener` Bonjour integration, but the cheap fix is to defer the publish by a single run-loop turn.

### Regression risk

- **Layout recursion fix:** Touches app startup sequencing. Existing tests don't cover `AppDelegate` directly (instantiated by SwiftUI). Must verify server still starts immediately at launch (TKT-019 requirement). Low-medium risk.
- **Bonjour fix:** A small timing defer keeps the public API unchanged and is low risk. A full `NWListener` migration would be medium risk and is only a fallback.


## Proposed Solution

### Fix 1 — Remove side effects from `body`; defer startup past first layout

1. Delete `let _ = appDelegate.register(...)` from `RemoteDeployApp.body`. Move registration to `.onAppear` on `MenuBarView` (reading the three `@EnvironmentObject` values there).
2. In `AppDelegate.applicationDidFinishLaunching(_:)` and `AppDelegate.register(...)`, wrap the startup `Task` dispatch in `DispatchQueue.main.async { ... }` so `performStartup()` runs strictly *after* the current layout pass.
3. Keep the existing `didRegister` / `didPerformStartup` guards — startup still runs exactly once.

### Fix 2 — Silence the Bonjour `EINVAL` chatter

Minimal fix first:

1. Move `bonjourAdvertiser.start(...)` out of the server-start `Task` and schedule it with `DispatchQueue.main.asyncAfter(deadline: .now() + 0.1)` after `appState.serverRunning = true`, so the HTTP listener on :8080 is fully established before mDNSResponder is told about it.
2. Add a defensive `txtDict = txtDict.filter { !$0.value.isEmpty }` in `BonjourAdvertiser.start(...)`.

If noise persists on macOS 15, fall back to migrating `BonjourAdvertiser` to `NWListener`-based Bonjour. Captured as an optional follow-up step.

### Why this approach

- Targets root timing causes, not symptoms.
- Preserves TKT-019's "server listening at launch" guarantee — a 100 ms defer on Bonjour is imperceptible.
- Keeps `BonjourAdvertiser`'s public API unchanged.
- Avoids a risky `NWListener` rewrite unless proven necessary.
