## Description

Every time the iOS companion is rebuilt and reinstalled (the user builds the
companion via RemoteDeploy itself, downloads the new `.ipa` over the air,
installs it, and relaunches), the app shows the pairing screen again -- the saved
connection is gone. Meanwhile the Mac never forgets: it keeps every paired-device
record forever, so each re-pair leaves an orphan behind. The live store had grown
to 12 records (a dozen `iPhone` / `Browser` entries, oldest from April), each with
its own still-valid token.

Root cause (two sides):

1. **Companion (phone):** the saved `{url, token, serverName}` blob lives in the
   iOS Keychain. A Keychain item only survives a reinstall if the new build lands
   in the same keychain access group (derived from the code-signing
   application-identifier). The companion target declares **no entitlements file
   at all** (`project.yml` set `CODE_SIGN_ENTITLEMENTS` only on the macOS targets),
   so it relies on the default access group. Across rebuild -> re-sign -> OTA
   install, the credential is not carrying over, so `KeychainStore.load()` returns
   nil and `ConnectionManager.restoreConnection()` bails to the pairing screen.

2. **Mac (host):** `PairingRouteHandler.pair()` always appends a new device record
   and never collapses repeats, so re-pairs pile up orphaned records whose tokens
   remain valid (a minor security smell as well as clutter).

Not caused by TKT-065 (that change is companion-only UI and never touches the
server, auth, or token storage).

## Reproduction Steps (bugs only)
1. Pair the iOS companion with the Mac.
2. Rebuild + reinstall the companion (OTA `.ipa` install) and relaunch.
3. Observe: the app asks to pair again.
4. On the Mac, the Devices list shows a new record each time; old ones remain.

## Acceptance Criteria
- [x] The companion has a stable, explicit `keychain-access-groups` entitlement so
      the saved credential survives reinstalls/re-signs (team id is stable, so the
      group `$(AppIdentifierPrefix)com.remotedeploy.companion` is stable).
- [x] On a successful pair, the Mac revokes older paired-device records that share
      the new device's name, so repeated re-pairs of the same phone collapse into a
      single record instead of accumulating orphans. The loopback
      "Menu bar (local)" record is never touched.
- [x] Existing pairing/auth tests still pass; a new test covers the dedupe.
- [x] Mac fix shippable via `scripts/ship-deploy.sh`; companion fix is built/
      deployed by the user via the app.

## Caveat (documented, not yet resolved)
If the OTA install **deletes** the old app before reinstalling (rather than
updating in place), iOS removes the app's Keychain items regardless of
entitlements, and the entitlement alone will not prevent one final re-pair. In
that case the durable fix is to make the install an in-place update (same
app-id, no delete). The Mac-side dedupe is the safety net either way: even when
the phone must re-pair, the Mac stays clean.

## Follow-up (post-deploy findings)
Testing the deployed build surfaced two things:

1. **Pairing-completion regression (FIXED).** The dedupe made the Mac's pairing
   dialog hang. `PairDeviceView` / `PairBrowserView` detected a completed pairing by
   a device-COUNT increase, but deduping a same-name re-pair revokes the old record
   as the new one is added, so the count stays flat and the dialog never completed.
   The phone pairs fine throughout -- only the Mac UI's completion signal was stuck.
   Fixed by detecting a NEW device id (set difference) instead of a count delta.

2. **Keychain entitlement was a no-op (keychain survival still UNRESOLVED).**
   `$(AppIdentifierPrefix)com.remotedeploy.companion` resolves to the same string as
   the implicit default access group (the `application-identifier` group), so making
   it explicit changed nothing -- the companion still re-pairs on every reinstall.
   The real cause is NOT the access group. Open options:
   - Instrument `KeychainStore.load()` to log the OSStatus / auth result on the next
     reinstall (definitive diagnosis: item-not-found vs auth-fail vs -34018).
   - Investigate whether the OTA install deletes the app vs updates in place; if it
     deletes, switch to in-place update OR iCloud-Keychain sync (synchronizable
     item -- trades the device-only posture for cross-reinstall persistence).
   - Accept re-pair as inherent to the rebuild loop and just keep it painless
     (this dedupe + the completion fix already do that).

3. **The keychain entitlement was actively HARMFUL -- REVERTED.** New symptom from
   testing: force-quitting + relaunching (no reinstall) also forgot the pairing.
   Root cause: the explicit `keychain-access-groups` entitlement adds a provisioning
   requirement. RemoteDeploy's own IPA export (`XcodeBuildEngine.generateExportOptions
   Plist`) uses automatic signing with NO `provisioningProfiles` and the archive runs
   without `-allowProvisioningUpdates`, so the existing profile -- which lacks
   keychain-access-groups -- is used as-is. The app is then signed with an entitlement
   its profile does not grant, so every keychain op fails at runtime with `-34018`
   (errSecMissingEntitlement). Nothing persists; the in-memory cache (`lockedCache`)
   masks it during a session, so it only surfaces on a cold start. Removed the
   entitlements file + the `CODE_SIGN_ENTITLEMENTS` line (revert to the implicit
   `application-identifier` group, which the profile DOES grant). This should restore
   force-quit persistence. The cross-REINSTALL forgetting (the original report) is a
   separate, still-open issue (see options above) -- reinstall can wipe the local
   keychain regardless of entitlements.

4. **ACTUAL ROOT CAUSE of force-quit-forgets: a bug in `hasStoredCredentials()`.**
   On-device diagnostic read `save=ok stored=false read=n/a` -- i.e. the keychain
   WRITE succeeded, the item was never re-read, yet the existence check said
   "none". The "key fix" (ecac9fb) gated restore behind `hasStoredCredentials()`,
   which queried with `kSecUseAuthenticationUISkip`. That flag does NOT report an
   auth-gated item as "exists" -- it SILENTLY OMITS any `.userPresence`-protected
   item from results, so the query returned `errSecItemNotFound`. Net: the check
   always returned false for the real (Face-ID-gated) credential, so
   `restoreConnectionIfNeeded()` short-circuited, `load()` (and thus the Face ID
   prompt) never ran, and the app re-paired on every cold start. The vanished Face
   ID prompt was the tell. Fix: check existence by requesting `kSecReturnAttributes`
   (readable without auth even for ACL items) with `kSecUseAuthenticationUIFail`,
   instead of the omit-on-auth `...UISkip`. Surgical -- the write, the `.userPresence`
   gate (TKT-022), and the scene-phase restore are all left intact.
