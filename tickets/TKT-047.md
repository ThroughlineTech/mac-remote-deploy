## Description

Regression introduced by TKT-046 (which wrapped `BuildLogStreamView` in a collapsed-by-default `DisclosureGroup`).

When the user is on the Build tab and the Build Log disclosure is in its collapsed state (caret/chevron showing, log hidden), **the app effectively freezes**:

1. The tab bar at the bottom (Projects / Build / Installs / Settings) does not respond to taps — the user cannot navigate away from the Build tab.
2. The screen layout is also visibly off: a large empty band sits between the navigation title "Build" at the top and the project picker / Build & Deploy button, which have been pushed roughly to the vertical center of the screen. (See attached screenshot.) This suggests the spacer / `.frame(maxHeight: .infinity)` arrangement added in TKT-046 to anchor the controls is fighting some other layout constraint.

Hypothesis: the conditional `Spacer()` added after the disclosure (only when collapsed) is interacting with the `NavigationStack` / outer `VStack` in a way that either (a) inserts a hit-test-blocking transparent layer over the tab bar, or (b) causes a SwiftUI layout / state-update loop that hangs the main thread. The "kinda freezes" symptom — UI visible but unresponsive to taps — is consistent with either.

This is high-priority because **the only escape is force-quitting the app**: the user cannot tap any other tab. The tab bar being unreachable is a much worse UX than the original "doom scroll" problem TKT-046 set out to fix.

Expanding the disclosure (if even possible from a frozen state) may also be affected; needs to be checked during investigation.

## Reproduction Steps (bugs only)

1. Launch the companion app (post-TKT-046, i.e. latest `main`).
2. Tap the **Build** tab.
3. Observe: the Build Log disclosure renders collapsed by default (correct per TKT-046's intent).
4. Try to tap **Projects**, **Installs**, or **Settings** in the bottom tab bar.
5. Expected: the app navigates to the tapped tab.
6. Observed: the tap is ignored. The user is stuck on the Build tab. Force-quit is the only way out.

(See attached screenshot showing the unusual blank vertical band between the "Build" title and the project picker, along with a collapsed "Build Log" row.)

## Acceptance Criteria

- [ ] On the Build tab with the Build Log collapsed, the bottom tab bar responds to taps and the user can navigate away to any other tab without delay.
- [ ] On the Build tab with the Build Log collapsed, the project picker and Build & Deploy button sit at their natural position (top of the content area, just below the navigation title "Build") — not pushed to the vertical middle by a runaway spacer.
- [ ] Expanding the Build Log disclosure still works and the log fills the remaining vertical space below the controls (TKT-046's expanded behavior is preserved).
- [ ] Collapsing the disclosure again returns to the responsive, top-anchored state.
- [ ] The fix does not re-introduce the original TKT-046 problem (log expanded by default, doom scrolling, auto-scroll past terminal state).
- [ ] No main-thread hang is observable in Instruments / Time Profiler when the disclosure is collapsed and the user attempts to interact with other UI elements.

## Investigation

### TKT-046 changes (the regression)

Commit `3e50530` ("TKT-046: collapse build log by default and stop auto-scroll on terminal") rewrote the log area of `BuildControlView` from a plain `BuildLogStreamView()` to a `DisclosureGroup` wrapper, and added a conditional trailing `Spacer()`. The relevant additions, from `RemoteDeployCompanion/Views/Build/BuildControlView.swift` (current state after TKT-046, lines 130–157):

```swift
Divider()

// Build log area — collapsed by default (TKT-046).
DisclosureGroup(isExpanded: $isLogExpanded) {
    BuildLogStreamView()
        .frame(maxHeight: .infinity)          // <-- greedy child #1
} label: {
    HStack(spacing: 6) {
        Image(systemName: "doc.text.magnifyingglass")
        Text("Build Log") ...
        Spacer()
    }
}
.padding(.horizontal)
.padding(.vertical, 8)

// Push everything to the top when the log disclosure is
// collapsed (TKT-046).
if !isLogExpanded {
    Spacer()                                  // <-- greedy child #2 (conditional)
}
```

Note also the outer structure (line 26–27): `NavigationStack { VStack(spacing: 0) { ... } }`. The controls section `VStack(spacing: 12) { ... }.padding()` (line 29) has no frame / no alignment — it's just a natural-size child of the outer `VStack(spacing: 0)`.

### Root cause

There are TWO greedy vertical-space claimants stacked in the same `VStack(spacing: 0)` when the disclosure is collapsed:

1. **`BuildLogStreamView().frame(maxHeight: .infinity)` inside the `DisclosureGroup` content closure.** `DisclosureGroup` on iOS does **not** remove its content from the layout tree when collapsed — it animates the content's effective reveal but still participates in parent layout measurement, and SwiftUI's `maxHeight: .infinity` on the content propagates up as a preference for "take all remaining space." In practice this means the collapsed `DisclosureGroup` still claims a very tall (effectively infinite) frame in the parent `VStack`, even though only the label row is visible. The `ScrollView` inside `BuildLogStreamView` sits underneath — invisible but still a real, hit-testing view — and extends down over the tab bar. That is exactly the "tab bar ignores taps" symptom: taps land on the 0-or-near-0-height-but-actually-tall ScrollView instead of the `TabView`'s bar. Force-quit is the only escape because no tap can reach another tab item.

2. **The conditional trailing `Spacer()`** — `if !isLogExpanded { Spacer() }` on line 155. When collapsed, SwiftUI now sees two greedy children in the same `VStack(spacing: 0)`: the effectively-infinite `DisclosureGroup` content frame **and** the `Spacer()`. SwiftUI resolves that by splitting remaining space between them. The `Spacer()` wins the portion above the disclosure's measurement, pushing the fixed-size controls `VStack` down into roughly the vertical center — which matches the screenshot exactly (empty band above the picker, Build & Deploy in the middle, more empty space below). If the `DisclosureGroup` content were genuinely 0-height when collapsed (as it would be with a plain `if isLogExpanded` swap), the trailing `Spacer()` alone would simply push the controls to the top and the empty band would be below — the observed above-the-controls band is the tell that a second greedy sibling exists.

Both symptoms (freeze + layout-to-center) share a root cause: **`.frame(maxHeight: .infinity)` on a child inside a collapsed `DisclosureGroup` is a layout antipattern on iOS.** The trailing conditional `Spacer()` made it observable by competing for the same space and pushing the fixed controls into the middle; without the conditional `Spacer()`, the same invisible ScrollView overlay would still cover the tab bar and still block taps, but the controls would sit at the top, so the bug would be "tab bar frozen" without the "shoved to middle" visual. They're two faces of the same SwiftUI layout bug.

A secondary contributing factor: the disclosure group's `BuildLogStreamView` contains a `ScrollView { LazyVStack { ... } }` backed by `webSocketClient.buildLogLines` — even with 0 visible frame, its hit-test layer is a full `ScrollView`, which very aggressively consumes touches. Any tap that lands within its reported frame — which here is "everything below the Build Log label down to the bottom of the screen, including the tab bar area" — is swallowed.

### Why the other tabs don't have this

- `ProjectListView` (`RemoteDeployCompanion/Views/Projects/ProjectListView.swift:20`) uses `NavigationStack { Group { ... } }` where the `Group` resolves to either a `ProgressView`, `ContentUnavailableView`, or a `List` — none of these are greedy-inside-a-disclosure.
- `InstallHistoryView` (`RemoteDeployCompanion/Views/History/InstallHistoryView.swift:14`) uses the same `NavigationStack { Group { ... } }` + `List` shape.
- `RemoteSettingsView` follows the same single-child shape (Form/List).

None of them wrap a greedy child inside a collapsed `DisclosureGroup` or use an explicit outer `VStack(spacing: 0)` with a conditional trailing `Spacer()`. `BuildControlView` is unique in using a multi-section `VStack(spacing: 0)` with a collapsible log section, and that's exactly where the TKT-046 change collided with SwiftUI's layout rules.

### Existing UI tests

- `RemoteDeployCompanionUITests/ProjectDetailReactiveStatusUITests.swift` is the TKT-044 regression test. It uses `XCUIApplication`, `app.launchArguments = ["--auto-pair", ...]`, and `tapTab("Projects", in: app)` via `app.tabBars.buttons[name]` — so the harness for tab-bar interaction already exists. There is **no** existing test that covers the Build tab with a collapsed log or asserts that all four tabs remain tappable from the Build tab.
- `RemoteDeployCompanionUITests/ScreenshotTests.swift` is marketing/screenshot-only, not a regression net.

Adding a tab-bar-reachability UI test is feasible with the existing `--auto-pair` launch argument + `tapTab` helper.

### Regression Risk (for the proposed fix): **Low**

The fix is a local rewrite of `BuildControlView.body`'s log section only, touching `~25` lines. It restores the TKT-046 behaviour (collapsed-by-default log, collapse-on-new-build, line-count badge) while swapping a `DisclosureGroup` for an explicit `if`/`else` conditional render that SwiftUI layouts predictably. No API change to `BuildLogStreamView`, no change to `BuildManager`, no WebSocket or state plumbing changes. The only possible regression surface is animation polish — `DisclosureGroup` animates its reveal automatically; the `if`/`else` approach will animate only if wrapped in `withAnimation { ... }` on toggle. That's a minor UX concern, not a correctness one, and is covered by the plan.


## Proposed Solution

Replace the `DisclosureGroup` + `.frame(maxHeight: .infinity)` + conditional trailing `Spacer()` pattern with an **explicit `if`/`else` conditional render** of the log area, plus a regular toggle button as the header. The key idea: when collapsed, the log view is **not in the view tree at all** (not just hidden), so it cannot claim vertical space and cannot hit-test over the tab bar. When expanded, the log view is the only greedy child and legitimately fills the remaining space.

Sketch of the target layout:

```swift
VStack(spacing: 0) {
    // Controls section (unchanged) — fixed natural height
    VStack(spacing: 12) { ... }.padding()

    Divider()

    // Build Log header — always a tappable row, no DisclosureGroup.
    Button {
        withAnimation(.easeInOut(duration: 0.2)) {
            isLogExpanded.toggle()
        }
    } label: {
        HStack(spacing: 6) {
            Image(systemName: isLogExpanded ? "chevron.down" : "chevron.right")
            Image(systemName: "doc.text.magnifyingglass")
            Text("Build Log").font(.subheadline).fontWeight(.medium)
            if !webSocketClient.buildLogLines.isEmpty {
                Text("(\(webSocketClient.buildLogLines.count) lines)")
                    .font(.caption).foregroundColor(.secondary)
            }
            Spacer()
        }
        .contentShape(Rectangle())
    }
    .buttonStyle(.plain)
    .padding(.horizontal)
    .padding(.vertical, 10)

    if isLogExpanded {
        BuildLogStreamView()
            .frame(maxHeight: .infinity)
    } else {
        Spacer(minLength: 0)
    }
}
```

Why this over the alternatives:

- **Why not `.frame(maxHeight: isLogExpanded ? .infinity : nil)` on the `DisclosureGroup` itself?** Still uses `DisclosureGroup`, which still has iOS's "content participates in layout even when collapsed" quirk. Even a bounded outer frame doesn't guarantee the inner `ScrollView` stops hit-testing over the tab bar.
- **Why not `.fixedSize(horizontal: false, vertical: true)` on the `DisclosureGroup`?** Would likely fix the layout centering but not the hit-test overlay, because `.fixedSize` is a measurement hint, not a clip.
- **Why not `.clipped()` on the `DisclosureGroup`?** `.clipped()` clips drawing, NOT hit-testing. The invisible ScrollView would still swallow taps.
- **Why not keep `DisclosureGroup` and drop just `.frame(maxHeight: .infinity)`?** Then when expanded, the log can't grow to fill — reverting to the TKT-017/043 "fixed-height tail" behaviour, which defeats TKT-046's expanded-view intent.
- **Why the explicit `if`/`else` wins:** SwiftUI's conditional rendering fully instantiates or fully removes the view from the tree. No layout participation, no hit-test participation, no half-animated reveal. Exactly `Spacer()` when collapsed → controls pinned to top. Exactly `BuildLogStreamView().frame(maxHeight: .infinity)` when expanded → log fills remaining space. Predictable, testable, matches the layout pattern already used elsewhere in the app.

The custom header row is a 1:1 UX replacement: same icon, same label, same line-count badge, chevron direction driven by state. Wrapping the toggle in `withAnimation` preserves the TKT-046 reveal animation. Collapse-on-new-build (`isLogExpanded = false` inside `.onChange(of: buildManager.isBuilding)`) is untouched.

## Files Changed

- `RemoteDeployCompanion/Views/Build/BuildControlView.swift` — replaced the collapsed-by-default `DisclosureGroup` (whose inner `.frame(maxHeight: .infinity)` + conditional trailing `Spacer()` was swallowing touches over the tab bar) with an explicit `Button` toggle header plus an `if isLogExpanded { BuildLogStreamView().frame(maxHeight: .infinity) } else { Spacer(minLength: 0) }` conditional render. The button header is animated (`withAnimation(.easeInOut(duration: 0.2))`), uses `chevron.down` / `chevron.right` for the disclosure affordance, preserves the magnifying-glass icon / "Build Log" label / line-count badge, uses `.buttonStyle(.plain)` + `.contentShape(Rectangle())` so the whole row is tappable, and carries `accessibilityIdentifier("BuildLogToggle")` for UI testing. The existing `.onChange(of: buildManager.isBuilding)` handler (collapse-on-new-build, build-start timestamp) is unchanged. `BuildLogStreamView`'s internal auto-scroll guard (`guard buildManager.isBuilding else { return }`) and its snap-to-tail `.onChange(of: buildManager.isBuilding)` are untouched.
- `RemoteDeployCompanionUITests/BuildTabTabBarReachabilityUITests.swift` — new UI regression test with two cases: (1) `test_buildTab_collapsedLog_allowsTabBarNavigation` launches with `--auto-pair`, navigates to the Build tab, asserts the log toggle + Build & Deploy button are visible, then taps Projects and Installs and asserts each tab bar navigation actually succeeded; (2) `test_buildTab_expandThenCollapse_preservesTabBarReachability` does an expand/collapse round-trip on the Build Log and then asserts Settings tab navigation still works. Helpers (`waitForConnectedUI`, `tapTab`) are inline copies of the `ProjectDetailReactiveStatusUITests` pattern. No `project.yml` change required — the UI test target uses a directory-based source list, so the new file is picked up automatically by `xcodegen generate`.
