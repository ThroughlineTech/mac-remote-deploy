## Description

The web PWA's WebSocket connection cannot authenticate, so the browser's live
build log and live build-status silently never stream. The build still triggers
(the POST response gives the initial status), but no `buildlog` / `buildstatus`
frames arrive, and the connection reconnect-loops in the background every 3s.

This has been broken since WS bearer auth was added (TKT-011 / TKT-024). The iOS
companion is unaffected because it sets the header on its own `URLRequest`; the
browser never could. It is out of scope for Phase 4 (browser pairing, TKT-057),
but it is the gap between "a browser can pair and browse" (now true) and "a
browser is a true first-class client with live logs" (not yet).

## Root cause (verified at this SHA)

The WS upgrade handshake is authenticated by the `Authorization` header only, and
browsers cannot set that header on a WebSocket handshake (the `WebSocket`
constructor takes only `url` and `protocols`).

- `NIODeployServer.startHTTPS` `shouldUpgrade` passes only `head.headers` to the
  authenticator and discards the query string
  (`head.uri.split(separator: "?").first`); `Sec-WebSocket-Protocol` is never read.
  (`RemoteDeploy/Services/NIODeployServer.swift`, ~line 283-296)
- `webSocketAuthenticator` is `auth.authenticate(headers:)`
  (`RemoteDeploy/AppDelegate.swift`, ~line 585).
- `AuthMiddleware.authenticate(headers:)` matches only a
  `Authorization: Bearer <token>` header (`RemoteDeploy/API/AuthMiddleware.swift`).
- The PWA opens `new WebSocket(.../api/v1/ws)` with no credentials
  (`RemoteDeploy/Resources/pwa/app.js`, `connectWS`), so the upgrade is rejected,
  `HTTPHandler` returns 401 for `/api/v1/ws`, and `ws.onclose` retries every 3s.

## Proposed solution

Accept a browser-settable credential channel in addition to the existing
`Authorization` header (keep the header path so the iOS client is unchanged).

Preferred: the `Sec-WebSocket-Protocol` subprotocol, which the browser CAN set via
`new WebSocket(url, ['bearer', token])`. The raw token is 8 hex chars, a valid
subprotocol token. This keeps the credential out of the URL (so it is not exposed
in `GET /api/v1/ws` access-log lines).

1. `app.js` `connectWS()`: `new WebSocket(url, ['bearer', state.token])`.
2. `AuthMiddleware`: add a path that, when no valid `Authorization` header is
   present, parses `Sec-WebSocket-Protocol` (comma-separated), and if it contains
   the `bearer` marker, validates the accompanying token value the same way
   (SHA-256 -> device lookup).
3. `NIODeployServer` `shouldUpgrade`: on success, echo the selected subprotocol
   back in the 101 response headers (`Sec-WebSocket-Protocol: bearer`) -- some
   browsers fail the handshake if the server does not confirm a requested
   subprotocol.

Alternative considered: `?token=<raw>` query param (simpler, but leaks the token
into URLs/logs; would also require not logging the query for `/api/v1/ws`).

## Implementation (done)

Took the preferred `Sec-WebSocket-Protocol` approach (token stays out of URLs/logs).

- `RemoteDeploy/API/AuthMiddleware.swift` - added `authenticateWebSocket(headers:)`:
  tries the `Authorization` header first (native/iOS unchanged), then falls back
  to a `bearer, <token>` value in `Sec-WebSocket-Protocol`. Extracted a private
  `validate(rawToken:)` shared by both paths; the REST `authenticate(headers:)`
  still ignores the subprotocol.
- `RemoteDeploy/AppDelegate.swift` - WS authenticator now calls
  `authenticateWebSocket`.
- `RemoteDeploy/Services/NIODeployServer.swift` - `shouldUpgrade` echoes
  `Sec-WebSocket-Protocol: bearer` in the 101 response when offered (URLSession
  and browsers fail the handshake unless the server confirms an offered subprotocol).
- `RemoteDeploy/Resources/pwa/app.js` - `connectWS()` opens
  `new WebSocket(url, ['bearer', state.token])`.
- Tests: new `RemoteDeployTests/AuthMiddlewareWebSocketTests.swift` (7 cases,
  deterministic); updated `WebSocketUpgradeTests` to wire the production WS
  authenticator and added `testSubprotocolUpgradeAcceptsAndBroadcastsFlow` (no
  Authorization header, token via subprotocol -> upgrade + broadcast received).
