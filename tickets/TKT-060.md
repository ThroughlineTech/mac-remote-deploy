## Description

Phase 6 of the backend-decoupling plan (docs/backend-decoupling-plan.md). Run the
backend (NIO server + BuildCoordinator + stores) as a separate headless
LaunchAgent process; the menu bar app becomes a pure client process. Done when
the backend keeps serving and building for web/iOS clients with the menu bar app
not running, and the menu bar reconnects on relaunch.

This phase is marked optional/gated in the plan. The gate (run the server
independently of the menu bar's lifecycle) has been chosen explicitly. The macOS
constraint still applies: both processes run as per-user LaunchAgents in the Aqua
login session (no pre-login daemon, no headless/background operation) -- the
payoff is lifecycle decoupling, not background operation.

## Investigation finding (scope correction)

The plan's Phase 6 sketch (3 steps: extract core, give it a main, ship menu bar
as a client) assumes Phase 3 fully turned the menu bar into an API client. It did
not. Phase 3 client-ified the main surfaces only:

- DECOUPLED (go through MenuBarClient/APIClient already): ServerStatusSection,
  BuildControlsSection, BuildLogView, ProjectsListSection (data), Settings tabs
  (Server/Projects/Push/Devices-list), menu bar icon/status.
- STILL IN-PROCESS (reach into ServiceContainer's server-side objects directly,
  will not exist in the menu bar process after the split):
  - Setup wizard: TailscaleSetupStep (`tailscaleProvider`), CertificateSetupStep
    (`tailscaleProvider.generateCertificate`, `certificateProvider`),
    ProjectSetupStep + ProjectFormView (`buildEngine.detectSchemes`,
    `projectStore.save`), SetupAssistantView (`configurePushNotifiers`).
  - Pairing: PairDeviceView / PairBrowserView (`pairingHandler.registerPendingToken`,
    poll `pairedDeviceStore.device(forTokenHash:)`).
  - UtilitiesSection: `ipaImporter.importIPA` (writes into the serve dir);
    `notificationManager` (desktop notifications -- inherently local, stays).

Existing REST contract (Packages/RemoteDeployShared/.../API/APIEndpoint.swift):
pair/unpair, status, projects CRUD, build trigger/status/cancel/history,
installs (list + delete + clear-all), settings get/update, filesystem
browse + schemes, devices list/revoke, ws. MISSING for the still-in-process
flows above:
1. Mint a pending pairing token (Mac-side "Pair Browser/Device" action).
2. Tailscale cert generation (setup wizard step 2).
3. IPA upload (UtilitiesSection import, if it must work cross-process).

`APIClient` and `WebSocketClient` already live in the shared package, so a
separate client process can use them as-is.

## Lifecycle that must move to the headless process (from AppDelegate)

AppDelegate currently owns all of: BuildManager.configure, BuildCoordinator
construction, settings load + `applySettingsFromStore`, project load +
`refreshProjectsFromStore`, push-notifier config, Tailscale status poll,
`prepareServer` (API router + :8080 listener), `startServer`/HTTPS +
`reconcileHTTPS`, Bonjour, the `.projectsDidChange`/`.settingsDidChange`/
`.restartServerRequested` reconcilers, loopback-token minting
(`configureMenuBarClient`), and `applicationWillTerminate` server stop. All the
server-owning parts move into the headless target's entry point; the menu bar
keeps only client-side concerns (MenuBarClient config against loopback,
notification permission, window/UI lifecycle).

Cross-process note: settings/project writes already route through the server's
API, so the server's own `SettingsStore.update`/`projectStore` writes post the
in-process `.settingsDidChange`/`.projectsDidChange` that drive HTTPS reconcile
and slug-registry resync -- those reconcilers just need to live in the server
process. The menu bar's old in-process reconcilers become unnecessary (it polls).

## Proposed staged plan (each stage builds + tests green)

- Stage A (server, additive, low risk): add the 3 missing endpoints + handlers
  + tests. No client changes; nothing breaks.
- Stage B (client-ification): rewire setup wizard, pairing views, and IPA import
  to the API via MenuBarClient/APIClient; remove direct server-object reads.
  Still single-process, fully testable end to end before splitting.
- Stage C (target split): add the headless `RemoteDeployServer` target in
  project.yml with its own entry point holding the moved lifecycle; the menu bar
  target drops the server-side files. `xcodegen generate`; build both.
- Stage D (packaging): two LaunchAgent plists / login item; parameterize
  build-release.sh + deploy.sh + ship-deploy.sh allowlist for two products.
- Stage E (verification): exercise the human ship gate.

## Open decisions
- IPA import in the split: build an upload endpoint (Stage A) vs. keep it local /
  disable when not co-located. (Recommend: upload endpoint for true parity, but
  it is the lowest-value piece -- can be deferred.)
- Bundle IDs / product names for the two products and login-item registration
  mechanism (SMAppService vs. LaunchAgent for the menu bar).

## Gotchas
- project.yml is XcodeGen-managed; run `xcodegen generate` after target changes;
  never hand-edit the .xcodeproj. Resources/pwa/** is a folder reference.
- Two processes must not both bind :8080/:8443 -- only the server binds.
- Cert/keychain: the agent runs in the same login session and inherits the login
  keychain; verify codesign/notarization still works from the agent.
- Keep source files under ~300 lines; factor first if a change pushes past that.
- ASCII only.

## Stage C blueprint (target split) -- executable next step

Menu bar target needs (small, well-bounded set):
- RemoteDeployApp.swift (rewrite: slim @main; drop the server-owning
  ServiceContainer; keep AppState + MenuBarClient + a NotificationManager).
- Views/** (21 files), Models/AppState.swift, Services/MenuBarClient.swift.
- Local helpers also needed by the menu bar: Services/NotificationManager.swift,
  Services/QRCodeGenerator.swift, Services/{Prowl,Pushover,Ntfy}Notifier.swift
  (+ their PushNotifying protocol), Logging.swift, RemoteDeployError.
- The shared package already provides APIClient/WebSocketClient + all data models.

Server target `RemoteDeployServer` (headless, LSUIElement) needs everything else:
API/**, Managers/BuildManager, Services server-side (NIODeployServer + handler +
response generator, stores, engines, providers, BuildCoordinator, Bonjour,
Tailscale, cert provider, ipaImporter, settings/runtime/install stores,
ProcessRunner, manifest/install-page generators, EnvironmentChecker,
LocalDeployManager) + Protocols/** + a new headless entry point.

Files compiled into BOTH targets (AppKit-bound, can't go in the cross-platform
package): Logging, NotificationManager, push notifiers, QRCodeGenerator,
RemoteDeployError, AppState. List them in both targets' `sources` in project.yml.

Headless lifecycle: extract AppDelegate's server startup (configure
BuildManager+Coordinator, loadSettings, refreshProjectsFromStore,
configurePushNotifiers, checkTailscaleStatus, prepareServer, startServer,
reconcileHTTPS, status poll, Bonjour, the .projectsDidChange/.settingsDidChange
reconcilers, applicationWillTerminate stop) into the server target's entry
point. Drop the UI bits (appState-as-projection, menuBarClient). The server
keeps an AppState-like config holder (AppState can be reused as a plain config
object) for serverPort/cert/hostname that BuildCoordinator + startServer read.

Cross-process loopback token handoff: the SERVER mints the loopback token at
startup (as configureMenuBarClient does today), writes its hash to
paired_devices.json (already does), AND writes the RAW token to
`~/Library/Application Support/RemoteDeploy/loopback_token` (0600). The menu bar
reads that file at startup and configures MenuBarClient(baseURL: 127.0.0.1:8080,
token: rawToken); if absent (server not up yet), poll/retry. This avoids needing
the server's token-hashing in the menu bar.

project.yml: add target `RemoteDeployServer` (application, macOS, LSUIElement,
bundle id com.remotedeploy.server, own Info.plist + entitlements -- network
server/client + user-selected files; verify codesign/keychain from the agent)
and a scheme. Keep `RemoteDeploy` as the menu bar client (LSUIElement, network
client only). Run `xcodegen generate`. Gate: BOTH targets build; full test
suite green (tests target the server target).
