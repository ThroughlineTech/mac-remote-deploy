## Description

The 2026-04-08 shipped-ticket audit ([tickets/ANOMALIES-2026-04-08.md](tickets/ANOMALIES-2026-04-08.md)) surfaced six shipped tickets whose acceptance criteria or implementation plans were not fully met. Rather than re-opening each individually, this ticket batches all six cleanups into a single coordinated cleanup pass.

The six anomalies, each to be addressed as its own sub-task within this one ticket:

1. **TKT-008** — `JSONBuildHistoryStore` is wired up and functional but has no unit tests. AC required coverage for `append`, cap/trim behavior, `load`, and query.
2. **TKT-009** — Bundle-ID regex validation lives only in the SwiftUI `ProjectSetupStep`. `DeferredSettingsUpdater` does not validate the bundle ID, so a programmatic `PUT /api/v1/settings` can persist a malformed value.
3. **TKT-011** — `WebSocketManager` has unit tests but is dead code: `NIODeployServer` never installs a `WebSocketUpgrader`/`WebSocketChannelHandler`, no bearer-token auth on the upgrade path, and the iOS client has no reconnect-with-backoff.
4. **TKT-012** — The MenuBarView decomposition landed with 4 of 5 promised subview files (missing `ServerStatusSection.swift`) and `MenuBarView.swift` is 112 lines vs. the AC's <100.
5. **TKT-014** — Setup Assistant validation is missing the "scheme not empty" check. `ProjectSetupStep` saves `project.scheme = selectedScheme` unconditionally with no `schemeError` state.
6. **TKT-019** — The startup-via-AppDelegate fix works in production, but the promised focused unit test (`RemoteDeployTests/AppDelegateStartupTests.swift`) was never written.
7. **TKT-021** — Code changes landed and merged, but the acceptance criteria (no `_NSDetectedLayoutRecursion` warning, no `send failed: Invalid argument ×2` in console) were never validated on device. The commit message explicitly deferred manual verification ("required on device after landing") and the ticket was marked shipped anyway. The fix is plausible but unconfirmed. Ticket body's `Files Changed`, `Test Report`, and `Verification Checklist` sections are also empty.

Investigation should produce a concrete per-item plan (files to touch, tests to add, order of operations) so implementation can run as a single batched cleanup PR rather than six ad-hoc fixes.

## Investigation

Each sub-item verified by reading the current code. File references are to `main` as of 2026-04-08.

### 1. TKT-008 — `JSONBuildHistoryStore` has no unit tests
- [RemoteDeploy/Services/JSONBuildHistoryStore.swift](RemoteDeploy/Services/JSONBuildHistoryStore.swift) implements `append`, `recentBuilds`, in-memory caching behind `OSAllocatedUnfairLock`, disk persistence under `~/Library/Application Support/RemoteDeploy/build-history.json`, and trimming at `maxRecords = 100`. Init accepts a `directory: URL?` override, which is the hook tests will use.
- `loadFromDisk()` already swallows decode failures and returns `[]`, so the "corrupt file" path is test-friendly.
- [RemoteDeployTests/](RemoteDeployTests/) only contains `Mocks/MockBuildHistoryProvider.swift` — no file exercising the JSON store itself. Confirms the audit.
- Regression risk: **low**. New test file, no production code change.

### 2. TKT-009 — Bundle-ID validation duplicated / missing at the programmatic boundary
- The regex lives only in [RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift:38-45](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift#L38-L45).
- **Important correction to the ticket premise:** `SettingsData` ([Packages/RemoteDeployShared/.../Models/SettingsData.swift](Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/SettingsData.swift)) has no `bundleID` field — it carries `serverPort`, `hostname`, `certPath`, `keyPath`, `pushNotificationConfig`. Bundle ID is a **per-project** value on `ProjectConfig` ([Packages/RemoteDeployShared/.../Models/ProjectConfig.swift:25](Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/ProjectConfig.swift#L25)). So `DeferredSettingsUpdater.updateSettings(_:)` literally cannot validate a bundle ID — the concept doesn't exist in its input.
- The *programmatic* entry point for a bundle ID is [RemoteDeploy/API/Routes/ProjectsRouteHandler.swift](RemoteDeploy/API/Routes/ProjectsRouteHandler.swift), which owns `POST /api/v1/projects` and `PUT /api/v1/projects/:id`. That is the real gap TKT-009's AC was trying to close — the original ticket author conflated `DeferredSettingsUpdater` with "the programmatic settings path".
- Plan will therefore: (a) extract one shared `BundleIDValidator` into `RemoteDeployShared`, (b) use it from `ProjectSetupStep`, (c) use it from `ProjectsRouteHandler` on create + update, and (d) add tests at both `ProjectsRouteHandler` and `DeferredSettingsUpdaterTests` levels. The `DeferredSettingsUpdaterTests` contribution will be a single regression test documenting that settings updates are never expected to carry a bundle ID (captured via a comment + empty-field assertion), so the AC line is satisfied without inventing a non-existent field. This deviation from the literal AC will be flagged clearly in the Test Report.
- Regression risk: **medium**. Touches the REST API contract — any companion version submitting a slightly malformed bundle ID (trailing dot, digit-first segment) will now get a 400 where previously it got 200.

### 3. TKT-011 — WebSocket not wired in, no auth, no client reconnect
- [RemoteDeploy/Services/NIODeployServer.swift:178-184](RemoteDeploy/Services/NIODeployServer.swift#L178-L184) installs `NIOSSLServerHandler` → `configureHTTPServerPipeline` → `HTTPHandler`. There is no `NIOWebSocketServerUpgrader` in the `configureHTTPServerPipeline(withServerUpgrade:)` call. The `HTTP` listener at :8080 (line 197-208) is also upgrade-less.
- `webSocketManager` is instantiated at [NIODeployServer.swift:53](RemoteDeploy/Services/NIODeployServer.swift#L53) but has no reader in `HTTPHandler`/`NIOResponseGenerator` — dead code confirmed.
- Bearer-token auth for REST routes is in `AuthMiddleware` — the upgrade path needs to call the same paired-device token check *before* responding `101 Switching Protocols`. The cleanest hook is the `shouldUpgrade` callback on `NIOWebSocketServerUpgrader`, which receives the HTTP request head and can read `Authorization: Bearer ...`.
- The companion's [RemoteDeployCompanion/Services/WebSocketClient.swift](RemoteDeployCompanion/Services/WebSocketClient.swift) `connect` has no reconnect-with-backoff — on `.failure` it sets `isConnected = false` and stops. Needs a capped exponential backoff loop (e.g. 1s, 2s, 4s, 8s, 16s cap) gated on an `isActive` flag set at `connect`/`disconnect`.
- Existing `WebSocketManagerTests.swift` explicitly flags the integration as deferred ([WebSocketManagerTests.swift:7-12](RemoteDeployTests/WebSocketManagerTests.swift#L7-L12)).
- `RemoteDeployIntegrationTests/` is the right home for the upgrade/auth integration test (spawns a real `NIODeployServer` with a self-signed cert — see `APIRouterFactoryTests.swift` for the pattern).
- Regression risk: **medium-high**. Installing a `WebSocketUpgrader` changes the HTTPS pipeline for every request, and a bad `shouldUpgrade` can regress the REST paths. Must explicitly verify existing REST integration tests still pass.

### 4. TKT-012 — `MenuBarView` still 112 lines, `ServerStatusSection.swift` missing
- Subview files present in [RemoteDeploy/Views/MenuBar/](RemoteDeploy/Views/MenuBar/): `MenuBarHeaderSection.swift`, `ProjectsListSection.swift`, `BuildControlsSection.swift`, `UtilitiesSection.swift`. `ServerStatusSection.swift` is missing.
- [MenuBarView.swift](RemoteDeploy/Views/MenuBarView.swift) is 112 lines. The overage comes from: (a) the inline `struct ProjectRowView` helper at lines 83-112, and (b) the boundary-error `.alert(...)` modifier at lines 36-55, and (c) `handleAPIBuildRequest` at lines 58-80.
- Interesting wrinkle: there is no "server status row" currently rendered inside `MenuBarView` — server/Tailscale status appears inside `MenuBarHeaderSection`. So `ServerStatusSection.swift` as named in the AC either (a) should be extracted out of `MenuBarHeaderSection` to reach the 5-file count, or (b) should own the "server running / stopped / error" row that's arguably part of header. Cleanest read of the AC: split header into *account/hostname* (stays in `MenuBarHeaderSection`) + *server status* (new `ServerStatusSection`). This also drops the header below its current size and keeps `MenuBarView` slimmer.
- To hit <100 lines in `MenuBarView.swift`, also extract `ProjectRowView` into its own `ProjectRowView.swift` under `MenuBar/`, and extract `handleAPIBuildRequest` into a small `MenuBarBuildDispatcher` helper (or move it onto `BuildManager` as a `triggerAPIBuild(notification:)` convenience).
- Regression risk: **low**. Pure view decomposition — covered implicitly by the app compiling.

### 5. TKT-014 — Scheme empty-check missing
- [ProjectSetupStep.swift:158-184](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift#L158-L184) renders the scheme picker, but there is no `schemeError` state and `saveProject()` at [line 367-370](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift#L367-L370) unconditionally assigns `project.scheme = selectedScheme`. If detection returns zero schemes the user sees a "No schemes detected / Retry" label but the step will still allow advancing and persist an empty scheme.
- Fix pattern: mirror `bundleIDError` — add `@State private var schemeError: String?`, populate it when `detectedSchemes.isEmpty` or `selectedScheme.isEmpty`, render an inline `.red` caption, and short-circuit `saveProject()` on `schemeError != nil`.
- Testable surface: the three `validate*` helpers are private `func`s on `ProjectSetupStep`. Easiest is to lift them into a free `ProjectSetupValidators` namespace (or `enum ProjectSetupValidators`) so `RemoteDeployTests` can cover them directly without instantiating the SwiftUI view. A validator test is a one-line addition for the empty-scheme case.
- Regression risk: **low**.

### 6. TKT-019 — Promised `AppDelegateStartupTests.swift` never landed
- [AppDelegate.swift](RemoteDeploy/AppDelegate.swift) implementation is correct: `didRegister` + `didPerformStartup` guards, both `register()` and `applicationDidFinishLaunching()` dispatch startup if the other has already run, and `performStartup()` bails if `didPerformStartup` is already true.
- No `RemoteDeployTests/AppDelegateStartupTests.swift` exists.
- The test can exercise `AppDelegate` directly — it's `@MainActor`, its state dependencies (`AppState`, `ServiceContainer`, `BuildManager`) are `ObservableObject`s you can instantiate standalone, and `performStartup()` is currently `private`. Simplest approach: expose a test-only `func testPerformStartup() async { await performStartup() }` or make `performStartup()` `internal` — second option is cleaner and only affects access level. Even simpler: drive it via the public `register` + a manual `applicationDidFinishLaunching(Notification(name: .init("")))` call, which already mirrors what the OS does.
- Care: `performStartup()` calls `checkTailscaleStatus()` which shells out via `tailscaleProvider`. Tests must inject a mock `TailscaleProviding` via `ServiceContainer`, and a no-op deploy server mock so we don't actually bind :8443.
- Three scenarios, as per AC: (a) register → then applicationDidFinishLaunching, assert startup ran exactly once, (b) applicationDidFinishLaunching → then register, assert startup ran exactly once, (c) register but never applicationDidFinishLaunching → assert startup did NOT run. In AppDelegate as currently written, case (a) runs via the `if didRegister, !didPerformStartup` branch at line 112, and case (b) runs via the `if !didPerformStartup, NSApp != nil` branch at line 78. Case (b) has an `NSApp != nil` guard that the test environment satisfies (`NSApplication.shared` is non-nil in XCTest on macOS).
- Regression risk: **low**. New test file only; if the test has to widen `performStartup()` access to `internal`, that's a surgical production change.

### 7. TKT-021 — Code shipped without on-device verification; ticket body empty
- Code changes from [TKT-021.md](tickets/TKT-021.md) Steps 1-4 are present in [AppDelegate.swift](RemoteDeploy/AppDelegate.swift) (both startup dispatches wrapped in `DispatchQueue.main.async`, Bonjour moved to `asyncAfter(deadline: .now() + 0.1)`) and presumably in `BonjourAdvertiser` / `RemoteDeployApp` (not re-read here, but the commit `e340955` touched them).
- `Files Changed`, `Test Report`, `Verification Checklist` sections are empty.
- The fix is plausible on first principles (layout recursion: defer mutations past first layout pass; Bonjour EINVAL: defer publish 100ms past bind), but the AC explicitly requires *observed absence* of `_NSDetectedLayoutRecursion` and `send failed: Invalid argument` lines in the console at startup — that is a manual, on-device observation and nothing in the repo demonstrates it was done.
- Action here is to run the app, capture the console output, and record the result in TKT-021's body sections retroactively. If a warning is still present, we have Step 7 of the original plan (migrate `BonjourAdvertiser` to `NWListener`) as a fallback.
- Regression risk: **low**. Observation-only unless the fallback kicks in.

## Proposed Solution

Single cleanup branch `ticket/tkt-024-audit-cleanup`, merged as one PR. Ordered to isolate risk.

**Approach per sub-item:**

1. **TKT-008** — Add `RemoteDeployTests/JSONBuildHistoryStoreTests.swift`. Use a temp directory via `init(directory:)`. Cover: append-then-load, cap/trim at 100, missing-file → `[]`, corrupt-file → `[]` + logs, and a query-style test using `recentBuilds()`.

2. **TKT-009** — Extract `BundleIDValidator` into `Packages/RemoteDeployShared/Sources/RemoteDeployShared/Validation/BundleIDValidator.swift` as a free `enum` with one `static func validate(_:) -> String?`. Use it in `ProjectSetupStep.validateBundleID` and in `ProjectsRouteHandler` on POST + PUT. Add regression tests in `RemoteDeployTests/API/ProjectsRouteHandlerTests.swift` (accepted/rejected). Add one note-style test in `DeferredSettingsUpdaterTests.swift` asserting `SettingsData` has no bundle-ID surface, with a comment linking to TKT-024 and TKT-009 explaining the AC realignment. Document the deviation in the PR description.

3. **TKT-011** — In `NIODeployServer.start(...)`, switch `configureHTTPServerPipeline(...)` to the `withServerUpgrade:` variant, passing a `NIOWebSocketServerUpgrader` whose `shouldUpgrade` closure:
   - Reads `Authorization: Bearer ...` from the HTTP request head.
   - Looks up the token via the same `pairedDeviceStore` used by `AuthMiddleware` (add a small `TokenAuthenticating` protocol if needed to avoid pulling the whole middleware into `NIODeployServer`).
   - Returns an upgrade response only if the token is valid; otherwise returns `nil` and lets the HTTP pipeline emit `401`.
   - On upgrade success, installs a `WebSocketChannelHandler` that registers the channel with `webSocketManager` and routes inbound `subscribe` frames.
   Apply the same upgrade wiring to the plain-HTTP listener at :8080, or explicitly skip it (plain-HTTP WS is `ws://`, acceptable for local WiFi). Recommend **skip plain-HTTP** — keeps auth on the TLS side only and matches the existing `rejectPairingOverHTTP` stance.
   - On the iOS side: add a reconnect loop in `WebSocketClient` (1s / 2s / 4s / 8s / 16s cap) triggered on `.failure` in `receiveMessages`, gated on an `isActive` flag set in `connect`/`disconnect`.
   - Integration test in `RemoteDeployIntegrationTests/` proving: unauthed upgrade → 401, authed upgrade → 101 + subscribe + broadcast delivered.
   - If the reconnect-with-backoff on the iOS side proves too large for this cleanup, carve it into a follow-up ticket with justification (AC explicitly allows this).

4. **TKT-012** — Create `RemoteDeploy/Views/MenuBar/ServerStatusSection.swift` by extracting the server-running/Tailscale row out of `MenuBarHeaderSection.swift`. Move `ProjectRowView` into its own file under `MenuBar/ProjectRowView.swift`. Move `handleAPIBuildRequest` onto a small `MenuBarBuildDispatcher` helper (or into `BuildManager`). Target: `MenuBarView.swift` ≤ 95 lines, giving margin below the 100-line bar. Insert the new `ServerStatusSection()` as a sibling of `MenuBarHeaderSection()` in the main `VStack`. Update `project.yml` XcodeGen spec if it enumerates files explicitly (check first).

5. **TKT-014** — Lift `validateBundleID`, `validateTeamID`, `validatePath`, and a new `validateScheme(_:)` into `enum ProjectSetupValidators` in `RemoteDeploy/Views/SetupSteps/ProjectSetupValidators.swift`. `ProjectSetupStep` calls through to the enum. Add `@State private var schemeError: String?`; populate from the enum on scheme pick and on `detectedSchemes` change; render inline red caption; short-circuit `saveProject()` if `schemeError != nil`. Add `RemoteDeployTests/ProjectSetupValidatorsTests.swift` with empty-scheme coverage + the existing cases.

6. **TKT-019** — Add `RemoteDeployTests/AppDelegateStartupTests.swift`. If `performStartup()` must be exposed, widen it to `internal` with a `// Exposed for AppDelegateStartupTests` comment. Inject mock `TailscaleProviding` + `MockDeployServer` via a test-only `ServiceContainer` constructor. Three test cases matching the AC.

7. **TKT-021** — Build and run the host app from Xcode with the current fix in place. Capture the first 30 seconds of console output. Grep for `_NSDetectedLayoutRecursion` and `send failed: Invalid argument`. Fill in `Files Changed` (already known from commit `e340955`), `Test Report` (what was observed on launch), and `Verification Checklist`. **If either warning reappears**: execute Step 7 of TKT-021's implementation plan (migrate `BonjourAdvertiser` → `NWListener`) and/or revisit layout recursion remediation. The NWListener path is a fallback only — the primary path is observation + retroactive documentation.

**Why this approach over alternatives:**
- Single PR matches AC; ordered sub-tasks let us bisect regressions cheaply.
- TKT-009 is rescoped to the correct boundary (`ProjectsRouteHandler`) because the literal AC references a non-existent field; pretending otherwise would leave the real programmatic gap open.
- TKT-011 adds upgrade wiring at the real seam (`configureHTTPServerPipeline(withServerUpgrade:)`) rather than doing a custom upgrade in `HTTPHandler`, matching SwiftNIO idioms and minimizing surface area.
- TKT-012 targets ≤95 lines rather than the bare 100 so trivial future edits don't re-break the AC.

**Tradeoffs:**
- TKT-011 is the riskiest change. If integration tests fail, we have a documented carve-out path (client reconnect → follow-up ticket) per AC. Upgrade wiring itself is non-negotiable — the ticket explicitly requires no silent drops of the server-side work.
- Widening `AppDelegate.performStartup()` to `internal` is a small encapsulation loss in exchange for direct test coverage. Acceptable since the whole `AppDelegate` is already effectively package-private.
