## Description

Two issues with TKT-048's Expo project setup in the host app:

### 1. Scheme dropdown shows all CocoaPods sub-schemes

When adding an Expo project, `detectSchemes(at:)` runs `xcodebuild -list` against the generated `.xcworkspace`. For a React Native project this returns dozens of internal schemes (boost, DoubleConversion, EXConstants, Expo, ExpoAsset, ExpoFont, ExpoModulesCore, FBLazyVector, React, React-Core, RCT-Folly, hermes-engine, Pods-CFB, etc.) alongside the actual app scheme (`CFB`). The user sees a huge confusing dropdown when they only care about `CFB`.

For Expo projects, the scheme list should be filtered to only show the main app target scheme(s) — not the CocoaPods dependency schemes. The app scheme name can be derived from `app.json` → `expo.name` or from the `.xcodeproj` name in the `ios/` directory (e.g., `CFB.xcodeproj` → scheme `CFB`).

### 2. Bundle ID field not auto-populated

TKT-048 step 12 was supposed to parse `app.json` → `expo.ios.bundleIdentifier` and auto-fill the Bundle ID field in the settings UI. The field is blank when adding the cards-for-bongs project, meaning either the auto-detection didn't trigger or the parsed value isn't wired to the UI field.

(Team ID being blank is expected — it's not in `app.json` and always requires manual entry.)

## Reproduction Steps (bugs only)

1. Open RemoteDeploy host app → Settings → Add Project.
2. Select "Expo (React Native)" type.
3. Browse to `/Users/plympton/src/_project-throughline/cards-for-bongs/cfb/`.
4. Observe: Scheme dropdown shows ~40+ schemes including all CocoaPods internal targets. Bundle ID field is empty.
5. Expected: Scheme dropdown shows only `CFB`. Bundle ID shows `com.cfb.cardgame`.

## Acceptance Criteria

- [ ] For Expo projects, the scheme picker shows only the main app scheme(s) — CocoaPods dependency schemes are filtered out.
- [ ] The filtering heuristic is robust: use the Expo app name from `app.json` or the `.xcodeproj` name in `ios/`, not a hardcoded blocklist of pod names.
- [ ] Bundle ID is auto-populated from `app.json` → `expo.ios.bundleIdentifier` when an Expo project directory is selected.
- [ ] Existing Xcode project scheme detection is unaffected (no filtering for native projects).

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Bug 1: Scheme dropdown shows all CocoaPods sub-schemes

**Root cause:** `ExpoBuildEngine.detectSchemes(at:)` (line 141, `RemoteDeploy/Services/ExpoBuildEngine.swift`) resolves the `ios/` directory and then delegates directly to `xcodeEngine.detectSchemes(at: iosDir)` (line 176) with no filtering. `XcodeBuildEngine.detectSchemes(at:)` (line 242, `RemoteDeploy/Services/XcodeBuildEngine.swift`) resolves to the `.xcworkspace` in `ios/` (preferring workspace over project, line 265), runs `xcodebuild -list -workspace`, and returns every scheme found in the "Schemes:" section via `parseSchemesFromOutput()` (line 489). For an Expo-generated workspace this includes ~40+ CocoaPods dependency schemes (boost, React-Core, RCT-Folly, ExpoModulesCore, Pods-CFB, etc.) alongside the actual app scheme.

**Call path:** `ProjectSetupStep.detectSchemes()` (line 409) → `serviceContainer.buildEngine` (`BuildEngineRouter`, line 66) → `expoEngine.detectSchemes(at:)` → `xcodeEngine.detectSchemes(at: iosDir)` → unfiltered scheme list returned to the UI picker at line 416.

**The correct app scheme can be derived from the `.xcodeproj` name** inside `ios/`. Expo prebuild generates `ios/<AppName>.xcodeproj` where `<AppName>` matches `expo.name` in `app.json`. For the CFB project: `ios/CFB.xcodeproj` → scheme `CFB`. The `.xcodeproj` approach is more reliable than parsing `app.json` because the scheme name always matches the project file name (Expo convention), and the `ios/` directory is already resolved by the time filtering is needed.

### Bug 2: Bundle ID field not auto-populated

**Root cause:** `autoDetectExpoApp()` (line 297, `ProjectSetupStep.swift`) correctly parses `app.json` → `expo.ios.bundleIdentifier` and sets `bundleID` (line 327). However, `applySelectedPath()` calls `autoDetectExpoApp()` at line 291 and then immediately calls `detectSchemes()` at line 293. When scheme detection completes asynchronously, `selectedScheme` is set to `schemes.first ?? ""` at line 430, which triggers the `.onChange(of: selectedScheme)` handler at line 198. That handler **unconditionally resets `bundleID = ""`** (line 200) and then calls `detectBuildSettings()` which runs `xcodebuild -showBuildSettings` against the Expo workspace. For Expo projects, `PRODUCT_BUNDLE_IDENTIFIER` in the xcodebuild output is either missing or a template variable `$(PRODUCT_BUNDLE_IDENTIFIER)` (which the parser correctly skips at line 399), so `bundleID` remains empty.

**Timeline:**
1. `autoDetectExpoApp()` sets `bundleID = "com.cfb.cardgame"` (synchronous, immediate)
2. `detectSchemes()` fires async Task, returns control
3. Async Task completes, sets `selectedScheme = "CFB"` on MainActor
4. `.onChange(of: selectedScheme)` fires, resets `bundleID = ""`
5. `detectBuildSettings()` runs, fails to recover the bundle ID from xcodebuild

## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

### Fix 1: Filter schemes in `ExpoBuildEngine.detectSchemes(at:)`

After delegating to `xcodeEngine.detectSchemes(at: iosDir)`, filter the returned list to only schemes that match a `.xcodeproj` name found in `ios/` (excluding `Pods.xcodeproj`). This is the right filtering layer because:
- It keeps `XcodeBuildEngine.detectSchemes` unchanged (native projects stay unaffected).
- The `ios/` directory is already resolved and available.
- The `.xcodeproj` naming convention is deterministic for Expo projects.

```swift
// In ExpoBuildEngine.detectSchemes(at:), after line 176:
let allSchemes = try await xcodeEngine.detectSchemes(at: iosDir)

// Filter to schemes matching a non-Pods .xcodeproj in ios/
let iosContents = (try? FileManager.default.contentsOfDirectory(atPath: iosDir)) ?? []
let appProjectNames = iosContents
    .filter { $0.hasSuffix(".xcodeproj") && $0 != "Pods.xcodeproj" }
    .map { ($0 as NSString).deletingPathExtension }

if appProjectNames.isEmpty {
    return allSchemes  // fallback: no filtering if heuristic fails
}

let filtered = allSchemes.filter { appProjectNames.contains($0) }
return filtered.isEmpty ? allSchemes : filtered
```

### Fix 2: Skip bundle ID reset when Expo auto-detection already populated it

In the `.onChange(of: selectedScheme)` handler (line 198), only reset `bundleID` for Xcode projects. For Expo projects where `autoDetectExpoApp()` already set the bundle ID, preserve it.

```swift
.onChange(of: selectedScheme) {
    // Only reset bundle ID for Xcode projects — Expo projects
    // get their bundle ID from app.json, not xcodebuild.
    if projectType != .expo {
        bundleID = ""
    }
    teamID = ""
    schemeError = ProjectSetupValidators.validateScheme(selectedScheme)
    detectBuildSettings()
}
```

Alternatively, skip the `detectBuildSettings()` call entirely for Expo projects since `xcodebuild -showBuildSettings` is not useful against an Expo-generated workspace. This is cleaner:

```swift
.onChange(of: selectedScheme) {
    schemeError = ProjectSetupValidators.validateScheme(selectedScheme)
    if projectType != .expo {
        bundleID = ""
        teamID = ""
        detectBuildSettings()
    }
}
```

### Dependencies
None.

### Regression Risk
**Low.** Both fixes are scoped to Expo-only code paths:
- Fix 1 adds post-filtering inside `ExpoBuildEngine.detectSchemes` only; `XcodeBuildEngine.detectSchemes` is untouched, so native Xcode project scheme detection is unaffected.
- Fix 2 adds a `projectType != .expo` guard. Xcode projects continue to reset bundle ID and run `detectBuildSettings()` exactly as before.
- The fallback in fix 1 (return unfiltered list if no `.xcodeproj` matched) prevents silent breakage if the Expo `ios/` directory layout changes.
- Existing `BuildEngineRouterTests` cover the dispatch logic and remain valid.
