## TKT-045: Companion Build tab stays stuck when xcodebuild fails at preflight

**Branch:** `ticket/tkt-045-companion-build-preflight-stuck`
**Files changed:** 5 (1 prod, 2 test, 1 project.pbxproj regen, 1 ticket file) | **Tests:** 330 passing (+3 new) | **Risk:** low-to-medium

### What changed

- `XcodeBuildEngine` now captures the last 8 stderr lines into a ring buffer and includes the most recent line in the thrown `BuildError.xcodebuildFailed` message instead of the generic `"xcodebuild exited with code N"`.
- Error-branch and cancel-branch termination handlers now deterministically finish the log continuation so in-flight stderr lines flush before the build manager's `for await` loop exits.
- Two new real-xcodebuild unit tests (`XcodeBuildEngineFailureTests.swift`) covering invalid scheme + invalid destination preflight failures.
- One new integration test (`BuildManagerWebSocketIntegrationTests.swift`) that wires a real `XcodeBuildEngine` into a real `NIODeployServer` and asserts the terminal `buildstatus` WebSocket frame carries real xcodebuild stderr.

- [ ] Build & Deploy button becomes re-enabled
- [ ] Fix handles invalid destination, missing scheme, project file not opened

## TKT-046: Companion build log should be collapsible and stop auto-scrolling after build

**Branch:** `ticket/tkt-046-collapsible-build-log`
**Files changed:** 1 (`RemoteDeployCompanion/Views/Build/BuildControlView.swift`) | **Tests:** 311 passing (host unchanged); companion builds clean | **Risk:** low

### What changed

- `BuildControlView` gained `@State private var isLogExpanded = false` ŌĆö log is **collapsed by default**.
- `BuildLogStreamView()` is now wrapped in a `DisclosureGroup` with label "Build Log", a line-count badge, and a `doc.text.magnifyingglass` SF Symbol.
- When collapsed, a `Spacer()` anchors the Build & Deploy section near the top; when expanded the log gets full remaining height.
- `BuildLogStreamView` now observes `BuildManager` as a second `@EnvironmentObject` (already provided at app root, no plumbing changes needed).
- The auto-scroll handler (`.onChange(of: buildLogLines.count)`) now does `guard buildManager.isBuilding else { return }` ŌĆö stops yanking the viewport once the build reaches a terminal state.
- New `.onChange(of: buildManager.isBuilding)` transitions: `false ŌåÆ true` resets `isLogExpanded = false` and snaps the scroll to the tail, giving each fresh build a clean collapsed-by-default state.

### Acceptance Criteria (copied from ticket)

- [ ] Build log is collapsed by default on the Build tab ŌĆö a disclosure control expands it on demand.
- [ ] When collapsed, the primary Build & Deploy button and build status remain the dominant UI.
- [ ] Auto-scroll only happens while a build is actively running; once terminal, auto-scroll stops.
- [ ] When the log is expanded, the user can manually scroll without being fought by auto-scroll after completion.
- [ ] Starting a new build re-enables auto-scroll for that build.
