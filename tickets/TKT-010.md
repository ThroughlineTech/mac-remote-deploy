## Description
Each module defines its own error enum (`BuildError`, `APIError`, `ConnectionError`, etc.) with no shared hierarchy. Create `Packages/RemoteDeployShared/Sources/RemoteDeployShared/Error/RemoteDeployError.swift` with a unified `LocalizedError` enum used across both targets. Map module-specific errors into it where useful.

## Acceptance Criteria
- [x] New `RemoteDeployError` enum in shared package at `Packages/RemoteDeployShared/Sources/RemoteDeployShared/Error/RemoteDeployError.swift`
- [x] Cases cover at minimum: `serverStartFailed`, `buildFailed`, `pairingFailed`, `networkError`, `fileNotFound`, `validationFailed`, plus a `.unknown(message:)` fallback
- [x] Each case has `errorDescription`, `failureReason`, `recoverySuggestion` per `LocalizedError`
- [x] Convenience init `RemoteDeployError(wrapping: any Error)` collapses any underlying error into `.unknown` with the localized description preserved
- [x] Existing 11 per-module enums **wrapped** at boundaries (not replaced — see Investigation rationale)
- [x] At least one boundary call site in each target uses `RemoteDeployError` to prove cross-target consumption works (e.g. one site in `RemoteDeployApp.swift` and one in `ConnectionManager.swift`)
- [x] Unit tests for `RemoteDeployError` cover: every case's `errorDescription`/`failureReason`/`recoverySuggestion`, and the `wrapping:` init
- [x] All 236 existing tests still pass
- [x] Build clean across both targets

These enums are **typed at the throw site** — `XcodeBuildEngine` throws `BuildError.archiveFailed(stderr)`, not a stringly-typed wrapper. That's idiomatic Swift and worth preserving. Replacing them all with `RemoteDeployError` cases would be a 11-file rewrite with extensive test churn (e.g. `ProwlNotifierTests` catches `ProwlError` cases by name).

### Existing tests that catch specific enums
- [PushNotifierTests.swift](RemoteDeployTests/PushNotifierTests.swift): 37 tests, several `do/catch` blocks catching `ProwlError`/`PushoverError`/`NtfyError` cases.
- Other test files: not catching enum cases directly per quick search.

### Where errors actually surface today (the "boundaries")
- **macOS**: `print("...failed: \(error.localizedDescription)")` calls in `RemoteDeployApp.swift` (Tailscale, server start, settings save, projects load) — silent to the user, just console-spammed.
- **iOS**: `ConnectionManager.pair(...)` throws `ConnectionError` cases that bubble up to `ServerDiscoveryView` and `TokenEntryView` and render as error text. `APIClient.swift` throws `APIClientError` cases consumed by views.
- **No central error sink yet.** TKT-007 (UI error surfaces) is the ticket that adds `@Published var error: AppError?` to AppState/ConnectionManager. TKT-010 should land the *type* that TKT-007 will plug into, but should not itself touch UI.

### Existing patterns to follow
- Shared package layout: [Packages/RemoteDeployShared/Sources/RemoteDeployShared/](Packages/RemoteDeployShared/Sources/RemoteDeployShared/) currently has `Models/` and `API/` subdirectories. New `Error/` subdirectory fits the convention.
- LocalizedError conformance pattern: each existing enum implements `var errorDescription: String?` only. None implement `failureReason` or `recoverySuggestion` — TKT-010 raises the bar by adding those for the new type.

### Regression Risk — **LOW**
Pure additive change:
- New file in shared package, no modifications to existing types
- Optional opt-in at boundary call sites (small, isolated edits to print() lines)
- 11 existing enums + their tests untouched
- Shared package builds for both macOS and iOS targets — TKT-004 and TKT-003 already exercised it heavily so the package is healthy

Risk vectors:
1. **Sendable/concurrency.** New enum must conform to `Sendable` (most error types do). Trivial.
2. **Public visibility.** Cases must be `public` to be visible from both consumers. Easy to forget on one case.
3. **iOS link errors.** If the new file imports anything not available on iOS (e.g. `os.Logger` is fine but some Foundation pieces differ), the iOS build breaks. Mitigation: only import `Foundation`.
4. **Test catch-block churn.** None expected since I'm not touching existing enums. But if I accidentally rename an existing enum case while editing nearby, tests catch it.

## Proposed Solution

### Approach: additive wrapping, not replacement

Keep the 11 module-specific enums **exactly as they are**. They're typed, useful at the throw site, and well-tested. Add a new top-level `RemoteDeployError` in the shared package that:

1. Provides 7 named cases for the error categories from the ticket (6 + `.unknown` fallback)
2. Implements full `LocalizedError` (description / failureReason / recoverySuggestion)
3. Has a convenience `init(wrapping: any Error)` that collapses any error into `.unknown(reason:)` — **the bridge from module-specific enums to the unified type**
4. Has static factory helpers for the named cases so call sites read naturally

Boundary code (currently a handful of `print()` calls in `RemoteDeployApp.swift` and a few iOS views) is updated to use `RemoteDeployError` where it makes the message clearer. **Most boundary work waits for TKT-007** — TKT-010 just lands the type and proves it builds for both targets.

### Why this approach over full replacement
- **Minimal blast radius.** 1 new file vs 11 modified files + ~37 test changes.
- **Preserves throw-site type safety.** `BuildError.cancelled` is more expressive than `RemoteDeployError.buildFailed("cancelled")` at the throw site.
- **Forward-compatible.** TKT-007 will plug `RemoteDeployError` into AppState's new `error: RemoteDeployError?` field. With the wrapping init, any thrown enum becomes a `RemoteDeployError` at the catch site without rewriting throw sites.
- **Doesn't fight Swift idioms.** Per-module typed errors are the Swift convention; a giant catch-all enum would be the anti-pattern.

### Tradeoffs
- **The "consolidated" goal is partial.** Strict consolidation would mean one enum to rule them all. We're not doing that. The unified type is the *boundary* type, not the *throw-site* type. The ticket description says "Existing per-module enums either replaced or wrapped" — wrapping is explicitly an option.
- **Two layers of error description.** When a `BuildError.archiveFailed("…")` gets wrapped into `RemoteDeployError.unknown("Archive failed: …")`, you're chaining `errorDescription` calls. The bridge preserves the original message via `error.localizedDescription`, so no information is lost, but a stack trace would show the wrapping.

### Sketch of the new type

```swift
public enum RemoteDeployError: LocalizedError, Sendable {
    case serverStartFailed(reason: String)
    case buildFailed(reason: String)
    case pairingFailed(reason: String)
    case networkError(reason: String)
    case fileNotFound(path: String)
    case validationFailed(field: String, reason: String)
    case unknown(reason: String)

    public init(wrapping error: any Error) {
        if let already = error as? RemoteDeployError { self = already }
        else { self = .unknown(reason: error.localizedDescription) }
    }

    public var errorDescription: String? { ... }      // user-facing one-liner
    public var failureReason: String? { ... }         // technical detail
    public var recoverySuggestion: String? { ... }    // what the user should do
}
```
