## Description

In the companion iOS app, navigating to Projects â†’ selecting a project â†’ tapping "Build & Deploy" appears to do nothing from the user's perspective. The server does start building (visible on the host), but the companion app shows no loading indicator, progress bar, or status feedback during the build.

Eventually the companion app switches to a project info page showing the URL slug, but this is not useful information â€” the user wants to see build progress, success/failure status, and (on success) an install prompt or download link.

The core issue is a missing feedback loop: the companion fires the build request but doesn't subscribe to or display build status updates.

## Reproduction Steps
1. Open the RemoteDeploy Companion app on iOS.
2. Navigate to the Projects tab.
3. Select a project and tap "Build & Deploy."
4. Observe: nothing visually happens in the companion app. No spinner, no progress, no status.
5. Check the host Mac â€” the server is building the project.
6. Eventually the companion switches to a project info page with the URL slug â€” not useful.

## Acceptance Criteria
- [ ] After tapping "Build & Deploy," the companion app immediately shows a loading/progress indicator (spinner, progress bar, or status text).
- [ ] Build status updates from the server are displayed in real time (e.g., "Building...", "Signing...", "Deploying...").
- [ ] On build success, the companion shows a clear success state with an actionable next step (e.g., "Install" button or link).
- [ ] On build failure, the companion shows an error message with the failure reason.
- [ ] The URL slug page is either removed or made secondary to the build progress view.
- [ ] `xcodebuild test` passes.

## Investigation

There are three distinct problems causing the "no feedback" experience:

### Root Cause 1: WebSocket not connected on restored sessions

`ConnectionManager.restoreConnection()` (line 88-101 of `RemoteDeployCompanion/Services/ConnectionManager.swift`) does NOT call `webSocketClient.connect(baseURL:token:)`. WebSocket is only connected during `pair()` (line 141) and `connect()` (line 165). Since the normal launch path restores from Keychain, the WebSocket is never opened, so:
- No live build log lines arrive (the `BuildLogStreamView` is always empty).
- No `buildstatus` WebSocket messages arrive.
- The `WebSocketClient.latestStatus` property (line 20 of `WebSocketClient.swift`) is populated from WebSocket frames but is **never observed** by any view or by the companion `BuildManager`.

### Root Cause 2: ProjectDetailView doesn't observe BuildManager (wrong instance)

`ProjectDetailView` (line 118-219 of `ProjectListView.swift`) creates its own `BuildManager()` instance in `init` (line 129) via `@ObservedObject`. This is a throwaway instance â€” the real shared build manager lives at `connectionManager.buildManager`. While the view does reference `connectionManager.buildManager` in the button action (line 152) and status display (line 164), the `@ObservedObject private var buildManager` on line 124 is never used meaningfully and is confusing. The actual status display logic at lines 164-185 correctly reads from `connectionManager.buildManager`, so this is a minor issue â€” the real problem is Root Causes 1 and 3.

### Root Cause 3: No navigation to build progress after tapping "Build & Deploy" from Projects tab

When the user taps "Build & Deploy" on `ProjectDetailView`, the build fires (line 152) and the button label changes to "Building..." (line 192-194), but:
- The user stays on the static project info page showing URL slug, bundle ID, etc. â€” not useful during a build.
- There is no navigation to the Build tab where the live log and elapsed timer live.
- There is no inline progress view beyond a tiny spinner and "Building..." text in the button section.
- On build success, the `buildStatus` shows "Build Succeeded" with no install link or next action.
- The `BuildStatusInfo.message` on success contains the IPA path (from server-side `NIODeployServer+BuildEventBroadcasting.swift` line 53), not an install URL â€” useless to the user.

### Root Cause 4: Companion BuildManager polls server status but doesn't use project-specific build endpoint

The companion's `BuildManager.startPolling()` (line 68-101 of `BuildManager.swift`) polls `client.getStatus()` which returns `ServerStatus.buildStatus`. This works but:
- It does NOT use the project-specific `GET /api/v1/projects/:id/build` endpoint (which exists in `APIClient.getBuildStatus(projectID:)` at line 76).
- The server's progress messages ("Starting build...", "Archiving...", etc.) are broadcast via WebSocket `buildstatus` channel but the companion only receives them if the WebSocket is connected (see Root Cause 1).
- The polling interval is 2 seconds which is reasonable as a fallback.

### Regression Risk: Medium

Changes touch the connection lifecycle (WebSocket auto-connect), build flow navigation, and project detail view. The WebSocket reconnect logic is already robust (exponential backoff from TKT-011/TKT-024), but adding auto-connect on restore needs testing. The navigation change from Projects tab to Build tab is new behavior.

## Proposed Solution

**A. Fix WebSocket connection on session restore** â€” Add `webSocketClient.connect(...)` to `restoreConnection()` so live build log and status updates work from app launch, not just after fresh pairing.

**B. Wire WebSocket build status into BuildManager** â€” Have the companion `BuildManager` observe `WebSocketClient.latestStatus` for real-time status updates in addition to its existing REST polling fallback. This gives instant status transitions instead of 2-second polling lag.

**C. Add build progress overlay to ProjectDetailView** â€” When a build is triggered from the project detail page, show an inline build progress section with:
  - A prominent progress indicator with phase text ("Building...", "Signing...", etc.)
  - Elapsed time counter (reuse the `TimelineView` pattern from `BuildControlView`)
  - On success: show install URL as a tappable link + "Open Install Page" button
  - On failure: show error message prominently
  - A "View Full Log" button that switches to the Build tab

**D. Clean up the unused `@ObservedObject buildManager` in ProjectDetailView** â€” Remove the dead code that creates a throwaway `BuildManager()` instance.
