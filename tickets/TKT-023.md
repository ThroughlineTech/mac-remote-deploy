## Description
After TKT-022 landed Face ID support, the first-prompt UX is correct (Face ID is offered, single prompt), but the failure-recovery path is broken. When Face ID fails â€” cover face, look away, or fail the biometric scan â€” the app immediately drops to the unpaired UI (the "Scan QR Code / Nearby Servers" discovery screen), as if no credentials exist. Expected: iOS should fall back to the device passcode prompt automatically, the user enters it, and the app connects normally.

Root cause is almost certainly the wrong `LAPolicy` in [KeychainStore.authenticatedContext()](RemoteDeployCompanion/Services/KeychainStore.swift). TKT-022 used `.deviceOwnerAuthenticationWithBiometrics`, which is **biometrics-only with a manual fallback button** that requires the user to tap "Use Passcode" after a failure. The correct policy for "try biometrics, auto-fall-back to passcode" is `.deviceOwnerAuthentication` â€” that one presents biometrics first and, on failure, automatically transitions to the passcode screen without user action.

Additional consideration: even when the user does tap "Use Passcode" with `.deviceOwnerAuthenticationWithBiometrics`, it appears we're not catching the "fallback button tapped" path correctly and the continuation resolves to `success: false`, dumping the user back to the unpaired screen.

## Reproduction Steps
1. Launch the companion app on a Face ID device
2. Cover your face / look away / fail the Face ID scan
3. Observe: the Face ID prompt disappears and the app shows the "Scan QR Code / Nearby Servers" discovery screen
4. Expected: iOS should show the passcode entry screen; after entering the passcode, the app connects

## Investigation

### Root cause â€” CONFIRMED

[KeychainStore.swift:157-159](RemoteDeployCompanion/Services/KeychainStore.swift#L157-L159):
```swift
let policy: LAPolicy = context.canEvaluatePolicy(.deviceOwnerAuthenticationWithBiometrics, error: &policyError)
    ? .deviceOwnerAuthenticationWithBiometrics
    : .deviceOwnerAuthentication
```

On a device with Face ID enrolled, `canEvaluatePolicy(.deviceOwnerAuthenticationWithBiometrics)` returns true, so TKT-022 selects `.deviceOwnerAuthenticationWithBiometrics`. That policy is **biometrics-only**:
- Shows Face ID prompt
- On biometric failure (cover face, look away, mismatch), iOS does NOT automatically present the passcode screen
- The user must explicitly tap the "Use Passcode" fallback button (which only appears after several failures and requires `localizedFallbackTitle` to be set)
- If the user doesn't tap fallback â€” or if `evaluatePolicy` returns `success: false` before the fallback button is offered â€” the continuation resolves with `success: false` and `authenticatedContext()` returns nil, dumping the user into the unpaired discovery UI

The correct policy for "try biometrics first, auto-fall-back to passcode on failure" is **`LAPolicy.deviceOwnerAuthentication`**. Apple's docs are explicit:

> User authentication with biometry, Apple Watch, or the device passcode. **When evaluated, this policy prompts the user for biometric authentication first. If that fails, it falls back to asking for the device passcode.**

This is the standard "unlock your sensitive data" pattern used by Apple Wallet, Notes (locked notes), Keychain Access, 1Password, etc. The policy handles the biometric-to-passcode transition internally â€” our code just calls `evaluatePolicy` once and gets a single success/failure callback after the user either succeeds (via either path) or cancels.

### Why TKT-022 picked the wrong policy

The comment on line 155-156 was aspirational:
```swift
// Prefer biometrics with passcode fallback. If biometrics aren't available
// (e.g. not enrolled), .deviceOwnerAuthentication still offers passcode.
```

The author (me) conflated "biometrics with fallback" with `.deviceOwnerAuthenticationWithBiometrics`. In reality:
- `.deviceOwnerAuthenticationWithBiometrics` = **biometrics only**, with optional manual "Use Passcode" button via `localizedFallbackTitle` after repeated failures
- `.deviceOwnerAuthentication` = **biometrics first, then automatic passcode fallback**

The second one is what we wanted all along.

### Why `localizedFallbackTitle` didn't save us

TKT-022 sets `context.localizedFallbackTitle = "Use Passcode"` at line 152. This customizes the fallback button text, but the button only appears after two failed biometric attempts in `.deviceOwnerAuthenticationWithBiometrics` mode. A single cover-your-face doesn't show it. And even if the user did tap it, `.deviceOwnerAuthenticationWithBiometrics` handles the fallback via `LAError.userFallback` â€” our code treats any non-success as failure and bails.

### Regression Risk â€” **LOW**

Swapping one enum case. Behavior changes:
- **Old (buggy):** Face ID only, failure dumps user into discovery UI
- **New (correct):** Face ID first, auto-passcode fallback, user cancel still goes to discovery

The only thing the `canEvaluatePolicy(.deviceOwnerAuthenticationWithBiometrics)` check was doing was branching the policy choice. With the new approach we don't need it at all â€” `.deviceOwnerAuthentication` handles the "no biometrics enrolled" case internally by going straight to passcode.

Tests: no existing tests touch LAContext (biometrics can't be exercised in XCTest). Zero test churn expected.

## Proposed Solution

### Change `LAPolicy` to `.deviceOwnerAuthentication`

Replace the branched policy selection with a single constant:
```swift
return await withCheckedContinuation { continuation in
    context.evaluatePolicy(.deviceOwnerAuthentication, localizedReason: authReason) { success, error in
        ...
    }
}
```

Delete the `canEvaluatePolicy` check â€” unneeded. Delete the now-unused `policyError` variable. Optionally keep `localizedFallbackTitle` but it's redundant with `.deviceOwnerAuthentication` (which doesn't show a fallback button; iOS handles the transition automatically). Keep it for cosmetic consistency.

### Tradeoffs

- **First-time users who haven't set a passcode.** `.deviceOwnerAuthentication` requires the device to have a passcode set. On devices with no passcode (rare in 2026 but possible), `canEvaluatePolicy(.deviceOwnerAuthentication)` returns false and `evaluatePolicy` fails immediately. We'll catch that in the failure branch and return nil. The user would have to set a passcode in Settings. This is an extreme edge case â€” the whole point of `.userPresence`-guarded keychain items is that the user has a device lock.
- **Passcode-only fallback UX.** Some users might prefer explicit biometric-retry vs immediate passcode fallback. The iOS standard behavior is auto-fallback, and Apple's own apps follow this pattern, so users expect it.
