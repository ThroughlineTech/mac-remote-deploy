## Description

In the iOS companion app's **Projects** tab, the project detail view doesn't reactively update when a build is triggered from inside it. Specifically:

1. Go to **Projects** tab → tap a project → the project detail view appears (Name, Bundle ID, Team, Scheme, etc., with a **Build & Deploy** button at the bottom)
2. Tap **Build & Deploy**
3. Nothing visibly changes on the detail view. No progress indicator, no state change, no "Building…" section
4. Navigate to any other tab (e.g. **Build**, **Settings**) and come back to **Projects** → tap the same project → now the detail view shows the expected **Build Status** section with "Build Succeeded", "Open Install Page", and "View Build Log"

So the build state is fetched/refreshed on view appearance but isn't being **observed reactively** — the detail view either:
- Doesn't subscribe to the WebSocket build-status stream at all (it only does a one-shot fetch in `.onAppear`), or
- Subscribes but doesn't apply the updates to its local `@State` / `@Published` binding, or
- Holds onto a stale snapshot of the project that doesn't get replaced when the store emits an update

Related but distinct from TKT-043, which is about the separate **Build** tab getting stuck in a "Building…" state. This ticket is specifically about the **Projects → project detail** screen not reflecting build state changes in real time.

See the two attached screenshots: the first is the detail view right after tapping Build & Deploy (no build status visible), the second is the same view after navigating away and back (build status now shown with "Build Succeeded").

## Reproduction Steps (bugs only)

1. Open the companion app, go to **Projects**
2. Tap any project to open the detail view
3. Tap **Build & Deploy** at the bottom of the detail view
4. Observe: nothing changes on the screen
5. Wait for the build to actually complete on the Mac
6. Still nothing changes on the detail view
7. Switch to any other tab (Build/Installs/Settings), then tap **Projects** again, then tap the same project
8. Observe: the **Build Status** section now appears with the completed build's status

## Acceptance Criteria

- [ ] Tapping **Build & Deploy** on the project detail view immediately reflects the "Building…" state (spinner / progress / disabled button) without leaving the view
- [ ] As the build progresses, the detail view updates live (via WebSocket or the same subscription mechanism the Build tab uses)
- [ ] When the build completes, the **Build Status** section appears on the detail view without requiring a tab switch
- [ ] Success, failure, and cancelled states all render correctly in-place

## Investigation
<!-- Agent fills this during /ticket-investigate -->

- Root cause: the Projects detail screen reads build state from a nested observable (`connectionManager.buildManager`) but does not observe that nested object directly. In [RemoteDeployCompanion/Views/Projects/ProjectListView.swift](RemoteDeployCompanion/Views/Projects/ProjectListView.swift), `ProjectDetailView` is declared with only `@EnvironmentObject var connectionManager` and computes state from `connectionManager.buildManager` ([RemoteDeployCompanion/Views/Projects/ProjectListView.swift](RemoteDeployCompanion/Views/Projects/ProjectListView.swift#L118), [RemoteDeployCompanion/Views/Projects/ProjectListView.swift](RemoteDeployCompanion/Views/Projects/ProjectListView.swift#L126), [RemoteDeployCompanion/Views/Projects/ProjectListView.swift](RemoteDeployCompanion/Views/Projects/ProjectListView.swift#L132), [RemoteDeployCompanion/Views/Projects/ProjectListView.swift](RemoteDeployCompanion/Views/Projects/ProjectListView.swift#L270)).
- `ConnectionManager` exposes `buildManager` as `let buildManager = BuildManager()` and does not forward `buildManager.objectWillChange` through its own `objectWillChange` ([RemoteDeployCompanion/Services/ConnectionManager.swift](RemoteDeployCompanion/Services/ConnectionManager.swift#L40)). That means `BuildManager` publishes updates, but `ProjectDetailView` is not invalidated by those updates.
- The live status pipeline itself is present: `ConnectionManager` wires `buildManager.observeWebSocketStatus(webSocketClient)` in restore/pair/connect flows ([RemoteDeployCompanion/Services/ConnectionManager.swift](RemoteDeployCompanion/Services/ConnectionManager.swift#L105), [RemoteDeployCompanion/Services/ConnectionManager.swift](RemoteDeployCompanion/Services/ConnectionManager.swift#L152), [RemoteDeployCompanion/Services/ConnectionManager.swift](RemoteDeployCompanion/Services/ConnectionManager.swift#L177)); `BuildManager` updates `@Published` status from WebSocket and polling ([RemoteDeployCompanion/Services/BuildManager.swift](RemoteDeployCompanion/Services/BuildManager.swift#L100)).
- Why tab switch "fixes" it: navigating away/back recreates the detail view, which re-reads the latest snapshot from `BuildManager`; without recreation there is no reactive invalidation.
- Regression risk: medium. The fix changes observation wiring on a high-traffic detail view, but stays scoped to Projects detail and keeps existing build trigger/cancel and network flows intact.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

- Make `ProjectDetailView` observe `BuildManager` directly via `@ObservedObject`.
- Inject the shared `connectionManager.buildManager` into `ProjectDetailView` from the Projects navigation destination.
- Replace detail-view reads/actions from `connectionManager.buildManager.*` to `buildManager.*` so `@Published` changes in `BuildManager` trigger immediate re-render in-place.
- Keep connection lifecycle, WebSocket subscriptions, and polling logic unchanged to avoid cross-ticket scope creep into TKT-043.
- Add a companion UI regression test that validates immediate in-view transition after tapping **Build & Deploy**.
