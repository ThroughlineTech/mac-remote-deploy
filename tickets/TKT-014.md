## Description
[SetupAssistantView.swift](RemoteDeploy/Views/SetupAssistantView.swift) and the 5 step views in [SetupSteps/](RemoteDeploy/Views/SetupSteps/) work but lack visual progress, real-time validation, and per-step error recovery. Add progress bar, completion checkmarks, inline field validation (bundle ID regex, path existence, cert readability), and in-step error display.

## Acceptance Criteria
- [x] Progress indicator showing "Step N of 5" — **already exists** at [SetupAssistantView.swift:105](RemoteDeploy/Views/SetupAssistantView.swift#L105) (verify still present)
- [x] Completed steps show green checkmarks — **already exists** at [SetupAssistantView.swift:82-90](RemoteDeploy/Views/SetupAssistantView.swift#L82-L90); user can navigate back via clickable circles at [line 75-93](RemoteDeploy/Views/SetupAssistantView.swift#L75-L93)
- [x] Real-time validation feedback for: bundle ID format (reverse-DNS regex), team ID length (10 chars), project path exists, cert/key paths exist and are readable, scheme not empty
- [x] Errors shown in-step under the offending field, not as modal alerts
- [x] Last visited setup step persists in `UserDefaults` so app relaunch resumes mid-setup
- [x] All 249 existing tests still pass

## Investigation

### Current state (post-grep, post-read)
[SetupAssistantView.swift](RemoteDeploy/Views/SetupAssistantView.swift) (200 lines) — **two of the five acceptance criteria are already implemented**:
- Numbered circle indicator with "Step N of 5" text label is at lines 70-108
- Green-fill + checkmark for past steps at lines 82-90 (`if step.rawValue < currentStep.rawValue`)
- Clickable circles for back navigation at lines 75-93

What's missing:
1. **Inline field validation.** [ProjectSetupStep.swift](RemoteDeploy/Views/SetupSteps/ProjectSetupStep.swift) (326 lines) collects bundle ID, team ID, scheme, project path — none with realtime validation. [CertificateSetupStep.swift](RemoteDeploy/Views/SetupSteps/CertificateSetupStep.swift) (217 lines) collects cert and key paths — no readability check. Errors only surface when the user advances and the build fails downstream.
2. **In-step error display.** Today: `print("Build settings detection failed: ...")` (now `Logger.build.error` after TKT-005) but never shown to the user. Each step needs a `@State var fieldError: String?` and inline `Text(error).foregroundColor(.red)` rendering.
3. **Resume mid-setup.** Setup state lives only in `@State currentStep` — closing the wizard loses it. Need to write `currentStep.rawValue` to `UserDefaults` on change and read it back in `init`.

### Regression Risk — **LOW**
View-only changes. No protocol or data model touches. Existing 249 tests don't cover the wizard (it's UI-only). The only way to break things is a SwiftUI compile error, which is caught immediately by `xcodebuild`.

## Proposed Solution

### Approach
**View-layer additive changes only.** No new files. Each step view gets an inline validation pattern + error display, the assistant gets `UserDefaults` persistence for the current step, and the existing checkmark/progress UI stays as-is (the AC was based on the audit's outdated view of the file).

### Validation rules
| Field | Check | Step file |
|---|---|---|
| Bundle ID | matches `^[A-Za-z][A-Za-z0-9-]*(\.[A-Za-z][A-Za-z0-9-]*)+$` | ProjectSetupStep |
| Team ID | exactly 10 alphanumeric chars | ProjectSetupStep |
| Project path | non-empty + `FileManager.fileExists` | ProjectSetupStep |
| Scheme | non-empty | ProjectSetupStep |
| Cert path | exists + `isReadableFile` | CertificateSetupStep |
| Key path | exists + `isReadableFile` | CertificateSetupStep |

Each rule returns `String?` (nil = valid). Inline `if let error { Text(error).foregroundColor(.red).font(.caption) }` under the field.

### UserDefaults persistence
Single key `"RemoteDeploy.setupAssistant.currentStep"` stores the int rawValue. Set in `currentStep`'s `didSet` (or via a binding wrapper). Read in `SetupAssistantView.init` to seed `@State`. Clear when the user finishes the Done step.
