## Description

`NIODeployServer.start(port:certPath:keyPath:)` hardcodes `port: 8080` for the secondary plain-HTTP listener at [RemoteDeploy/Services/NIODeployServer.swift:269](RemoteDeploy/Services/NIODeployServer.swift#L269). The HTTPS port is parameterized, but the HTTP one isn't.

Symptoms:
- **Test flakes:** `HTTPServerIntegrationTests.testPairingOverHTTP` and any sibling integration tests fail with `bind(descriptor:ptr:bytes:): Address already in use (errno 48)` whenever anything else on the machine is holding port 8080. Observed during TKT-024 and TKT-026 ship runs.
- **Production friction:** if a user already runs another service on 8080 (common — it's a popular alt-HTTP port), RemoteDeploy silently falls back to `httpChannel = nil` (note the `try?` on line 269) and the local-WiFi discovery path stops working without any visible error.

There's already a `state.httpPort: Int = 8080` field on `NIODeployServer.ServerState` ([NIODeployServer.swift:26](RemoteDeploy/Services/NIODeployServer.swift#L26)) and a public `httpPort` getter ([line 64-66](RemoteDeploy/Services/NIODeployServer.swift#L64-L66)) — but the field is never written to by `start()`. The plumbing is half-done.

## Reproduction Steps
1. Launch a second RemoteDeploy host (or any process that binds :8080) on the Mac.
2. Run `xcodebuild test -project RemoteDeploy.xcodeproj -scheme RemoteDeploy -destination 'platform=macOS'`.
3. `HTTPServerIntegrationTests.testPairingOverHTTP` fails with `errno 48`.
4. Also: companion app cannot discover the Mac over local WiFi (silent failure).

## Acceptance Criteria
- [ ] `NIODeployServer.start` accepts an `httpPort:` parameter (default 8080 for source compatibility).
- [ ] `state.httpPort` is actually set to the passed value and exposed correctly via the public getter.
- [ ] `HTTPServerIntegrationTests` and `WebSocketUpgradeTests` / `BuildManagerWebSocketIntegrationTests` pass an ephemeral random port (e.g. `Int.random(in: 49152...65000)`) in setUp, same pattern they already use for `serverPort`. No more :8080 collisions.
- [ ] Production callers (`AppDelegate.startServer`) pass 8080 explicitly, or a new `appState.httpPort` setting if we want it user-configurable.
- [ ] When the HTTP bind fails, log the error (don't silently swallow via `try?`).
- [ ] `xcodebuild test` passes reliably even when a pre-existing process holds :8080.

## Investigation

### Current state

`NIODeployServer.start(...)` takes `port`, `certPath`, `keyPath` — no HTTP port. Inside, line 269:

```swift
let httpChannel = try? await httpBootstrap.bind(host: "0.0.0.0", port: 8080).get()
```

Three things wrong with this line:

1. **Hardcoded port.** Should be a parameter.
2. **`try?` swallows the error silently.** A port conflict becomes `httpChannel = nil`, no log, no `appState.currentError`, no user-facing signal. The HTTP listener just… doesn't start. Companion discovery silently breaks.
3. **`state.httpPort` is dead.** It defaults to 8080 but `start()` never assigns it from the actual bind result, so the public `httpPort` getter returns the default even if the bind happened on a different port in the future.

The integration tests already know how to pick random ports — `HTTPServerIntegrationTests.swift:86` does `serverPort = Int.random(in: 49152...65000)`. Same pattern just needs to extend to httpPort.

### Call sites that need updating

- **Production:** `AppDelegate.startServer()` at [RemoteDeploy/AppDelegate.swift:347](RemoteDeploy/AppDelegate.swift#L347) calls `server.start(port:, certPath:, keyPath:)`. One call site.
- **Integration tests:**
  - `RemoteDeployIntegrationTests/HTTPServerIntegrationTests.swift` — 11 call sites of `server.start(...)`, all in individual test methods.
  - `RemoteDeployIntegrationTests/WebSocketUpgradeTests.swift` — 2 call sites.
  - `RemoteDeployIntegrationTests/BuildManagerWebSocketIntegrationTests.swift` — 1 call site.

All integration tests will need an `httpPort = Int.random(in: 49152...65000)` in setUp and to pass it through. Non-trivial diff but mechanical.

### Regression risk

- **Low.** The default-parameter approach (`httpPort: Int = 8080`) keeps the existing `AppDelegate` call site working with no change. The only runtime behavior change is:
  - **Better**: HTTP bind errors now get logged instead of silently swallowed.
  - **Neutral**: integration tests no longer collide on :8080.
- **Companion Bonjour TXT record** publishes `httpPort` (see [BonjourAdvertiser.buildTXTRecord](RemoteDeploy/Services/BonjourAdvertiser.swift)) and the companion reads it from the TXT. If production starts using a dynamic port someday, the TXT already carries the real value so companion discovery should still work. But for now, production keeps 8080 by default to match existing behavior and avoid requiring a settings migration.

## Proposed Solution

Three small changes in two commits. Low total surface area (~10 files touched for tests, 2 for production).

### Commit 1 — Make port configurable, stop swallowing errors

1. Add `httpPort: Int = 8080` parameter to `NIODeployServer.start(port:certPath:keyPath:)`. Default preserves the existing production call site unchanged.
2. Inside `start()`, write `state.httpPort = httpPort` before the bind (already the pattern for `state.port = port` on line 189).
3. Replace the `try?` with a real `do { try await ... } catch { Logger.server.error(...) }` so HTTP bind failures are visible. Don't throw — keep the "HTTP is best-effort, HTTPS is the primary path" semantics.
4. Update `AppDelegate.startServer()` to pass `httpPort: 8080` explicitly — documents the default at the call site rather than hiding it in `start()`.

### Commit 2 — Integration tests use random http ports

Update every `server.start(...)` call site in the three integration test files to pass a random `httpPort` from setUp. Pattern:

```swift
// In setUp:
serverPort = Int.random(in: 49152...65000)
httpPort = Int.random(in: 49152...65000)
while httpPort == serverPort { httpPort = Int.random(in: 49152...65000) }

// In each test:
try await server.start(port: serverPort, httpPort: httpPort, certPath: certPath, keyPath: keyPath)
```

### Why this approach

- **Default parameter over breaking change**: zero churn on production call sites.
- **Log on failure over throw**: HTTP listener is best-effort (companion can also reach us via HTTPS+Tailscale); throwing would regress the user-visible behavior for anyone on a cellular/no-Tailscale setup with WiFi blocked.
- **Random port in tests over "kill conflicting processes"**: makes the test suite hermetic. No more test flakes from the dev running the actual host app in the background.

### Tradeoffs

- Not exposing `httpPort` as a user-configurable setting in `AppState` / settings UI. That's a bigger change (new settings field, persistence, SwiftUI wiring) and out of scope for this ticket. Filed mental note: if a user reports the :8080 conflict in production, follow-up ticket to expose it.
