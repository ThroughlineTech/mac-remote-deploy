## Description

Split into `NIODeployServer.swift` (lifecycle + project registration), `HTTPHandler.swift` (request routing), `NIOResponseGenerator.swift` (manifest/install page/IPA serving). Pure structural refactor.

## Investigation

### Current state of NIODeployServer.swift
**685 lines** containing two top-level types:

1. **`NIODeployServer` class** ([lines 12-252](RemoteDeploy/Services/NIODeployServer.swift#L12-L252), ~250 lines)
   - State (event loop group, channels, locks, projects map, base URL)
   - Project registration (`registerProject`, `unregisterProject`, `setBaseURL`, `registeredProjects`, `currentBaseURL`)
   - Server lifecycle (`start`, `stop`)
   - Protocol conformance + properties

2. **`HTTPHandler` class** ([lines 263-685](RemoteDeploy/Services/NIODeployServer.swift#L263-L685), ~430 lines)
   - Channel handler conformance (`channelRead`)
   - Request dispatch (`handleRequest`) — 130 lines including the switch over `subpath` for install page / manifest / IPA
   - Response writers (`sendDataResponse`, `sendResponse`, `sendFileResponse`, `sendNotFound`) — ~100 lines
   - Index page generator (`buildIndexPage`) — ~40 lines
   - PWA static file server (`servePWAFile`) — ~55 lines
   - HTML escaping helper

The `HTTPHandler` is the bulk of the file. Its `handleRequest()` method does both **routing** (figure out which response to send) and **inline response generation** (build the install page HTML, generate the manifest XML inline, serve the IPA file inline).

### Coupling and access constraints
- `HTTPHandler` has stored properties (`server`, `requestHead`, `requestBody`, `apiRouter`, `rejectPairingOverHTTP`) — these must stay in the file that declares the class.
- The `send*` helpers all take `ChannelHandlerContext` and use `context.channel.allocator` / `wrapOutboundOut`. They depend on `HTTPHandler.OutboundOut` typealias, so they have to live as instance methods on `HTTPHandler` (or as extension methods on it).
- `buildIndexPage` and `servePWAFile` similarly need to call `sendResponse` / `sendNotFound`. They can move to an extension on `HTTPHandler` in another file.
- `handleRequest`'s `case _ where subpath.isEmpty`, `case "manifest.plist"`, `case "app.ipa"` blocks each contain inline response building. Extracting them to private methods is a small but worthwhile refactor that creates clean targets for the third file.
- `manifestGenerator` and `installPageGenerator` are declared `fileprivate` on `NIODeployServer` ([lines 33, 36](RemoteDeploy/Services/NIODeployServer.swift#L33-L36)) — `fileprivate` works only within the same file, so splitting requires upgrading these to `internal`.

### Tests covering the affected code
- [HTTPServerIntegrationTests.swift](RemoteDeployIntegrationTests/HTTPServerIntegrationTests.swift) — 11 tests that spin up a real `NIODeployServer` with a self-signed cert and exercise: install page rendering, manifest serving, IPA download, API pairing flow, auth, all-in. **This is the safety net for TKT-015.**
- [APIRouterFactoryTests.swift](RemoteDeployTests/APIRouterFactoryTests.swift) + the new [RemoteDeployTests/API/](RemoteDeployTests/API/) suite from TKT-003 — 93+ unit tests covering the API router and route handlers. They use mock dependencies so they don't exercise `NIODeployServer` directly, but they exercise the same `APIRequest`/`APIResponse` types and `apiRouter` field.

### Regression Risk — **LOW-MEDIUM**
- **Pure structural refactor.** No logic changes intended.
- Risk vectors:
  1. **Access modifier mistakes.** `private` → `internal` (or `fileprivate` → `internal`) on a few helpers; if I miss one, build fails clearly.
  2. **Lost imports.** Each new file needs `import NIO`, `import NIOHTTP1`, etc. — easy to forget, build fails clearly.
  3. **Capturing `self` in extension methods.** Should work the same as the original methods, but worth a careful review.
  4. **Subtle behavior change in extracted handler methods.** If I extract `serveInstallPage`/`serveManifest`/`serveIPA` from the inline switch statement, the call sites must pass the same arguments and the methods must do exactly what the inline code did. Mitigation: extract one at a time, run tests after each.
- The 11 integration tests cover every observable behavior of the server. If they pass, the refactor is correct.

## Proposed Solution

### Approach: extract-then-split

**Phase A — Refactor in place** (still one file, behavior-equivalent intermediate state):
- Extract `serveInstallPage(context:project:slug:)`, `serveManifest(context:project:slug:)`, `serveIPA(context:project:slug:head:)` as new private methods on `HTTPHandler`. Move the inline code from `handleRequest`'s switch into these methods. The switch then becomes three method calls.
- Run tests → all 247 still passing.

**Phase B — Split into three files:**

| File | Contents | Estimated size |
|---|---|---|
| `RemoteDeploy/Services/NIODeployServer.swift` | The `NIODeployServer` class only — state, registration, lifecycle. Upgrade `manifestGenerator`/`installPageGenerator` from `fileprivate` to `internal`. | ~252 lines |
| `RemoteDeploy/Services/HTTPHandler.swift` | The `HTTPHandler` class declaration with stored properties + `channelRead` + `handleRequest` (the routing dispatcher) + the `send*` response writers + `htmlEscape` helper. | ~280 lines |
| `RemoteDeploy/Services/NIOResponseGenerator.swift` | `extension HTTPHandler { ... }` containing `buildIndexPage`, `servePWAFile`, `serveInstallPage`, `serveManifest`, `serveIPA`. These are the "what to actually send" methods that the routing dispatcher delegates to. | ~150 lines |

### Why this split
- **NIODeployServer.swift stays focused on the server lifecycle.** Anyone reading it learns "what is this thing and how does it start/stop" without wading through HTTP request parsing.
- **HTTPHandler.swift is the routing dispatcher.** Reading it tells you "what URL maps to what response," with low-level write helpers as a footer. The heavy response-building code lives elsewhere.
- **NIOResponseGenerator.swift is where you go to change response content.** "How do I tweak the install page?" → open this file. The extension pattern keeps the methods on `HTTPHandler` so they have full access to channel context, but separates them physically.

### Alternative considered + rejected
- **4-file split**: extract send writers into their own file. Cleaner separation of concerns, but the writers are <100 lines and tightly bound to `HTTPHandler.OutboundOut`. Splitting them adds an import + extension declaration for marginal benefit. Stuck with 3 to match the ticket's stated target.
- **Move response writers to a `NIOWriteHelpers.swift` protocol**: overkill for a refactor ticket. Out of scope.

### Tradeoffs
- **Extension methods can't have stored properties.** Not a problem here since none of the methods being moved store anything.
- **`fileprivate` → `internal` upgrades.** Two properties (`manifestGenerator`, `installPageGenerator`) become `internal`. They're already internal-by-default to the module — `fileprivate` was the only thing keeping them file-scoped. The change widens access from "same file" to "same module," which is the same scope `internal` already has for everything else in the codebase. No real visibility cost.
