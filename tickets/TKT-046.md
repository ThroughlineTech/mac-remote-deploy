## Description
In the iOS companion app's Build tab, the build log output streams into the view and auto-scrolls continuously, creating "doom scroll hell". Two specific problems:

1. The log is shown expanded by default, dominating the screen below the Build & Deploy button — it should be collapsible (and collapsed by default) so the primary action and status stay the focus.
2. The log keeps auto-scrolling long after the build has finished ("Build Succeeded"). Once the build completes, new lines shouldn't be yanking the viewport around.

Having the full log content available is valuable for debugging — the ask is about *presentation*, not removing data. A disclosure/expand control that reveals the log on demand (and a sensible auto-scroll policy that stops once the build is done) would fix both issues.

## Reproduction Steps (bugs only)
1. Open the companion app and tap the Build tab.
2. Tap "Build & Deploy".
3. Observe the log area below the button fills with output and auto-scrolls.
4. Wait for "Build Succeeded" to appear.
5. Notice the view continues to scroll as late log lines arrive, even after the build is marked complete.

## Acceptance Criteria
- [ ] Build log is collapsed by default on the Build tab — a disclosure control (chevron/button) expands it on demand.
- [ ] When collapsed, the primary Build & Deploy button and build status remain the dominant UI.
- [ ] Auto-scroll only happens while a build is actively running; once the build reaches a terminal state (succeeded/failed), auto-scroll stops.
- [ ] When the log is expanded, the user can manually scroll through it without being fought by auto-scroll after build completion.
- [ ] Starting a new build re-enables auto-scroll for that build.

### Where the log view lives

The Build tab is [RemoteDeployCompanion/Views/Build/BuildControlView.swift](RemoteDeployCompanion/Views/Build/BuildControlView.swift). Two views live in this file:

1. **`BuildControlView`** ([lines 7-186](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L7-L186)) — the Build tab itself. Hierarchy (top to bottom):
   - `NavigationStack` → outer `VStack(spacing: 0)`
   - Project picker + Build & Deploy button + status row (all inside a padded `VStack`, [L25-L122](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L25-L122))
   - `Divider()` ([L124](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L124))
   - `BuildLogStreamView()` ([L127](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L127)) — currently rendered as a sibling that takes all remaining vertical space
   - `.toolbar` containing a trash button that calls `webSocketClient.clearLog()` ([L131-L137](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L131-L137))

2. **`BuildLogStreamView`** ([lines 189-222](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L189-L222)) — the scrolling log itself.
   - `@EnvironmentObject var webSocketClient: WebSocketClient` (since TKT-043)
   - Body: `ScrollViewReader { proxy in ScrollView { LazyVStack { ForEach(webSocketClient.buildLogLines.enumerated()) { ... }}}}`
   - Each line is a `Text(line)` with `.id(index)`, tinted red for `error:` and yellow for `warning:` ([L197-L201, L217-L221](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L197-L221)).
   - Background: `Color(.systemGroupedBackground)`.

### How auto-scroll works today

At [L207-L213](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L207-L213):

```swift
.onChange(of: webSocketClient.buildLogLines.count) {
    if let last = webSocketClient.buildLogLines.indices.last {
        withAnimation {
            proxy.scrollTo(last, anchor: .bottom)
        }
    }
}
```

This fires on **every append** to `buildLogLines`, with no gating whatsoever. Because xcodebuild's tail/stdout can keep streaming lines for a brief window after the build reaches a terminal state (and because some broadcasts arrive on the WebSocket out of order with the `buildstatus` frame that flips `isBuilding` to `false`), the log continues to yank the viewport even though "Build Succeeded" is already showing. Any manual scroll attempt by the user is immediately overwritten.

Note: `BuildLogStreamView` does not currently observe `BuildManager`, so it has no local signal about whether the build is still in progress. It only sees `webSocketClient.buildLogLines`.

### State the view can gate on

`BuildManager` ([RemoteDeployCompanion/Services/BuildManager.swift](RemoteDeployCompanion/Services/BuildManager.swift)) already publishes everything needed:

- `@Published var isBuilding = false` ([L16](RemoteDeployCompanion/Services/BuildManager.swift#L16)) — flipped to `false` in three places:
  - REST polling termination branches `"success"`, `"failure"`, `"idle"` ([L82-L97](RemoteDeployCompanion/Services/BuildManager.swift#L82-L97))
  - `cancelBuild()` ([L65](RemoteDeployCompanion/Services/BuildManager.swift#L65))
  - WebSocket `observeWebSocketStatus` on a `"success"` or `"failure"` frame ([L129-L132](RemoteDeployCompanion/Services/BuildManager.swift#L129-L132)) — this is the fast path (TKT-040)
- `@Published var buildStatus: BuildStatusInfo?` — carries `state` strings `"building" / "success" / "failure" / "idle"`.

`BuildControlView` already has `@EnvironmentObject var buildManager: BuildManager` at [L9](RemoteDeployCompanion/Views/Build/BuildControlView.swift#L9) (post TKT-043). `BuildLogStreamView` does NOT — it only has `webSocketClient`. Adding `@EnvironmentObject var buildManager: BuildManager` to `BuildLogStreamView` is trivial and consistent with TKT-043's pattern. The injection is in place via the app root (confirmed by `BuildControlView` being able to use it — same environment chain).

### How `buildLogLines` is populated

`WebSocketClient.buildLogLines: [String]` ([RemoteDeployCompanion/Services/WebSocketClient.swift:17](RemoteDeployCompanion/Services/WebSocketClient.swift#L17)).

- Reset to `[]` at the start of each `openSocket()` call ([L90](RemoteDeployCompanion/Services/WebSocketClient.swift#L90)).
- Appended on every `buildlog` WebSocket frame ([L178](RemoteDeployCompanion/Services/WebSocketClient.swift#L178)).
- Cleared manually via `clearLog()` ([L127-L129](RemoteDeployCompanion/Services/WebSocketClient.swift#L127-L129)), wired to the trash toolbar button.

There is no per-build reset — old log lines persist in the array until the socket reconnects or the trash is tapped. That means when a fresh build starts, new lines are appended to the tail of the existing buffer. This is relevant to the "re-enable auto-scroll on new build" acceptance criterion: we can detect a new build via `buildManager.isBuilding` flipping `false → true`, not via the line count resetting.

## Proposed Solution

Wrap `BuildLogStreamView` in a `DisclosureGroup` inside `BuildControlView`, driven by `@State private var isLogExpanded = false` — so the log is **collapsed by default** with a "Build Log" header and chevron. When expanded, the log view appears below the disclosure row and takes the remaining vertical space as it does today.

Inside `BuildLogStreamView`, inject `@EnvironmentObject var buildManager: BuildManager` and gate the existing `proxy.scrollTo(last, anchor: .bottom)` call on `buildManager.isBuilding == true`. This is the smallest, most surgical change that satisfies the "auto-scroll only while building" requirement: once `BuildManager` flips `isBuilding` to `false` (which TKT-043 already makes reliable via either the WebSocket fast path or the REST poll fallback), the `.onChange` handler will no-op for any late log lines, leaving the viewport wherever the user last saw it or scrolled to.

To also satisfy "Starting a new build re-enables auto-scroll for that build", we add an `.onChange(of: buildManager.isBuilding)` handler that — when the value transitions from `false → true` — snaps the scroll to the bottom once, establishing the initial position for the new build's stream. After that, the normal per-line auto-scroll takes over (gated on `isBuilding`). No additional state is needed.

**Collapse-state lifetime:** the disclosure should **reset to collapsed at the start of each new build** — i.e. when `buildManager.isBuilding` flips `false → true`, we also set `isLogExpanded = false`. Rationale: the motivating complaint is that the log dominates the Build tab; leaving it expanded across builds would recreate exactly that problem. A user who wants to keep watching can tap the chevron again; the cost of one tap per build is small compared to a screen full of logs each time. (The alternative — persisting expansion across builds — is rejected because it leaks the "doom scroll" back into the default experience for anyone who expanded it once and forgot.)

### Why `DisclosureGroup` over alternatives

- **Fixed-height collapsed region with a "Show log" button.** Rejected: still consumes vertical real-estate permanently, and re-implements what `DisclosureGroup` gives us for free.
- **Modal sheet presenting the log.** Rejected: too heavy for a quick glance, and breaks the "primary action and status stay the focus" criterion by requiring a round-trip. Users expect build logs to be available inline on iOS developer tooling.
- **Separate "Log" tab.** Rejected: adds navigation cost for what is, by nature, a secondary/contextual debugging aid. Also fragments the mental model of a single Build tab.
- **`DisclosureGroup` inline.** Chosen: native SwiftUI control, one-tap toggle, keeps the log in place contextually, zero navigation overhead, preserves scroll position automatically via SwiftUI's view identity rules, and matches the user's stated ask ("disclosure control (chevron/button)").

### Edge cases covered

- **Log cleared via toolbar trash while expanded** — `.onChange(of: buildLogLines.count)` fires with `count == 0`; `indices.last` returns nil, the guard skips, nothing to scroll to. Safe.
- **New build starts while the log is currently expanded from the previous build** — `isLogExpanded = false` reset snaps it shut. User can re-expand if they want to watch streaming.
- **User manually scrolls up mid-build to read an earlier line** — auto-scroll will still yank them back, because `isBuilding` is still true. This is intentional and matches today's behavior during an active build; the fix is narrowly about what happens **after** the build finishes. A "pause auto-scroll when user has scrolled up" behavior is a larger interaction design and out of scope for this ticket.
- **Build canceled via the cancel button** — `BuildManager.cancelBuild()` sets `isBuilding = false` ([BuildManager.swift:65](RemoteDeployCompanion/Services/BuildManager.swift#L65)), so auto-scroll stops for any trailing log lines. Correct.
- **REST poll terminal path vs WebSocket terminal path** — both set `isBuilding = false`; the gate is agnostic to which one fires first.
