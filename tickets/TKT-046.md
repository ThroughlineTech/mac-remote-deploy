## Description
In the iOS companion app's Build tab, the build log output streams into the view and auto-scrolls continuously, creating "doom scroll hell". Two specific problems:

1. The log is shown expanded by default, dominating the screen below the Build & Deploy button — it should be collapsible (and collapsed by default) so the primary action and status stay the focus.
2. The log keeps auto-scrolling long after the build has finished ("Build Succeeded"). Once the build completes, new lines shouldn't be yanking the viewport around.

Having the full log content available is valuable for debugging — the ask is about *presentation*, not removing data. A disclosure/expand control that reveals the log on demand (and a sensible auto-scroll policy that stops once the build is done) would fix both issues.

## Reproduction Steps (bugs only)
1. Open the companion app and tap the Build tab.
2. Tap "Build & Deploy".
3. Observe the log area below the button fills with output and auto-scrolls.
4. Wait for "Build Succeeded" to appear.
5. Notice the view continues to scroll as late log lines arrive, even after the build is marked complete.

## Acceptance Criteria
- [ ] Build log is collapsed by default on the Build tab — a disclosure control (chevron/button) expands it on demand.
- [ ] When collapsed, the primary Build & Deploy button and build status remain the dominant UI.
- [ ] Auto-scroll only happens while a build is actively running; once the build reaches a terminal state (succeeded/failed), auto-scroll stops.
- [ ] When the log is expanded, the user can manually scroll through it without being fought by auto-scroll after build completion.
- [ ] Starting a new build re-enables auto-scroll for that build.
