# Chain Review — 2026-04-11 10:55

Delegated batch collect of 2 companion-app UI bugs. Both tickets fix the same underlying class of issue (nested `ObservableObject` observation through non-`@Published` properties on `ConnectionManager`) in two different places: the Build tab (TKT-043) and the Projects detail view (TKT-044).

## Context — why these two tickets travel together

Both symptoms came from the same architectural flaw: `ConnectionManager` holds `buildManager: BuildManager` and `webSocketClient: WebSocketClient` as plain `let` properties. Because SwiftUI's `@EnvironmentObject`/`@ObservedObject` only observes the object it's bound to, child views that reached through `connectionManager.buildManager.isBuilding` were never invalidated when `BuildManager` published a change — `ConnectionManager`'s own `objectWillChange` never fired for nested changes.

Each ticket fixes its leaf view independently by observing the nested objects directly. Neither ticket touches `ConnectionManager` itself — that centralizing refactor would be a separate follow-up.

## Cross-cutting review

1. **Two different SwiftUI patterns for the same fix.** TKT-043 uses `@EnvironmentObject` injected at the tab level; TKT-044 uses `@ObservedObject` passed via init. Both are correct. The inconsistency is minor and not worth unifying. Preference: when both land on main, neither should feel out of place — but future companion views that need to observe `BuildManager` should pick one style consistently.

2. **The real fix is missing.** Neither ticket addresses the root cause in `ConnectionManager`: `buildManager` and `webSocketClient` should arguably be `@Published` or `ConnectionManager` should explicitly forward their `objectWillChange`. As it stands, **any other view that observes `connectionManager.buildManager.X` is at risk of the same bug.** Consider opening a follow-up ticket to centralize the fix in `ConnectionManager`, after which both TKT-043 and TKT-044's workarounds could be simplified.

3. **Test reports claim 311 tests passing** on the host `RemoteDeploy` scheme for both tickets. Neither ticket added tests that actually exercise the SwiftUI observation path end-to-end on the companion side. Manual verification is load-bearing here.

4. **Branches do NOT contain each other.** Both were created from `main` before either fix landed. When shipping both:
   - Ship one (e.g. TKT-043) first via `/ticket-ship TKT-043`.
   - Then `git rebase main` the second branch (TKT-044). There should be no conflicts — the two fixes touch different files (`BuildControlView.swift` vs `ProjectListView.swift`, different test files). The only shared file is `RemoteDeployCompanionApp.swift` which only TKT-043 touches.
   - Or use `/ticket-chain --ship TKT-043 TKT-044` which handles the rebase+ship sequence.
