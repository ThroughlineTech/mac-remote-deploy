## Description
On app launch, the menu bar icon shows the "disconnected" state (antenna.radiowaves.left.and.right.slash) and stays that way until the user clicks it. Clicking the icon opens the popover, which then shows "Server Running" and "Tailscale Connected" — meaning the startup work *did* happen, but either:

1. **UI-only bug:** the `MenuBarExtra` label isn't re-rendering when `appState.tailscaleConnected` / `serverRunning` flip to true, so the icon stays stale until the popover is opened. Cosmetic but confusing.
2. **Worse bug:** `performStartup()` is hooked to `MenuBarView.task { ... }`, which SwiftUI only invokes when the popover *first renders* — i.e. when the user clicks the icon. In that case, the server literally does not start until the user clicks, and pairing/OTA installs/API calls all fail silently for any period before that first click.

Suspicion leans toward #2 based on a read of [RemoteDeployApp.swift:37-42](RemoteDeploy/RemoteDeployApp.swift#L37-L42): `performStartup()` is invoked from a `.task { }` modifier on `MenuBarView`, gated by a `hasLaunched` flag. `MenuBarView` is the popover contents, not the `MenuBarExtra` label — so the task only runs after the first popover open.

Either way this needs fixing. If it's #2 (server doesn't start), it's a serious regression from user expectations of a menu-bar utility: the server should be listening from the moment the user logs in.

## Reproduction Steps
1. Quit RemoteDeploy (or restart the Mac so the app cold-launches)
2. Relaunch RemoteDeploy via Spotlight / Launchpad / login item
3. Observe the menu bar — icon shows disconnected state
4. Without clicking the menu bar icon, try from another device: open `https://<mac>.tailnet.ts.net:8443/` or hit `/api/v1/status`
5. Expected: server responds. Actual (suspected): server is not listening yet because `performStartup()` hasn't fired.
6. Click the menu bar icon — popover opens showing correct status, server is now running

- [x] The HTTPS server is listening on port 8443 within a few seconds of app launch, before any user interaction

## Investigation

### Root cause — CONFIRMED: worse bug (#2 from the ticket)

`performStartup()` is wired to a `.task { }` modifier on `MenuBarView` at [RemoteDeployApp.swift:39-44](RemoteDeploy/RemoteDeployApp.swift#L39-L44):

```swift
MenuBarExtra {
    MenuBarView()
        .environmentObject(appState)
        .environmentObject(serviceContainer)
        .environmentObject(buildManager)
        .task {
            guard !hasLaunched else { return }
            hasLaunched = true
            await performStartup()
        }
        ...
} label: {
    Label("RemoteDeploy", systemImage: appState.menuBarIconName)
}
```

**The key architectural mistake:** `MenuBarView()` is inside the `MenuBarExtra`'s content closure — i.e. it's the popover content, not the label. SwiftUI does NOT render popover content until the user clicks the menu bar icon. The `.task` modifier fires on first appearance, so `performStartup()` literally does not run until the first click.

What this means in practice:
- App cold-launches → `MenuBarExtra` label appears → `appState.menuBarIconName` is evaluated against default state (`tailscaleConnected = false`, `serverRunning = false`) → icon renders the "disconnected" antenna symbol
- No Tailscale check happens, no settings load, no projects load, no server start, no status polling, no BuildManager.configure, no API router wiring
- User clicks menu bar icon → popover renders for the first time → `.task` fires → `performStartup()` runs → server starts, settings load, status polling begins, icon updates to reflect real state
- Before that first click, the app is effectively inert: HTTPS server not listening on 8443, `/api/v1/status` returns nothing (no socket), iOS companion can't reach it, no OTA install links work

This is a regression-category bug. Users expect a menu-bar utility to start its server when the app launches, not "when you remember to click it."

### Why the label didn't alert us

The `label:` closure at [line 52-54](RemoteDeploy/RemoteDeployApp.swift#L52-L54) reads:
```swift
} label: {
    Label("RemoteDeploy", systemImage: appState.menuBarIconName)
}
```

This DOES observe `appState` via `@StateObject` → `@Published`, so the label would update when `tailscaleConnected`/`serverRunning` flip. But since `performStartup()` never runs, those values never flip. The label correctly renders the initial empty state — forever, until the user clicks.

### The four-step startup that's being skipped

[RemoteDeployApp.swift:95-137](RemoteDeploy/RemoteDeployApp.swift#L95-L137) — `performStartup()` does the following, all of which are being skipped until first click:

1. `serviceContainer.notificationManager.requestPermission()` — macOS notification permission prompt
2. `buildManager.configure(...)` — wires build dependencies (build engine, deploy server, notification manager, IPA importer, build history store) + push notification callback
3. `loadSettings()` — loads cert paths, hostname, push config from `settings.json`
4. `loadSavedProjects()` — reads projects from `UserDefaultsProjectStore`
5. `serviceContainer.configurePushNotifiers(...)` — instantiates Prowl/Pushover/ntfy clients from config
6. `await checkTailscaleStatus()` — queries `tailscale status`, sets `hostname` and `serverURL`
7. `startServer()` — the whole point of the app; starts NIO HTTPS listener on 8443 + HTTP on 8080, registers projects, wires API router
8. Show setup assistant if no projects
9. `startStatusPolling()` — 30s Tailscale status polling loop

**Everything above is deferred until the first popover click.** The server doesn't exist on the network. The iOS companion's Bonjour discovery will find nothing (because `bonjourAdvertiser.start(...)` is called inside `startServer()` at [line 281-287](RemoteDeploy/RemoteDeployApp.swift#L281-L287)).

### Why `MenuBarExtra` + `.task` is the wrong pattern for this

SwiftUI's `MenuBarExtra` with `.menuBarExtraStyle(.window)` renders the popover contents lazily — only when the popover opens. This is a conscious design choice by Apple to avoid running arbitrary view work when the app is idle. It's the right default for the 99% case of menu bar apps that only need to run logic when the user actually interacts.

For RemoteDeploy, we need the opposite: start real work on app launch regardless of user interaction. The correct place for that is an `NSApplicationDelegateAdaptor` hooked into `NSApplicationDelegate.applicationDidFinishLaunching(_:)`, which fires exactly once when the app finishes launching — no view rendering required.

### Existing tests
- No integration test covers "is the server listening after app launch." [HTTPServerIntegrationTests.swift](RemoteDeployIntegrationTests/HTTPServerIntegrationTests.swift) spins the server up directly via `NIODeployServer.start(...)`, bypassing `performStartup()` entirely, so it doesn't catch this bug.
- Zero tests touch `RemoteDeployApp.performStartup()` — it's a private method on a `struct App: App`, which is intrinsically hard to test.
- All 270 tests currently pass despite this bug because they don't exercise the app-launch code path.

### Regression Risk — **LOW**
The fix is a single architectural move: take the body of `performStartup()` and run it from an `NSApplicationDelegate.applicationDidFinishLaunching(_:)` instead of from `.task` on `MenuBarView`. The work itself doesn't change; only *when* it runs changes. The existing `hasLaunched` guard continues to prevent double-runs.

Risks:
1. **MainActor isolation.** `performStartup()` is currently implicitly `@MainActor` because it runs in a SwiftUI context. Moving it to `AppDelegate.applicationDidFinishLaunching` still runs on the main thread (AppKit lifecycle callbacks are main-thread), but the `@MainActor` annotation may need to be explicit.
2. **`@StateObject` access from a non-view context.** `AppState`, `ServiceContainer`, `BuildManager` are currently declared as `@StateObject` on the `RemoteDeployApp` struct. AppDelegate can't easily reach into them. We need to either:
   - (a) Move the state objects to the delegate and inject them as `@EnvironmentObject` from there via `@ObservedObject` + `StateObject` pattern
   - (b) Keep the state objects on the `App` struct and pass them into the delegate via a handshake during `init` or via `NSApplicationDelegateAdaptor` configuration
3. **Ordering:** the delegate fires `applicationDidFinishLaunching` after SwiftUI has started rendering scenes. Need to verify the label closure picks up state updates made from the delegate.

### Alternative considered: `.onAppear` on the label closure
Doesn't work. `Label` is not a container view that owns lifecycle modifiers meaningfully; `.onAppear` inside `label: { }` doesn't reliably fire before first click either. The AppDelegate is the correct place.

### Alternative considered: a dummy invisible `Window` that fires `.task` at launch
This is a hack. Opens a real AppKit window (even if zero-sized) purely to get a `.task` to fire. Brittle and would show in Mission Control or Window menu. Rejected.

## Proposed Solution

### Approach: `NSApplicationDelegateAdaptor` kick-starts performStartup()

Use SwiftUI's `@NSApplicationDelegateAdaptor` to install a delegate class that runs the startup work in `applicationDidFinishLaunching(_:)`. The delegate gets handed the `@StateObject`s via a "register" method called from the `App` struct's `body`.

### Shape of the change

1. New `final class AppDelegate: NSObject, NSApplicationDelegate` in `RemoteDeployApp.swift` (or its own file). Has optional references to `AppState`, `ServiceContainer`, `BuildManager`, and a closure that runs the actual startup body.

2. `RemoteDeployApp` adds `@NSApplicationDelegateAdaptor(AppDelegate.self) private var appDelegate`.

3. At the top of `body`, before the `MenuBarExtra` scene, wire the state objects into `appDelegate` via a method call. This happens every time `body` is evaluated but is idempotent because `appDelegate` only triggers startup once.

4. Remove the `.task { ... performStartup() }` modifier from `MenuBarView`.

5. Move the existing `performStartup()` body into a method the delegate can call. Either:
   - Keep `performStartup()` on `RemoteDeployApp` and have the delegate reach back via a closure, OR
   - Move it entirely onto the delegate itself.
   
   **Decision:** option (b) — move the whole startup body onto `AppDelegate`. Keeps responsibilities clean and avoids the weird "closure on a struct" dance. The delegate holds a `Startup` type that takes appState/serviceContainer/buildManager as init params.

### Idempotency + ordering

The delegate's `applicationDidFinishLaunching(_:)` runs exactly once per launch. We don't need the `hasLaunched` `@State` guard anymore — can delete it.

Ordering concern: the scene's `body` runs *before* `applicationDidFinishLaunching`, because SwiftUI constructs the scene graph first. But the scene body assigning state objects to the delegate is synchronous, so by the time `applicationDidFinishLaunching` fires, the delegate has everything it needs.

Test: if `body` evaluates before the delegate has been created, we might crash trying to assign. `@NSApplicationDelegateAdaptor` creates the delegate eagerly at `App.init` — confirmed by Apple docs. So the delegate exists before `body` runs, and the assignment is safe.

### Tradeoffs
- **One more class.** Adds an `AppDelegate` type. Small price.
- **Initial state observability:** now that startup runs immediately, the label closure reading `appState.menuBarIconName` will observe the transition from "disconnected" → "connected" naturally. No additional view changes needed.
- **Test coverage gap persists.** Integration-testing `NSApplicationDelegateAdaptor` lifecycle from XCTest is hard — you can't easily launch an `App` struct in-process from a test. I'll write a focused unit test on a new `Startup` type (or equivalent) that exercises the startup sequence with mock services, proving the logic runs correctly when invoked, even if we can't prove SwiftUI wires the delegate correctly at runtime without a real launch.

### Why not just move `performStartup()` inside `RemoteDeployApp.init()`?

Tempting but wrong. The `App` struct's `init()` is called before SwiftUI has finished setting up the environment, `@StateObject` machinery, or the AppKit event loop. Doing async work or touching AppKit-level stuff (like `NSApplication.shared.setActivationPolicy`) from `init()` is fragile. `applicationDidFinishLaunching` is the sanctioned hook.
