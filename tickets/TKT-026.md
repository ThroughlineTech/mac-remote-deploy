## Description

Xcode reports six warnings in `RemoteDeployCompanion` during normal builds. None block App Store submission in Swift 5 language mode, but all six will become hard errors if/when the project moves to Swift 6 language mode. One of them (`KeychainStore` lock/unlock across async) is also a **latent correctness bug**, not purely a style issue — holding a `Lock` across an `await` can cause real deadlocks or races because the resuming thread may not be the locking thread.

Observed during a TKT-024 build on 2026-04-08:

### `BonjourBrowser.swift`
- **"Switch must be exhaustive"** — a switch statement is missing a case (or `@unknown default`). 1-line fix.

### `KeychainStore.swift`
- **"Add `@preconcurrency` to suppress 'Sendable'-related warnings from module 'LocalAuthentication'"** — the `LocalAuthentication` framework hasn't been audited for Sendable. `@preconcurrency import LocalAuthentication` silences it.
- **"Instance method 'lock' is unavailable from asynchronous contexts; Use async-safe scoped locking instead; this is an error in the Swift 6 language mode"** (×1)
- **"Instance method 'unlock' is unavailable from asynchronous contexts; Use async-safe scoped locking instead; this is an error in the Swift 6 language mode"** (×2)
- **"Capture of 'context' with non-Sendable type 'LAContext' in a '@Sendable' closure"** — `LAContext` isn't Sendable and is being captured across an actor/task boundary.

## Severity ranking

1. **`KeychainStore` lock/unlock across async (the three ×1, ×2 warnings)** — latent correctness bug. Fix first regardless of Swift 6 timeline. Use `OSAllocatedUnfairLock` (same pattern the host app already uses in `JSONBuildHistoryStore.swift` and elsewhere) or restructure so no lock is held across `await`.
2. **`LAContext` Sendable capture + `@preconcurrency` import** — related, both address Sendable audit of `LocalAuthentication`. Adding `@preconcurrency import LocalAuthentication` may or may not cover the `LAContext` closure capture; likely need both. Estimate: 5 minutes.
3. **`BonjourBrowser` exhaustive switch** — 1-line fix. Do whenever.

## Investigation

Re-read both target files end-to-end. Each warning maps to a specific, well-understood root cause.

### Correction to the ticket description

The original ticket body flagged `KeychainStore`'s `NSLock` usage as a **latent correctness bug** ("holding a lock across an await can deadlock"). A close re-read shows this is **not actually happening** in the current code:

[RemoteDeployCompanion/Services/KeychainStore.swift:91-97](RemoteDeployCompanion/Services/KeychainStore.swift#L91-L97) locks, checks the cache, and unlocks cleanly **before** the `await authenticatedContext()` call at line 112. The lock is never held across a suspension point. Swift 6 flags it anyway because the compiler can't prove locks are scoped correctly — the checker is conservative, but the current code is actually correct.

The severity ranking in the ticket description should be downgraded accordingly: this is a **Swift 6 forward-compatibility** cleanup, not a production correctness fix. Still worth doing; just not the emergency I originally implied.

### Warning 1 — `BonjourBrowser` switch must be exhaustive

[RemoteDeployCompanion/Services/BonjourBrowser.swift:106-116](RemoteDeployCompanion/Services/BonjourBrowser.swift#L106-L116) switches on `NWTXTRecord.Entry`:

```swift
switch entry {
case .string(let value):
    return value
case .none:
    return nil
@unknown default:
    return nil
}
```

`NWTXTRecord.Entry` has three cases: `.none`, `.empty`, `.string(String)`. The switch handles `.none` and `.string` but misses `.empty`. The `@unknown default` only catches *future* cases the compiler doesn't know about — it does not backfill missing current cases.

**Fix:** add `case .empty: return nil` (or combine it with `.none`).

### Warnings 2–4 — `NSLock.lock` / `NSLock.unlock` unavailable from async contexts

Three warnings:
- `lock` ×1 (line 92 inside `load()`)
- `unlock` ×2 (lines 94 and 97 inside `load()`)

Root cause: `static let cacheLock = NSLock()` at [KeychainStore.swift:57](RemoteDeployCompanion/Services/KeychainStore.swift#L57). `load()` is `async`, and Swift 6 flags any `NSLock.lock()` / `unlock()` call inside an `async` function as "unavailable from asynchronous contexts" regardless of whether the lock is actually held across a suspension point. The same checker rule applies to `NSRecursiveLock`, `pthread_mutex_*`, etc.

The corresponding usage in the macOS host is `OSAllocatedUnfairLock<T>` (see [RemoteDeploy/Services/JSONBuildHistoryStore.swift:19](RemoteDeploy/Services/JSONBuildHistoryStore.swift#L19) and the NIO server's many lock sites). `OSAllocatedUnfairLock` has a `withLock { ... }` API that is async-safe because the lock is structurally scoped — you cannot accidentally hold it across an `await`.

**Fix:** replace `NSLock` + `nonisolated(unsafe) var cached` with `OSAllocatedUnfairLock<(url: String, token: String, serverName: String)?>`. Requires `import os` at the top of the file. All three call sites (`save`, `load`, `clear`, plus the `updateCache` helper) collapse into `cachedLock.withLock { ... }` calls.

Note: `OSAllocatedUnfairLock` is available on iOS 16+. The companion's deployment target is `iOS 17.0` ([project.yml:65](project.yml#L65)), so this is safe.

### Warnings 5 (`@preconcurrency`) and 6 (`LAContext` capture)

These two warnings are **related** — they're both Swift 6's reaction to `LocalAuthentication.framework` not yet being audited for `Sendable`. The problematic code is:

[KeychainStore.swift:162-173](RemoteDeployCompanion/Services/KeychainStore.swift#L162-L173):

```swift
return await withCheckedContinuation { continuation in
    context.evaluatePolicy(.deviceOwnerAuthentication, localizedReason: authReason) { success, error in
        if success {
            continuation.resume(returning: context)
        } else {
            ...
        }
    }
}
```

The outer closure passed to `withCheckedContinuation` is `@Sendable`. Inside it, `context` (type `LAContext`) is captured. `LAContext` is not `Sendable`, so the capture warns. Warning 5 is the compiler's hint that adding `@preconcurrency import LocalAuthentication` will silence related warnings from that module.

**Fix:** change `import LocalAuthentication` to `@preconcurrency import LocalAuthentication` at [KeychainStore.swift:26](RemoteDeployCompanion/Services/KeychainStore.swift#L26). `@preconcurrency` tells the compiler to treat types from the module as if they were `Sendable` for back-compat — the warning downgrades to a note or disappears. This is the same pattern the host app already uses for `NIOSSL` in [RemoteDeploy/Services/NIODeployServer.swift:11](RemoteDeploy/Services/NIODeployServer.swift#L11) (`@preconcurrency import NIOSSL`).

If `@preconcurrency` doesn't fully silence warning 6, the fallback is a small restructure: extract the closure body into a helper function that takes `LAContext` as a parameter, so the Sendable boundary is crossed via a function call rather than a capture.

### Regression risk

- **`BonjourBrowser` switch fix:** trivial. Adding `.empty` case with the same behavior as `.none`. **Low risk**.
- **`KeychainStore` lock migration:** medium surface area but narrow behavior. The cache is the only shared state, `OSAllocatedUnfairLock.withLock { ... }` is a near-drop-in replacement, and all call sites are in one file. Must preserve: (a) cache is populated on successful `load()`, (b) `save()` updates cache after writing keychain, (c) `clear()` nils the cache, (d) concurrent `load()` calls don't race on cache population. **Low-medium risk**.
- **`@preconcurrency` import:** zero-risk downgrade of a compiler warning. No behavioral change.

## Proposed Solution

Three surgical fixes in two files. Smallest-blast-radius changes first, lock migration last.

1. **`BonjourBrowser.swift`** — add `case .empty: return nil` to the `txtString` switch. One line.

2. **`KeychainStore.swift`** — change `import LocalAuthentication` to `@preconcurrency import LocalAuthentication`. One line. Silences warnings 5 and 6 without touching any logic.

3. **`KeychainStore.swift`** — replace `NSLock` + `nonisolated(unsafe) var cached` with `OSAllocatedUnfairLock<(url: String, token: String, serverName: String)?>`. Requires `import os`. Rewrite `save`, `load`, `clear`, `updateCache` to use `withLock { ... }`. Preserves all current semantics; makes Swift 6 happy.

### Why this approach over alternatives

- **`@preconcurrency` over restructuring the closure** — `@preconcurrency` is the Apple-blessed back-compat escape hatch for modules not yet Sendable-audited. Restructuring would introduce real code churn for zero runtime benefit. The host app already uses this pattern for `NIOSSL`.
- **`OSAllocatedUnfairLock` over `actor`** — an actor would require making `load`/`save`/`clear` all `async`, which would cascade into every call site and require `await` everywhere in the companion. A lock is a minimal-touch fix that matches the host app's pattern.
- **`.withLock { ... }` over separate `lock()` / `unlock()` calls** — the scoped form is the one Swift 6 actually allows in async contexts because the lock is structurally guaranteed to be released.

### Tradeoffs

- `OSAllocatedUnfairLock<Optional<Tuple>>` is slightly awkward to read. A small `struct CachedCredentials { let url: String; let token: String; let serverName: String }` wrapping the tuple would be cleaner. I'll add it as part of the migration — it's a natural cleanup since the existing `StoredCredentials` struct (line 46-50) already does this for the on-disk format.
- The `load()` method's cache-hit fast path currently reads the cache, releases the lock, then returns. With `withLock`, the read+return happens inside the closure. Identical semantics, slightly different shape.
