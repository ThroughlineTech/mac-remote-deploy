## Description

Phase 1 of the backend-decoupling plan (docs/backend-decoupling-plan.md). Build
execution and cancellation must no longer depend on any view. Today a remote
(API/web/iOS) build round-trips through `MenuBarView.handleAPIBuildRequest` via
an `.apiBuildRequested` NotificationCenter post, so remote builds only work once
the menu bar popover has been opened. Cancel is a hard-coded no-op
(`NoopBuildCanceler` returns false) even though the engines fully implement
`cancelBuild()`.

Introduce a view-independent `BuildCoordinator` that owns the build lifecycle and
is callable directly by the API, the menu bar button, and any future client. It
sources `serverURL`/`certPath`/`keyPath`/`serverPort`/`serverRunning` from
state/settings itself rather than receiving them from the view.

## Acceptance Criteria
- [x] A `BuildCoordinator` exists and is the single entry point for triggering and
      canceling builds; it resolves the project from `projectStore` by ID and
      sources server/TLS config itself (not from a view).
- [x] The API `buildTrigger` calls the coordinator directly (no NotificationCenter,
      no view), via `DirectBuildTrigger`.
- [x] The API `buildCanceler` actually terminates the running build via
      `buildEngine.cancelBuild()` (no more no-op).
- [x] The menu bar Build & Deploy button calls the coordinator.
- [x] The `.apiBuildRequested` Notification.Name and
      `MenuBarView.handleAPIBuildRequest` are removed; no build logic remains in
      any `View`.
- [x] All existing tests stay green (BuildManagerBroadcastTests,
      XcodeBuildEngineCancelTests, APIRouterFactoryTests, BuildEngineRouterTests,
      integration tests). 390 passed, 0 failed.

## Investigation

See docs/backend-decoupling-plan.md "Phase 1" for the full context. Key findings:

- `BuildManager.triggerBuild(project:serverURL:serverPort:certPath:keyPath:serverRunning:onServerStarted:)`
  is driven directly by `BuildManagerBroadcastTests` and
  `BuildManagerWebSocketIntegrationTests`, and `BuildLogView` /
  `BuildControlsSection` observe `BuildManager`'s `@Published` state. So
  `BuildManager` is kept intact as the orchestration executor + view-observable
  state holder; the coordinator delegates to it.
- Project create/edit/delete in the menu bar and settings always persist to
  `projectStore` (SettingsView, ProjectSetupStep, ProjectsListSection), so
  resolving the build project from `projectStore` by ID is safe for the menu bar
  path and does not regress it.
- The API build adapters are wired only in `AppDelegate.configureAPIRouter`; tests
  use mocks against the unchanged `BuildTriggering` / `BuildCanceling` /
  `BuildStatusProviding` protocols, so swapping the production adapters does not
  touch the unit tests. Integration tests reference `NoopBuildCanceler` directly,
  so those references are converted to a local stub.

## Proposed Solution

Approach: keep `BuildManager` as-is (orchestration + `@Published` view state);
introduce a `BuildCoordinator` as the view-independent build API surface that the
menu bar button and the API adapters both call.

- `BuildConfigProviding` (new protocol): the server/TLS config a build needs.
  `AppState` conforms to it (it already has every field).
- `BuildCoordinator` (`@MainActor` service): `triggerBuild(projectID:configuration:)`
  resolves the project from `projectStore`, sources config from
  `BuildConfigProviding`, delegates to `BuildManager.triggerBuild`.
  `cancelBuild()` (nonisolated) reads the engine's lock-protected status for the
  "was building" result and fires `buildEngine.cancelBuild()`.
  `currentBuildStatus()` (nonisolated) mirrors the existing AppStateBridge read of
  `BuildManager.buildStatus` so the API keeps reporting the persistent
  success/failure status.
- New API adapters: `DirectBuildTrigger` (replaces `NotificationBuildTrigger`),
  `CoordinatorBuildCanceler` (replaces `NoopBuildCanceler` in production),
  `CoordinatorBuildStatusProvider` (replaces `AppStateBridgeBuildStatusProvider`
  in production).
- Delete `.apiBuildRequested` + `MenuBarView.handleAPIBuildRequest` +
  `NotificationBuildTrigger`. Point `BuildControlsSection.performBuild` at the
  coordinator.

Note: `BuildManager` is intentionally kept as the @MainActor ObservableObject the
build-log window and menu bar observe (it owns the persistent build status/log and
the orchestration body). The coordinator is the view-independent entry point that
the API and menu bar both call; it delegates execution to BuildManager. This keeps
`BuildManagerBroadcastTests` / `BuildManagerWebSocketIntegrationTests` green while
removing every view from the build/cancel path.

## Follow-ups (non-blocking, for Phase 2)

- Double project lookup on trigger: `DirectBuildTrigger` validates the project
  exists, then `BuildCoordinator.triggerBuild` resolves it again. Harmless (the
  sync validation gives the API an accurate "Project not found" response before
  the main-actor hop), but collapsible once the store-backed source lands.
- Two build-status sources: `cancelBuild()` reads `buildEngine.status` for its
  "was building" result while the status endpoint reads `BuildManager.buildStatus`
  (engine clears its status when the build returns; BuildManager keeps the
  persistent success/failure). Phase 2's single-source-of-truth work naturally
  collapses these.
