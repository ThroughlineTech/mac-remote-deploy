## Description
macOS uses manual `AppState: ObservableObject` while iOS uses `@MainActor final class ConnectionManager` (cleaner). Convert macOS `AppState` to match the iOS pattern, and extract a `BuildManager` from `RemoteDeployApp` / `MenuBarView` so build state + log streaming is owned by a dedicated type (matching iOS `BuildManager`). Use reactive stream operators to derive `buildLog` from `buildLogStream` instead of manual append callbacks.

## Acceptance Criteria
- [x] `AppState` is `@MainActor final class` with `@Published` (was already `@MainActor class`, adds `final`)
- [x] New `RemoteDeploy/Managers/BuildManager.swift` owns build status, log buffer, cancel — observable via `@EnvironmentObject`
- [x] `MenuBarView.performBuild()` delegates to `BuildManager.triggerBuild()`; no orchestration in the view
- [x] `buildLog` is owned by `BuildManager` and updated from the `buildLogStream`
- [x] `AppStateBridge` updated to read build status from `BuildManager` (feeds the API layer)
- [x] All existing tests still pass (252 baseline)

## Investigation

### Current state
- `AppState` at [MenuBarView.swift:11](RemoteDeploy/Views/MenuBarView.swift#L11) is `@MainActor class AppState: ObservableObject` — already `@MainActor`, needs `final` added
- Build orchestration lives in `MenuBarView.performBuild()` at [MenuBarView.swift:325](RemoteDeploy/Views/MenuBarView.swift#L325) — ~80 lines orchestrating build + server start + notifications + push
- Build state (`buildStatus`, `buildLog`, `lastBuildResult`) is stored as @Published on AppState; read by MenuBarView, BuildLogView, and `AppStateBridge` (which feeds the API via TKT-004's adapters)
- iOS already has a `BuildManager` at [RemoteDeployCompanion/Services/BuildManager.swift](RemoteDeployCompanion/Services/BuildManager.swift) — same shape goal but iOS hits the API while macOS runs xcodebuild directly

### Approach
1. Create `RemoteDeploy/Managers/BuildManager.swift` — `@MainActor final class BuildManager: ObservableObject` with `@Published var buildStatus`, `buildLog`, `lastBuildResult`, `isBuilding`. Has a `triggerBuild(project:baseURL:)` method that pulls the logic from `MenuBarView.performBuild()` verbatim, plus a weak reference to the services it needs (buildEngine, deployServer, notificationManager, pushNotifiers callback).
2. Make `AppState` `final`. Remove `buildStatus`, `buildLog`, `lastBuildResult` from AppState (they migrate to BuildManager).
3. Update `RemoteDeployApp.swift`: instantiate `@StateObject var buildManager = BuildManager()`, wire services on app launch, add `.environmentObject(buildManager)` to scenes.
4. Update `AppStateBridge.buildStatusInfo()`: accept BuildManager in init, read `buildManager.buildStatus` instead of `appState.buildStatus`. This keeps the API layer reading the right source.
5. Update `MenuBarView`: `@EnvironmentObject var buildManager: BuildManager`; replace `performBuild()` call with `buildManager.triggerBuild(...)`; replace AppState reads of `buildStatus`/`buildLog`/`lastBuildResult` with buildManager reads.
6. Update `BuildLogView` similarly if it reads `appState.buildLog`.
7. Update `configureAPIRouter()` to pass buildManager into AppStateBridge.

### Risk: MEDIUM
Touches the whole build flow. 252 tests provide safety net. Main risks: missing a call site that reads from AppState's removed properties, or breaking the API-triggered build path.
