## Description

When a phone is enrolled via the companion's "Pair Another Device" flow (Phone A
mints a token and shows a QR; Phone B scans it), the new device pairs and IS
stored on the Mac -- but the Mac never reflects it: the menu bar's Devices list
keeps showing the old set and no notification fires. It looks like the pairing
"didn't stick."

Root cause: the device record is genuinely stored server-side (`pair()` saves it
before returning), but the menu bar only re-fetches the device list in three
places -- the Devices tab `.task` (on appear), after a pairing sheet dismisses,
and after a revoke. The 3-second status poll (`MenuBarClient.refreshAll`)
refreshes status / projects / installs / build history but NOT devices. So a
device that pairs out-of-band (no Mac-side sheet open, nobody polling) is invisible
until the user reopens the Devices tab. There was also no notification path for a
device pairing.

This is distinct from TKT-069 (which fixed the dedup that *revoked* the first
phone). With TKT-069 in place the device is now correctly stored; TKT-070 makes the
Mac actually surface it.

## Reproduction Steps (bugs only)
1. Pair Phone A. Open the Mac's Settings -> Devices tab (leave it open).
2. On Phone A: Pair Another Device -> show QR. Scan with Phone B.
3. Observe: Phone B pairs, but the Mac's Devices list still shows only Phone A and
   the count does not change. Reopening the tab finally shows Phone B.

## Fix

Menu-bar-side only (no server, shared-package, or protocol changes):

- `MenuBarClient.refreshAll()` (the 3s poll) now also calls `refreshDevices()`, so
  the Devices list stays live regardless of which window is open and out-of-band
  pairings appear within a poll interval.
- `refreshDevices()` tracks the set of known device ids and posts a "New Device
  Paired" desktop notification for any device that appears since the last fetch.
  The first fetch only seeds the baseline, so launching the app never alerts for
  devices that were already paired. Removals (revokes) never notify.

The menu bar app (`com.remotedeploy.app`) already requests notification permission
at launch (`MenuBarAppDelegate`) and already posts notifications elsewhere
(UtilitiesSection), so this reuses an established, authorized path -- no new
cross-process notification wiring.

### Notes / decisions
- Poll-based rather than a new WebSocket "devices" channel: device pairing is a
  rare, discrete event (unlike streaming build logs), and the menu bar already
  polls every 3s for everything else. A <=3s latency is acceptable and far less
  code/risk than a new broadcast channel + subscription.
- A Mac-initiated QR pairing (PairDeviceView) will now also fire the "New Device
  Paired" notification in addition to the sheet's "Device Paired!" confirmation.
  Mildly redundant but informative; left as-is.

## Files Changed
- `RemoteDeploy/Services/MenuBarClient.swift` -- add `knownDeviceIDs`; have
  `refreshDevices()` notify on newly appeared devices; call `refreshDevices()` from
  `refreshAll()`.
