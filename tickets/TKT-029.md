## Description

`.claude/ticket-config.md` declares `scripts/build-release.sh` as the Deploy command, so `/ticket-ship` runs it for every ticket. The script packages the macOS host (`RemoteDeploy` scheme) only — it does not touch `RemoteDeployCompanion`. That means:

- **Companion-only tickets** (TKT-026: Swift 6 cleanup) pay the full notarization cost to produce a DMG that's byte-for-byte equivalent to the previous ship.
- **Docs/tickets-only ships** (e.g. housekeeping commits that only move ticket files to `closed/`) would also pay full notarization if they went through `/ticket-ship`.
- **Apple notarization quota is not unlimited.** Rapid wasted notarizations are a real cost, and every notarization run also re-signs, re-stamples, and re-uploads ~27MB.

In this session, TKT-026 (`68af6db` through `25106b4`) shipped a companion-only change but still ran the full notarization pipeline and produced an identical-to-previous v2.0.1 DMG. `52ae2447-ab9b-4426-a536-ebdeb83857c0` is the wasted notarization ID.

`build-release.sh` already has a `--skip-notarize` flag ([scripts/build-release.sh:6](scripts/build-release.sh#L6)). The missing piece is **`/ticket-ship` knowing when to use it**.

## Reproduction Steps
1. `/ticket-investigate` and ship a companion-only ticket (any change under `RemoteDeployCompanion/`).
2. Observe `/ticket-ship` running the full `scripts/build-release.sh` including notarization.
3. Compare the resulting DMG bit-for-bit to the previous ship's DMG — they're equivalent from the host user's POV.

## Acceptance Criteria
- [ ] `/ticket-ship` **skips the Deploy step entirely** when the ticket's merged diff does not touch any of the paths packaged into the host app (e.g., `RemoteDeploy/`, `Packages/RemoteDeployShared/`, `project.yml` if it affects the host target, `scripts/build-release.sh` itself).
- [ ] `/ticket-ship` still runs the full Deploy when the diff touches host paths.
- [ ] The decision logic is documented in `.claude/ticket-config.md` or a new `.claude/ticket-ship-rules.md` so it's discoverable.
- [ ] A manual override exists (e.g., `--force-deploy` argument or env var) for edge cases where the auto-decision is wrong.
- [ ] When deploy is skipped, `/ticket-ship`'s output explicitly notes "deploy skipped: ticket did not touch host code" so the user isn't confused about the missing notarization log.

## Investigation

### Current state of `/ticket-ship`

The `/ticket-ship` skill reads the Deploy command from `.claude/ticket-config.md` unconditionally. Relevant block:

```
## Commands
- Test: xcodebuild test -project RemoteDeploy.xcodeproj -scheme RemoteDeploy -destination 'platform=macOS'
- Build: xcodebuild build -project RemoteDeploy.xcodeproj -scheme RemoteDeploy -destination 'platform=macOS'
- Deploy: scripts/build-release.sh
```

Phase 4 of the ship skill:
> If `.claude/ticket-config.md` has a Deploy command:
>   1. Push to remote
>   2. Run the Deploy command
>   3. Wait for deploy to complete

No gating on ticket content. Every ship = every deploy.

### What `build-release.sh` actually packages

[scripts/build-release.sh:47](scripts/build-release.sh#L47) builds the `RemoteDeploy` scheme (host only). The companion scheme is never touched. Only these paths affect the resulting binary:

- `RemoteDeploy/` — the host Swift sources
- `Packages/RemoteDeployShared/` — shared package (both targets link it, but a host rebuild is what actually produces the new DMG)
- `project.yml` — if it affects the host target's settings
- `scripts/build-release.sh` — script changes can change the DMG shape
- `RemoteDeploy.xcodeproj` — regenerated by xcodegen, but tracked

**Changes outside those paths cannot change the DMG contents.** Examples of ignorable paths:

- `RemoteDeployCompanion/` — the iOS companion, not packaged
- `RemoteDeployCompanionUITests/` — companion tests, not packaged
- `RemoteDeployTests/` — host unit tests, not included in Release config
- `RemoteDeployIntegrationTests/` — host integration tests, not included in Release config
- `tickets/`, `tickets/closed/` — ticket files
- `CLAUDE.md`, `README.md`, other docs
- `.claude/` — workflow config
- Binary-unchanged modifications to any tracked file (unlikely but possible)

### Two approaches

**Option A: Smart ship skill.** Modify `/ticket-ship` to diff `main..HEAD` at merge time, classify the changed paths, and skip the Deploy step if none are host-relevant. Pros: fully automatic. Cons: requires changing the skill; the allowlist has to stay in sync with `build-release.sh`; edge cases (script changes, XcodeGen regenerations) need careful handling.

**Option B: Deploy command dispatch script.** Replace the Deploy command with a wrapper that inspects the diff, then delegates to `build-release.sh` or no-ops. Pros: skill unchanged; logic lives in the same place as the script it guards. Cons: adds a layer of indirection.

**Option C: Per-ticket opt-out.** A ticket frontmatter field like `deploy: skip` that `/ticket-ship` reads before running the deploy. Pros: explicit, reviewable in the ticket body. Cons: requires the human or investigator to remember to set it.

**Recommendation: Option B.** New script `scripts/ship-deploy.sh` that:

1. Inspects the merge commit's diff (`git show --stat HEAD`) against an allowlist of host-relevant paths.
2. If any host-relevant path changed → exec `scripts/build-release.sh "$@"`.
3. If not → print "skipping notarized build: no host code changes" and exit 0.
4. Supports a `--force` flag to bypass the check.

Then update `.claude/ticket-config.md`'s Deploy line to `scripts/ship-deploy.sh`. Zero changes to the `/ticket-ship` skill. The dispatch script is the single source of truth for "what's a host change."

### Allowlist discovery

The allowlist for Option B's script, as glob patterns (in git pathspec form):

```
RemoteDeploy/**          — host sources
Packages/**              — shared package (every subpath affects linked code)
RemoteDeploy.xcodeproj/**— regenerated project file
project.yml              — XcodeGen spec (can affect host target)
scripts/build-release.sh — the deploy script itself
Info.plist               — version info embedded by the script
```

Everything else is non-deploy-relevant.

### Regression risk

- **Main risk: false negative** — a host change slips through and doesn't get a new DMG because the allowlist is incomplete. Mitigations:
  - Start the allowlist broad (err on the side of building) and narrow over time.
  - `--force` flag for manual override.
  - Ship output explicitly logs the decision and the diff paths examined, so "why didn't this deploy" is debuggable.
- **False positive** (builds when it didn't need to) is the *current* behavior, so this ticket can only make things better on that axis.
- **Version bumps**: if `Info.plist`'s `CFBundleShortVersionString` changes but no host code does, the allowlist catches it via `Info.plist`.

## Proposed Solution

Two commits. One for the dispatch script, one for the config + docs.

### Commit 1 — `scripts/ship-deploy.sh` dispatch wrapper

A short Bash script that:

1. Runs `git show --name-only HEAD` to list the files touched by the merge commit (`/ticket-ship` always makes a merge commit before the deploy step, so HEAD at deploy time is the merge).
2. Greps against the allowlist of host-relevant globs.
3. If match → `exec scripts/build-release.sh "$@"`.
4. If no match → print a skip message and exit 0.
5. `--force` arg short-circuits the check and always runs the build.

Script skeleton:

```bash
#!/bin/bash
# ship-deploy.sh — dispatches to build-release.sh only when the most recent
# commit touched host code. Saves Apple notarization quota on companion-only
# and docs-only ships. TKT-029.
set -euo pipefail

# --force bypasses the gate and always runs the release build.
if [[ "${1:-}" == "--force" ]]; then
    echo "ship-deploy: --force requested, running full release build"
    shift
    exec "$(dirname "$0")/build-release.sh" "$@"
fi

# List files touched by the current HEAD commit (typically the ticket's merge commit).
changed=$(git show --name-only --pretty=format: HEAD | grep -v '^$' || true)

# Allowlist of host-relevant globs. Any match here = we need to deploy.
host_patterns=(
    '^RemoteDeploy/'
    '^Packages/'
    '^RemoteDeploy\.xcodeproj/'
    '^project\.yml$'
    '^scripts/build-release\.sh$'
    'Info\.plist$'
)

needs_deploy=false
for file in $changed; do
    for pat in "${host_patterns[@]}"; do
        if echo "$file" | grep -qE "$pat"; then
            needs_deploy=true
            break 2
        fi
    done
done

if $needs_deploy; then
    echo "ship-deploy: host code changed, running full release build"
    exec "$(dirname "$0")/build-release.sh" "$@"
else
    echo "ship-deploy: no host code changes in HEAD, skipping release build"
    echo "  (pass --force to build anyway)"
    echo "  inspected files:"
    echo "$changed" | sed 's/^/    /'
    exit 0
fi
```

### Commit 2 — `.claude/ticket-config.md` update

- Change `Deploy: scripts/build-release.sh` to `Deploy: scripts/ship-deploy.sh`.
- Add a short comment under the Commands section explaining what `ship-deploy.sh` does and when to `--force` it.

### Why this approach over alternatives

- **Option A (smart skill)** would require modifying `~/.claude/commands/ticket-ship.md`. Out of scope — this is a per-project concern, not a workflow-level one. The dispatch script keeps the fix co-located with the project.
- **Option C (frontmatter opt-out)** puts the burden on the human. Defeats the point of automation.
- **Just adding `--skip-notarize` as a default** would build the app (and sign it) but skip the upload — still wasteful CPU and disk, and produces an unsigned artifact that confuses the user. The gate should skip the whole deploy.

### Tradeoffs

- **Allowlist maintenance**: if someone adds a new top-level folder that affects the host (unlikely — `RemoteDeploy/`, `Packages/`, `project.yml` have been stable for a long time), they'll need to add it to the allowlist. Mitigation: the skip-message prints the inspected files so a developer watching the ship output can notice "huh, I expected a build."
- **Merge commit granularity**: the check inspects HEAD, which is the merge commit after `/ticket-ship` phase 3. If the merge was fast-forward-only (no merge commit), `git show HEAD` will show the last real commit instead, which is still correct. Verified: `/ticket-ship` uses `--no-ff` for merges, so HEAD is always a merge commit carrying the full ticket diff.
