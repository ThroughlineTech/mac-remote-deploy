## Description

Phase 3 of the backend-decoupling plan (docs/backend-decoupling-plan.md). The
inversion: the menu bar UI stops sharing in-process objects (AppState,
BuildManager, BuildCoordinator, stores) and instead talks to its own server over
HTTP/WebSocket -- the same endpoints the web PWA and iOS companion already use.
After this, all three frontends are uniform clients of one API.

Decision (locked by the plan): full HTTP-client inversion, not the lighter
"in-process core via the client protocol" variant.

Depends on Phases 1 (TKT-054, headless BuildCoordinator) and 2 (TKT-055, single
source of truth) -- both merged.

- [x] Settings (Server/Projects/Notifications/Devices) reads/writes go through

## Proposed Solution

### Foundation (done)
- Move `APIClient.swift` + `WebSocketClient.swift` from
  `RemoteDeployCompanion/Services/` into
  `Packages/RemoteDeployShared/Sources/RemoteDeployShared/API/`, made `public`.
  `APIClient.init` gains an optional injectable `URLSession` (defaults to the
  cert-validating session) for testability. The companion's call sites
  (`ConnectionManager`, `BuildManager`, `BuildControlView`) resolve the moved
  types via their existing `import RemoteDeployShared`.
- `RemoteDeployTests/SharedAPIClientTests.swift` round-trips all endpoints via a
  stub `URLProtocol`.

### Local client wiring
- At startup, mint a loopback token: generate a raw token, upsert a
  `PairedDevice` named "Menu bar (local)" (hash stored), keep the raw token in
  memory, and construct `APIClient`/`WebSocketClient` against
  `http://127.0.0.1:8080`. Stored on `ServiceContainer`.

### Section conversion (incremental, each shippable)
status -> projects (list/create/edit/delete) -> build (trigger/cancel + WS log)
-> installs -> settings. Each section repointed from `@EnvironmentObject`
in-process objects to the client.

## Gotchas
- Menu bar now depends on the server being up. When the server is not started
  (cert not configured) or :8080 fails to bind, show a connecting/disabled state.
- Loopback over plain HTTP avoids TLS-trust friction; the menu bar uses a
  pre-seeded token and never calls `/api/v1/pair` (pairing over HTTP is blocked).
- Main-actor reentrancy: the client is async; the views are `@MainActor`.

## Decision (resolved): always-on :8080 API listener
The Settings window does first-time server config (cert paths -> start server)
AND ongoing tweaks. A pure HTTP client cannot do the first if the API only comes
up with HTTPS. Resolved by splitting the server: `startAPIListener` binds the
plain-HTTP API on :8080 at launch regardless of certs; `startHTTPS` binds the
TLS install server when certs exist. Setting cert paths via the client triggers
`.settingsDidChange`, and AppDelegate's start-only `reconcileHTTPS()` brings
HTTPS up. This aligns with the eventual Phase 6 process split. The "Restart
Server" control (stop HTTPS only, keep the API listener) handles port/cert
changes while running.

## Review Notes
- **[accepted] Poll cost.** `MenuBarClient` polls 4 endpoints (status, projects,
  installs, build history) every 3s for the life of the app, popover open or not.
  Only `status` is needed to keep the menu bar icon fresh when the popover is
  closed. Fine for a local loopback tool; throttle later if it matters. Cheap
  win: when the popover is closed, refresh `status` only (and refresh the rest on
  popover-open), or widen the interval. External changes (e.g. a project created
  by the web client) therefore appear on the next poll (<= 3s), not instantly --
  acceptable.
- **[accepted] WS re-render coupling.** `MenuBarClient` forwards the
  WebSocket's `objectWillChange`, so every live `buildlog` line re-renders any
  view observing the client. SwiftUI only renders visible views, so the cost is
  only paid while the popover or Build Log window is open, but it over-couples the
  popover to per-line log traffic. Cheap win: drop the forwarding and have
  `BuildLogView` observe `menuBarClient.webSocket` directly (`@ObservedObject`),
  so only the log window churns per line; derive `isBuilding` from the polled
  status instead of the live frame.
