## Description
On every cold launch of the iOS companion app, the user is prompted for device passcode / Face ID **three times in a row** before the app becomes usable. The count appears to track the number of configured projects, suggesting the app is reading the Keychain-stored bearer token once per project during startup. The Keychain items were created with `.userPresence` access control (per the security audit hardening), so each read triggers a biometric / passcode prompt.

Expected: exactly one biometric prompt per cold launch (or zero if a recent unlock is still valid within the system's grace window). The token should be read once into memory at launch, then reused for the lifetime of the process.

## Reproduction Steps
1. Force-quit the iOS companion app
2. Relaunch it
3. Observe: passcode / Face ID prompt fires
4. Pass it → observe: prompt fires again
5. Pass it → observe: prompt fires a third time
6. Pass it → app finally becomes usable

## Acceptance Criteria
- [x] Exactly **one** biometric prompt per cold launch (or zero within the OS unlock grace window)
- [x] The three keychain fields (url, token, serverName) are collapsed into a single keychain item so reading them only triggers one `.userPresence` challenge
- [x] Migration path: existing users with the old 3-item layout still unlock on first launch (accepting 3 prompts once), then future launches only prompt once
- [x] No regression: pairing still works, unpairing clears the blob AND any legacy items, launching after unpair does NOT prompt (nothing to read)
- [x] iOS build clean
- [x] All 270 existing tests still pass

## Investigation

### Root cause — CONFIRMED

The user's "1 per project" theory was close but wrong. **It's actually 1 prompt per stored keychain item**, and there are three of them.

[KeychainStore.swift:28-35](RemoteDeployCompanion/Services/KeychainStore.swift#L28-L35):
```swift
static func load() -> (url: String, token: String, serverName: String)? {
    guard let url = get(key: urlKey),      // prompt #1
          let token = get(key: tokenKey),  // prompt #2
          let name = get(key: nameKey) else { // prompt #3
        return nil
    }
    return (url, token, name)
}
```

Each of the three items (`server_url`, `server_token`, `server_name`) is stored as a separate `kSecClassGenericPassword` entry guarded by `.userPresence` access control ([set at lines 63-68](RemoteDeployCompanion/Services/KeychainStore.swift#L63-L68)). `SecItemCopyMatching` on a `.userPresence`-guarded item presents a fresh LocalAuthentication challenge on every call, within the same process, without any grace window. Three reads → three prompts. Every. Single. Launch.

The coincidence: the user has 3 projects, and the number of prompts also happens to be 3. It's not correlated — if they had 5 projects it'd still be 3 prompts, and if they had 1 project it'd still be 3 prompts. The count is tied to the 3 stored keychain items, not to project count.

### Why it regressed

`.userPresence` was added in commit `bad0ee3` (the v2.0.1 security hardening pass, in response to the security audit). Before that, the items used only `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` with no access control flag — which doesn't prompt at all. The audit recommended `.userPresence` to protect against physical theft, but the implementation landed without first collapsing the three items into one blob, so the prompt count multiplied.

### Call site inventory

Single reader:
- `ConnectionManager.restoreConnection()` at [line 82](RemoteDeployCompanion/Services/ConnectionManager.swift#L82) — `KeychainStore.load()` — this is the one invoked at cold launch

Writers (save):
- `ConnectionManager.pair()` [line 128](RemoteDeployCompanion/Services/ConnectionManager.swift#L128)
- `ConnectionManager.connect()` [line 153](RemoteDeployCompanion/Services/ConnectionManager.swift#L153)

Clear calls:
- `ConnectionManager.disconnect()` [line 166](RemoteDeployCompanion/Services/ConnectionManager.swift#L166)
- `ConnectionManager.init()` [line 47](RemoteDeployCompanion/Services/ConnectionManager.swift#L47) — `--reset-pairing` XCUITest launch arg

Only `load()` needs to change behavior. The public API stays identical.

### Regression risk — **LOW**
Self-contained change to `KeychainStore.swift`. Public surface (`save`/`load`/`clear`) unchanged. Risks:
1. **Migration.** Existing users on the old 3-item layout need their data read one last time (3 prompts, unavoidable) and rewritten as the new blob. Subsequent launches drop to 1 prompt.
2. **Decode errors.** If the JSON blob is ever corrupt, `load()` returns nil and the user has to re-pair. Same failure mode as the current "missing one of three items" case.
3. **Sendable.** KeychainStore is `static`-only, stateless. Safe.

## Proposed Solution

### Approach: single Codable blob with one-time migration from legacy 3-item format

Store all three fields as a single `Codable` struct (`StoredCredentials`) JSON-encoded into one keychain item keyed `"credentials"`. `load()` reads that one item → one prompt per launch.

Migration runs inside `load()` on first call after upgrade:
1. Try to read the new `"credentials"` blob via a new `getData(key:)`. If found, decode and return (1 prompt, steady state).
2. Else, fall through to legacy `loadLegacy()` which reads the three old items (3 prompts — one-time cost).
3. If legacy succeeded, immediately `save()` the new blob and `clearLegacy()` the three old items. Next launch will hit the new blob path and only prompt once.
4. If both paths fail, return nil (user must re-pair).

### Why not just remove .userPresence?
That was the security audit's ask — protect the bearer token against physical theft with a user-presence challenge. Removing it reverts a deliberate hardening change.

### Why not use a long-lived LAContext on the query?
An alternative: construct an `LAContext`, call `evaluatePolicy(.deviceOwnerAuthentication)` once explicitly, then pass the context into subsequent `SecItemCopyMatching` queries via `kSecUseAuthenticationContext`. Inside the context's lifetime (~10s by default), further reads don't prompt. That would fix the symptom with the current 3-item layout intact.

Rejected because: (a) more code, (b) tied to LAContext lifetime which adds timing fragility, (c) the blob approach is simpler and also wins on "single Keychain round-trip" efficiency, (d) blob is easier to extend if we ever add a fourth field.

### Tradeoffs
- **First launch after upgrade still shows 3 prompts.** Unavoidable — we have to read the legacy items once to migrate them. Documented; user sees it only once.
- **JSON blob is opaque in Keychain Access.** Can't inspect individual fields via the macOS Keychain Access app. Fine — this is an iOS app and users don't inspect it.
- **Adding a new field requires a Codable migration.** StoredCredentials versioning would be handled by standard `decodeIfPresent` patterns if/when that comes up. Minor concern.

## Implementation Plan

- [x] 1. Add a private `StoredCredentials: Codable` struct to `KeychainStore` with `url: String`, `token: String`, `serverName: String`.
- [x] 2. Add `private static let credentialsKey = "credentials"` alongside the existing `urlKey`/`tokenKey`/`nameKey` (the old keys stay defined because legacy `clearLegacy` needs them).
- [x] 3. Rewrite `save(url:token:serverName:)` to JSON-encode a `StoredCredentials` and call a new private `setData(key:data:)` helper that applies the same access-control rules (`.userPresence` on device, `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` in simulator).
- [x] 4. Rewrite `load()`:
  - First try `getData(key: credentialsKey)` → if non-nil, decode `StoredCredentials` and return the tuple (1 prompt).
  - Else call a new private `loadLegacy()` that runs the existing three-item read path (3 prompts).
  - If legacy succeeded, call `save(...)` with the loaded values (writes the new blob, 0 extra prompts since save doesn't read), then `clearLegacy()` to delete the 3 old items. Return the tuple.
  - If both fail, return nil.
- [x] 5. New private `getData(key:) -> Data?` helper mirroring `get(key:)` but returning the raw Data instead of decoding as UTF-8.
- [x] 6. New private `setData(key:data:)` helper — same as the existing `set(key:value:)` but takes Data directly.
- [x] 7. New private `clearLegacy()` helper — calls `delete(key:)` on all three old keys (no prompts; SecItemDelete doesn't require auth).
- [x] 8. Rewrite `clear()` to delete the new `credentialsKey` item AND call `clearLegacy()` so users who unpair after migration don't leave stale items behind.
- [x] 9. Keep `#if targetEnvironment(simulator)` branches intact for the new path.
- [x] 10. Build iOS companion scheme — clean.
- [x] 11. Run macOS test suite — all 270 existing tests still pass.
- [x] 12. Manual smoke test checklist (in TKT-020 review): cold-launch companion on device, expect 1 prompt (migration case: 3 prompts on first launch only, then 1); unpair → relaunch → expect no prompt; pair fresh → relaunch → expect 1 prompt.
