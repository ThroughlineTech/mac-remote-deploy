## Description
Small quality-of-life improvements for the iOS companion: haptic feedback on successful pair, last-build timestamp shown in `ProjectListView`, build duration shown in `BuildControlView`, pull-to-refresh on project list.

## Acceptance Criteria
- [x] `UINotificationFeedbackGenerator.notificationOccurred(.success)` fires inside `ConnectionManager.pair()` after the server returns `paired: true`
- [x] `ProjectListView` row shows the most recent build timestamp for each project (best-effort — fetched alongside the project list)
- [x] `BuildControlView` shows elapsed seconds (e.g. "Building... 12s") while a build is in progress, ticking once per second
- [x] Pull-to-refresh on `ProjectListView` — **already exists** at [ProjectListView.swift:65](RemoteDeployCompanion/Views/Projects/ProjectListView.swift#L65), verify still present
- [x] iOS companion build clean
- [x] No regression in existing flows

## Investigation

### State of each AC item

1. **Haptic on successful pair.** [ConnectionManager.pair()](RemoteDeployCompanion/Services/ConnectionManager.swift) currently logs success and saves credentials but no haptic. Add `UINotificationFeedbackGenerator().notificationOccurred(.success)` right after `KeychainStore.save(...)`.

2. **Last build timestamp per project.** Two paths:
   - **Path A — Add a new field to `ProjectConfig`.** Requires server-side bookkeeping: every successful build updates the project's `lastBuiltAt`. Cleanest but requires server changes.
   - **Path B — Use `GET /api/v1/builds` (build history) joined to projects in the iOS view.** No server changes; the build history is already exposed. **But TKT-008 (build history persistence) hasn't shipped yet** — `EmptyBuildHistoryProvider` returns `[]`. So this path returns nothing useful until TKT-008 lands.
   - **Path C — Use `GET /api/v1/installs` (install history)** as a proxy for "last activity." Already populated, indexed by project name (slug). Each install record has a timestamp. Not exactly "last build" but "last successful download" which is close.
   - **Decision:** Path C. Functional today, no dependency on TKT-008. Show "Last installed: 3h ago" instead of "Last built." If/when TKT-008 ships, swap to Path B with one line change.

3. **Build duration timer.** Track when `buildManager.isBuilding` flips true, store start time, render `Text("Building... \(elapsed)s")` updated by a `Timer.publish` every second. When `isBuilding` flips false, freeze the value or hide.

4. **Pull-to-refresh on ProjectListView.** Already implemented via `.refreshable { await loadProjects() }`. AC met.

### Files affected
- `RemoteDeployCompanion/Services/ConnectionManager.swift` — add haptic
- `RemoteDeployCompanion/Views/Projects/ProjectListView.swift` — fetch installs alongside projects, render "Last installed: X" per row
- `RemoteDeployCompanion/Views/Build/BuildControlView.swift` — add elapsed timer state + display
- `RemoteDeployCompanion/Services/APIClient.swift` — verify `getInstalls()` exists (it does, used by `InstallHistoryView`)

### Dependency on TKT-013
TKT-013 lands first (in the same batch). It adds proper loading/error/empty states to the same three views. TKT-017 is **purely additive on top of those state branches** — adds new UI elements inside the loaded/data branches without touching the new state machinery. No conflict.

View-only additive changes + one method addition. No data layer changes. iOS build is the only verification.

## Proposed Solution

### Approach
Three small additive changes, one per AC item.

**Haptic:** one line in ConnectionManager.
```swift
UINotificationFeedbackGenerator().notificationOccurred(.success)
```

**Last installed timestamp:** ProjectListView fetches both projects AND installs in `loadProjects()`. Maintain a `[ProjectID: Date]` map keyed by project name → most recent install timestamp. Render under the bundle ID line:
```swift
if let last = lastInstallByProject[project.id] {
    Text("Last installed \(last, format: .relative(presentation: .named))")
        .font(.caption2).foregroundColor(.secondary)
}
```

**Build duration timer:** Use `TimelineView(.periodic(from: startDate, by: 1.0))` or a `Timer.publish` ObservableObject. Cleanest is `TimelineView`:
```swift
if connectionManager.buildManager.isBuilding, let startedAt {
    TimelineView(.periodic(from: startedAt, by: 1.0)) { context in
        let elapsed = Int(context.date.timeIntervalSince(startedAt))
        Text("Building... \(elapsed)s")
    }
}
```
The view tracks `@State var buildStartedAt: Date?` set when `isBuilding` becomes true via `.onChange(of: isBuilding)`.

### Tradeoffs
- **"Last installed" ≠ "Last built."** Per Path C decision above, we use install records as a proxy. The label says "Last installed" not "Last built" to be honest with the user.
- **No build history dependency.** When TKT-008 ships, we can switch to actual build history with a one-line change.
- **Timer when view is hidden.** TimelineView pauses when offscreen, so no battery cost.
