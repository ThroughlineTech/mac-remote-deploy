## Description
Failures are silent: server start failure prints to console, settings save errors are swallowed, route handler errors never reach the UI. Add `@Published var error: AppError?` to `AppState` and `ConnectionManager`, surface via `.alert()` modifiers, and add retry buttons for transient operations.

## Acceptance Criteria
- [x] `AppState.error: RemoteDeployError?` published field
- [x] MenuBarView surfaces `appState.error` via `.alert()` with the error's `errorDescription` + `failureReason`
- [x] Boundary error sites in `RemoteDeployApp.swift` (server start, load projects, load/save settings, Tailscale check) set `appState.error` in addition to logging
- [x] iOS `ConnectionManager.error: RemoteDeployError?` published field already partly wired — extend for consistency
- [x] All 252 existing tests pass

## Investigation
After TKT-006 removed `buildStatus`/`buildLog`/`lastBuildResult` from AppState, the spot for `error` is obvious. TKT-010 already shipped `RemoteDeployError(wrapping:)`. Now just plumb it through.

Minimum viable wiring: add `@Published var error: RemoteDeployError?` to AppState, set it at the boundary sites (which already construct `RemoteDeployError` via wrapping for logging), add a single `.alert(isPresented:)` to `MenuBarView` that pops when `error != nil`. Provide a `Dismiss` button that clears.
