## Description

In the iOS companion app's **Build** tab, after tapping **Build & Deploy** the UI enters the "Building…" state with a ticking elapsed-time counter under the button. When the build finishes on the Mac, the Prowl push notification arrives on the phone (so the server-side build definitely completed and the notification path works), but the Build tab still shows "Building… 63s" (or whatever the final time is) and the counter keeps ticking indefinitely.

The companion never transitions out of the in-progress state, so:
- The Build & Deploy button stays disabled
- The spinner/ticker never stops
- The build log area shows only "Command line invocation:" (the initial line) and never receives the rest of the log
- There's no success / failure indication

Expected behavior: when the server reports the build as finished (success or failure), the companion Build tab should update to the finished state — stop the timer, re-enable the button, and show the final log + status.

Likely causes to investigate:
- WebSocket connection for live build logs dropping or never receiving the terminal "build finished" event
- Companion polling/subscription loop not being notified when build state transitions on the server
- Build status response from `/api/v1/projects/:id/build` not being consumed after kickoff

See the attached screenshot: the timer reads "Building… 63s" with "Starting build…" still shown as the sub-status even though the build is long done.

## Reproduction Steps (bugs only)

1. Open the companion app, go to the **Build** tab
2. Select the `rejog-ios` project
3. Tap **Build & Deploy**
4. Wait for the Prowl push notification indicating the build finished
5. Observe: the Build tab still shows "Building… {N}s" with the timer still incrementing, button still disabled, log area empty beyond the first line

## Acceptance Criteria

- [ ] When a build completes on the Mac (success or failure), the companion Build tab transitions out of the "Building…" state
- [ ] The elapsed-time ticker stops at the final duration
- [ ] The Build & Deploy button becomes enabled again
- [ ] The log area shows the build log (or at minimum a success/failure indicator)
- [ ] This works whether or not the phone was actively in the foreground while the build ran

## Investigation

### Findings

1. **The Build tab view is not directly observing the objects that actually publish build/log updates.**
	- `BuildControlView` only observes `ConnectionManager` (`@EnvironmentObject`) and then reads nested state from `connectionManager.buildManager`.
	- `BuildLogStreamView` does the same for `connectionManager.webSocketClient`.
	- `ConnectionManager` holds `buildManager` and `webSocketClient` as plain `let` properties, so their `objectWillChange` does not propagate through `ConnectionManager`.
	- Result: after entering "Building...", the view can remain stuck on stale nested state while the underlying manager objects continue changing.

2. **Host-side completion signaling appears intact.**
	- macOS `BuildManager` broadcasts terminal build states (`.success` / `.failure`) through `setBuildStatus(...)` and the WebSocket broadcaster.
	- WebSocket server path (`NIODeployServer+BuildEventBroadcasting`) emits `buildstatus` messages with the expected `BuildStatusInfo` JSON shape for terminal states.
	- Existing tests already cover host-side buildstatus payload shape (`NIODeployServerBuildEventBroadcastingTests`, `BuildManagerWebSocketIntegrationTests`).

3. **Protocol mismatch is unlikely; UI observation wiring is the probable break in the chain.**
	- Companion `WebSocketClient` decodes `buildstatus` into `BuildStatusInfo` correctly.
	- Companion `BuildManager.observeWebSocketStatus(...)` handles terminal states by setting `isBuilding = false`.
	- The missing link is the Build tab's direct observation of those publisher objects.

### Root Cause

The Build tab (`BuildControlView` + `BuildLogStreamView`) is bound to nested `ObservableObject` state through `ConnectionManager`, rather than observing `BuildManager` and `WebSocketClient` directly. This can leave the Build tab on stale "building" UI even after the underlying state transitions to success/failure.

### Regression Risk

**Medium** — fix is localized to Build tab wiring, but touches SwiftUI observation paths that drive button enabled/disabled state, elapsed timer behavior, and log rendering.


## Proposed Solution

Inject the shared `BuildManager` and `WebSocketClient` as environment objects into the Build tab only, and make `BuildControlView`/`BuildLogStreamView` observe those directly.

Why this approach:
- Keeps scope disciplined to the Build tab path (this ticket).
- Avoids global observation plumbing changes in `ConnectionManager` that would also alter Project detail behavior (TKT-044 scope).
- Preserves existing build orchestration, polling, and websocket protocol code.

Tradeoffs:
- Slightly more environment-object wiring at the Build tab call site.
- Does not address similar nested-observation patterns outside the Build tab (intentionally out of scope for this ticket).


## Implementation Plan

## Delegation Review
**Verdict:** approved

**Summary:** Investigation is thorough: root cause correctly traced to nested `ObservableObject` observation — `ConnectionManager` holds `buildManager` and `webSocketClient` as plain `let` properties, so their `objectWillChange` doesn't propagate through `ConnectionManager`'s own publisher. The fix injects both as `@EnvironmentObject` at the Build tab entry point in `MainTabView` and switches `BuildControlView` + `BuildLogStreamView` to observe them directly. Minimal, scoped, correct.

**Issues found:**
1. **Nit (test coverage direction):** the added integration test `testFailureBuildStatusBroadcastReachesSubscribedClient` verifies that the **host** correctly broadcasts terminal failure status frames over WebSocket — it does not directly verify that the **companion's** `BuildControlView` unsticks its UI when those frames arrive. The existing (pre-fix) test suite already covered the success-case host broadcast; this adds the failure case, which is useful but orthogonal to the actual bug being fixed on the companion side. A companion-side UI test (analogous to what TKT-044 added in `ProjectDetailReactiveStatusUITests`) would more directly regression-guard the Build tab fix. Not blocking — the fix itself is correct by inspection, and SwiftUI observation bugs are notoriously hard to reach from XCTest without exercising the real view hierarchy.
2. **Nit:** TKT-044 fixes the analogous bug on the Projects detail view using `@ObservedObject` with an explicit init, while this ticket uses `@EnvironmentObject` at the tab entry point. Both are correct SwiftUI patterns; the inconsistency is minor. The root cause (`ConnectionManager.buildManager` being a non-`@Published` `let`) remains untouched — both fixes work around it at the leaf views. A follow-up could address that centrally, out of scope here.

**Missing:** nothing from the acceptance criteria. All five AC bullets are addressed by the observation-wiring change — when the underlying `BuildManager.isBuilding` flips to `false` via the WebSocket terminal status handler (which is confirmed working end-to-end in the existing `BuildManagerWebSocketIntegrationTests`), the Build tab will now re-render in-place.
