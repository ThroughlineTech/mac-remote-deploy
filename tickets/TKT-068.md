## Description

The server's WebSocket `buildstatus` channel is a pure fire-and-forget event
stream: `WebSocketManager.broadcast` only writes to clients currently connected
and subscribed, and `subscribe()` does not send the current status to a newly
subscribing client. So any client that connects -- or reconnects, via the
backoff loop -- *after* the last status transition was broadcast never learns the
current build state. If it reconnects after a build finished, it stays pinned to
whatever it last saw (typically `"building"`), with no way to recover except a
later transition that may never come.

This is the root cause behind TKT-067's stuck menu bar spinner: a status channel
should be a *retained value* (last-write replayed to new subscribers), not just
an event stream. Build *log* lines are correctly append-only events; build
*status* is a single current value and should be stateful.

Fixing it at the source benefits every WebSocket client (the iOS companion too,
though its REST-poll fallback already self-heals), and removes the only way the
menu bar's live frame can go stale.

## Reproduction Steps (bugs only)
1. Connect a WebSocket client and subscribe to `buildstatus`.
2. Run a build to completion (terminal `success` broadcast goes out).
3. Drop and re-establish the WebSocket (or connect a fresh client) AFTER the
   build ended.
4. Observe: the (re)subscribed client receives no status frame at all and has no
   knowledge that a build ran or finished -- it must wait for the next transition.

## Acceptance Criteria
- [x] On subscribing to `buildstatus`, a client is immediately sent the current
      build status (same wire payload a transition broadcast carries).
- [x] `buildlog` is unchanged -- it stays an append-only event stream (no replay;
      clients backfill the log from build history).
- [x] Replay reads the same in-process status source as the REST poll, off the
      event loop (`BuildStatus` is a Sendable value load), so REST and WS agree.
- [x] No replay when the status source is unset or returns nil.
- [x] Existing WebSocket tests still pass; new tests cover replay-on-subscribe.

## Implementation Plan
- [x] Add a `buildStatusReplay: (@Sendable () -> String?)?` provider to
      `WebSocketManager`, set once at config time via `setBuildStatusReplay`.
- [x] In `subscribe(_:to:)`, when the channel is `buildstatus`, send the provider's
      current payload (if any) to just that client, outside the lock.
- [x] Factor the encode/send out of `broadcast` into shared `encodeMessage` /
      `send(text:to:)` helpers so replay and broadcast produce identical frames.
- [x] Wire the provider in `ServerLifecycle.configureAPIRouter`, encoding the
      current `BuildStatusInfo` from the same `buildStatusProvider` the REST
      status path uses.
- [x] Tests: replay on `buildstatus` subscribe; no replay on `buildlog`; no-op
      when provider unset or returns nil.

## Files Changed
- `RemoteDeploy/API/WebSocket/WebSocketHandler.swift` -- stateful `buildstatus`
  channel: `buildStatusReplay` provider, replay in `subscribe`, shared
  `encodeMessage`/`send` helpers.
- `RemoteDeployServer/ServerLifecycle.swift` -- wire the replay provider next to
  the WS authenticator, reading the existing `BuildManagerBuildStatusProvider`.
- `RemoteDeployTests/WebSocketManagerTests.swift` -- 4 replay-on-subscribe tests.
