## Description

RemoteDeploy currently only handles native Swift/Xcode projects (`xcodebuild archive` → sign → deploy). The user has a React Native (Expo) project at `/Users/plympton/src/_project-throughline/cards-for-bongs` that they want to build and deploy through RemoteDeploy as well.

### About the target project

- **Location:** `/Users/plympton/src/_project-throughline/cards-for-bongs/cfb/`
- **Structure:** npm monorepo with workspaces — `backend/` (API) and `app/` (Expo RN)
- **Framework:** Expo SDK 52, React Navigation 7
- **iOS bundle ID:** `com.cfb.cardgame`
- **Has native `ios/` directory** — either from `expo prebuild` or `expo run:ios` (native module support)
- **Build scripts:** `expo start`, `expo run:ios`
- **Test scripts:** `jest` (unit), `jest --testPathPattern='live-game'` (integration)

### What this ticket needs to figure out

1. **How does a React Native / Expo project get built for device deployment?** The build chain is different from native Xcode: likely `expo prebuild` → `xcodebuild archive` on the generated `ios/` workspace, or `eas build --local`, or `expo run:ios --device`. The investigation needs to determine which approach fits RemoteDeploy's architecture (which runs `xcodebuild` directly on the host Mac).

2. **What needs to change in RemoteDeploy's project model?** Currently the companion app stores project configs with `projectPath`, `scheme`, `destination` — all Xcode-native concepts. An Expo project needs a different set of config fields (npm workspace root, Expo app directory, maybe a prebuild step, possibly different signing).

3. **What changes to `XcodeBuildEngine` (or a new engine)?** The build pipeline needs to handle the Expo prebuild step, then the Xcode archive, or delegate to EAS. The engine needs to know it's dealing with an Expo project and run the right commands.

4. **What about the companion's log streaming?** The build log will include npm/Expo output in addition to xcodebuild output. The log stream should still work, but the companion might need to handle different log formats.

5. **Can it use the same deploy-to-device mechanism?** Once you have an `.ipa` or `.app`, the install path should be the same. The question is how to get there from an Expo project.

## Acceptance Criteria

- [ ] A user can add the cards-for-bongs Expo project in RemoteDeploy's project settings (host app) — with a path pointing at the project, and whatever RN-specific config fields are needed.
- [ ] Tapping Build & Deploy in the companion triggers a successful build of the Expo project (whatever the right build chain turns out to be — prebuild + xcodebuild, EAS local, etc.).
- [ ] Build log streams to the companion in real-time, showing both the Expo/npm phases and the Xcode phases.
- [ ] The built app gets installed on the connected device via the existing deploy mechanism.
- [ ] Existing native Xcode project builds are not affected — the feature is additive.
- [ ] The investigation documents which build approach was chosen and why.

## Investigation

### Current Build Pipeline (end-to-end)

**ProjectConfig model** (`Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/ProjectConfig.swift`):
- Shared type across host + companion. Fields: `id`, `name`, `projectPath`, `projectFile` (optional `.xcodeproj` name), `workspaceFile` (optional `.xcworkspace` name), `scheme`, `bundleID`, `teamID`, `provisioningProfile`, `buildConfiguration`, `urlSlug`, `exportMethod`, `platform`.
- Host app re-exports it at `RemoteDeploy/Models/ProjectConfig.swift` (line 1: `@_exported import struct RemoteDeployShared.ProjectConfig`).
- The `init(name:projectPath:)` constructor auto-generates `urlSlug` from the name, defaults to `Release` config, `development` export method, `iOS` platform.
- Backward-compatible `Decodable` init handles missing `platform` field.

**BuildEngineProtocol** (`RemoteDeploy/Protocols/BuildEngineProtocol.swift`):
- Protocol: `build(project:) async throws -> String` (returns IPA path), `cancelBuild() async`, `buildLogStream: AsyncStream<String>`, `status: BuildStatus`, `detectSchemes(at:) async throws -> [String]`.
- `BuildStatus` enum: `.idle`, `.building(progress:)`, `.success(ipaPath:)`, `.failure(error:)`.

**XcodeBuildEngine** (`RemoteDeploy/Services/XcodeBuildEngine.swift`):
- `build(project:)` pipeline (line 79-206):
  1. Creates `/tmp/RemoteDeploy/<name>.xcarchive` and `<name>_export` directories.
  2. Resolves workspace/project: checks `project.workspaceFile`, then `project.projectFile`, then suffix detection on `projectPath`, then directory scanning for `.xcworkspace`/`.xcodeproj`.
  3. Runs `xcrun xcodebuild archive` with scheme, destination (platform-based), archivePath, configuration, DEVELOPMENT_TEAM.
  4. Generates `ExportOptions.plist` with export method + team ID.
  5. Runs `xcrun xcodebuild -exportArchive` to produce IPA.
  6. Copies IPA to `~/Library/Application Support/RemoteDeploy/serve/<urlSlug>/app.ipa`.
- Process management: uses `Process` with stdout/stderr pipes, streams lines via `AsyncStream<String>.Continuation`, ring buffer for stderr tail on failure.
- `detectSchemes(at:)` runs `xcodebuild -list` and parses the "Schemes:" section.
- `cancelBuild()` terminates the running process with idempotent flag.

**BuildManager** (`RemoteDeploy/Managers/BuildManager.swift`):
- `@MainActor`, `ObservableObject`. Published state: `buildStatus`, `buildLog`, `lastBuildResult`.
- Dependencies injected via `configure(...)`: `BuildEngineProtocol`, `DeployServerProtocol`, `NotificationManager`, `IPAImporter`, `BuildHistoryStoring`, `BuildEventBroadcasting`.
- `triggerBuild(project:...)`: subscribes to `buildEngine.buildLogStream` in a Task, calls `buildEngine.build(project:)`, on success registers project with deploy server and starts server if needed, sends local + push notifications. On failure, records result and notifies.
- The broadcaster fans build status + log lines to WebSocket clients (companion app).

### Deploy Mechanism

**OTA (Over-The-Air) via HTTPS — NOT direct device install.** There is no `devicectl`, `ios-deploy`, or USB-based install.

The pipeline:
1. `XcodeBuildEngine.build()` produces an IPA at `~/Library/Application Support/RemoteDeploy/serve/<slug>/app.ipa`.
2. `NIODeployServer` (SwiftNIO HTTPS server) serves three routes per project slug:
   - `/<slug>/` — HTML install page with `itms-services://` link
   - `/<slug>/manifest.plist` — Apple OTA manifest XML (bundle ID, version, IPA URL)
   - `/<slug>/app.ipa` — the IPA binary
3. User opens `https://<tailscale-hostname>:8443/<slug>/` in Safari on iOS, taps Install.
4. iOS downloads the manifest, then the IPA, and installs it.

**Key implication:** The Expo build just needs to produce a signed IPA in the same serve directory. The entire OTA deploy pipeline downstream is project-type-agnostic.

### Build Strategy Evaluation

**Option 1: `expo prebuild --clean` → `pod install` → `xcodebuild archive`** (RECOMMENDED)
- Fits RemoteDeploy perfectly: the xcodebuild phase is identical to native projects.
- `expo prebuild --clean` regenerates `ios/` from `app.json` + native modules. Takes ~10-20s.
- `pod install` resolves CocoaPods. Takes ~30-60s.
- Then standard `xcodebuild archive` + export, which XcodeBuildEngine already does.
- Real-time log streaming works — each phase runs as a `Process` with piped stdout/stderr.
- Signing works the same way (DEVELOPMENT_TEAM, export method).
- Requires: `node`, `npm`, `npx` (for expo CLI), `cocoapods` (gem or bundler).
- The native `ios/` directory already exists with `CFB.xcworkspace` — we can even skip prebuild for incremental builds and only run it when `app.json` or native dependencies change.

**Option 2: `eas build --local --platform ios`**
- Not configured (no `eas.json`), would need setup.
- Produces a `.tar.gz` with a `.app` inside — NOT an IPA. Would need extra packaging.
- Opaque build process — harder to stream logs, control signing, or integrate with existing engine.
- Adds `eas-cli` as a dependency. Overkill for local builds.
- Rejected: poor fit for RemoteDeploy's architecture.

**Option 3: `expo run:ios --device`**
- Combines prebuild + xcodebuild + device install in one command.
- Takes over the install step — conflicts with RemoteDeploy's OTA deploy.
- Can't easily extract just the IPA.
- Rejected: conflicts with deploy pipeline.

**Option 4: `npx react-native run-ios`**
- Similar to Option 3 but bypasses Expo's prebuild.
- Less control, same conflicts.
- Rejected.

**Chosen: Option 1** — `expo prebuild` → `pod install` → `xcodebuild archive` → export IPA. This is the only option that cleanly decomposes into phases that map 1:1 to RemoteDeploy's existing architecture.

### Architecture Decision: New Engine vs. Extended Engine

**Create a new `ExpoBuildEngine` that implements `BuildEngineProtocol`.**

Reasons:
1. The pre-xcodebuild phases (npm install, expo prebuild, pod install) are substantial — they're not a small wrapper around xcodebuild.
2. `XcodeBuildEngine` is clean and focused on xcodebuild concerns. Mixing npm/expo logic in would violate SRP.
3. `BuildEngineProtocol` is the right abstraction — `BuildManager` already consumes it via the protocol and doesn't care which engine runs.
4. `ExpoBuildEngine` can internally delegate to `XcodeBuildEngine` for the archive+export phase, or duplicate the xcodebuild logic (since it's straightforward).
5. The protocol already supports everything needed: `build(project:)`, `buildLogStream`, `status`, `cancelBuild()`.

**ProjectConfig changes:**
- Add a `projectType` enum field: `.xcode` (default) or `.expo`.
- Add optional `expoAppDirectory` field: relative path from `projectPath` to the Expo app dir (e.g., `"app"` for monorepo, or `nil` for single-app repos).
- The existing fields (`scheme`, `bundleID`, `teamID`, `buildConfiguration`, `exportMethod`) are reused — they apply to the xcodebuild phase.
- `workspaceFile` gets auto-populated during prebuild detection (e.g., `"ios/CFB.xcworkspace"`).
- Backward-compatible: existing configs decode with `projectType` defaulting to `.xcode`.

**ServiceContainer / BuildManager changes:**
- `ServiceContainer` needs to hold both engines (or a factory that picks the right one).
- `BuildManager.triggerBuild()` selects the engine based on `project.projectType`.
- Alternatively, a `BuildEngineRouter` that wraps both engines and dispatches based on project type.

## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Strategy:** `expo prebuild --clean` → `pod install` → `xcodebuild archive` → export IPA. This decomposes the Expo build into phases where the final phase (xcodebuild) is identical to native projects, reusing the entire OTA deploy pipeline unchanged.

**Architecture:** Create a new `ExpoBuildEngine` conforming to `BuildEngineProtocol`. It runs the pre-xcodebuild phases (npm install, expo prebuild, pod install) as streamed `Process` invocations, then delegates to the existing xcodebuild archive+export logic. `BuildManager` selects the engine based on a new `projectType` field on `ProjectConfig`.

**Why this over alternatives:** EAS Build is not configured and produces `.app` not `.ipa`. `expo run:ios` takes over device install, conflicting with OTA deploy. The prebuild→xcodebuild approach is the only one that maps cleanly to RemoteDeploy's existing architecture and gives full control over signing, log streaming, and the IPA output.
