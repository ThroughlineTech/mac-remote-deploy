## Description

TKT-024 Commit 6 (commit `396fdb8`, shipped 2026-04-08) landed the full WebSocket transport for the `/api/v1/ws` endpoint: `NIOWebSocketServerUpgrader` in the HTTPS pipeline, bearer-token auth via the shared `AuthMiddleware`, `WebSocketChannelHandler` registered on upgrade, `WebSocketManager` managing subscriptions, and the companion's `WebSocketClient` with reconnect-with-backoff. An integration test (`RemoteDeployIntegrationTests/WebSocketUpgradeTests.swift`) exercises unauth→401, auth→101, and end-to-end subscribe+broadcast flows.

**What's missing:** no part of the server currently *calls* `WebSocketManager.broadcast(...)` with real events. The transport is plumbed end-to-end but has no emitters. The companion opens a WebSocket, subscribes to `"buildlog"` and `"buildstatus"`, and then sits there receiving nothing. The AC for TKT-024 / TKT-011 explicitly allowed this carve-out:

> If any sub-part is genuinely out of scope for this cleanup, it must be explicitly carved out into a follow-up ticket with justification, not silently dropped.

This is that follow-up.

## Scope — emitters to wire up

At least three things need to publish to the WebSocket:

1. **Build log stream** — `BuildManager.triggerBuild` already consumes `buildEngine.buildLogStream` and appends each line to `buildManager.buildLog`. Each appended line should also fan out as a `WSMessage(type: "buildlog", payload: <line>)` broadcast.

2. **Build status changes** — `BuildManager` publishes `@Published var buildStatus: BuildStatus` with cases `.idle`, `.building(progress:)`, `.success(ipaPath:)`, `.failure(error:)`. Every transition should fan out as a `WSMessage(type: "buildstatus", payload: <JSON-encoded BuildStatusInfo>)` broadcast. `BuildStatusInfo` and the conversion logic already exist in `AppStateBridge.buildStatusInfo()`.

3. **Install events** (optional) — `NIODeployServer.onIPADownload` already fires when the companion downloads an IPA. A `"install"` channel broadcast would let paired devices see install events in real time. Lower priority — could be deferred again to a smaller ticket if it bloats this one.

## Architecture question (for investigation)

Where does the broadcast call live?

- **Option A:** `BuildManager` directly calls `webSocketManager.broadcast(...)`. Requires giving `BuildManager` a reference to `WebSocketManager` (e.g., injected via `configure(...)`). Pros: direct, minimal indirection. Cons: couples `BuildManager` to the WS transport.
- **Option B:** `BuildManager` publishes to a `BuildEventPublishing` protocol, and `NIODeployServer` subscribes and forwards to its `webSocketManager`. Pros: decoupled, testable. Cons: more moving parts.
- **Option C:** Use Combine — `BuildManager`'s `@Published buildStatus` is already observable. A subscriber in `AppDelegate` or `ServiceContainer` could bridge it to `webSocketManager.broadcast`.

Recommend Option A for simplicity given this is a small internal project — but the investigation phase should confirm this is the right call.

The build log stream is the trickier piece: currently `BuildManager.triggerBuild` has a `Task { for await line in logStream { buildLog += line + "\n" } }`. That task is where the broadcast call would naturally go.

## Reproduction Steps (for the AC verification)
1. Launch the Mac host app and pair the companion app.
2. Trigger a build from the companion (or directly from the Mac menu bar).
3. Watch the companion's "live build log" UI (or the Xcode console of the companion).

Currently: no log lines appear via WebSocket.
After this ticket: each xcodebuild log line should appear on the companion as it's emitted, and build status transitions (building → success/failure) should update live.

## Acceptance Criteria
- [ ] `BuildManager.triggerBuild` fans out each buildLog line as a `WSMessage(type: "buildlog", payload: <line>)` broadcast via `WebSocketManager.broadcast`.
- [ ] Every `BuildStatus` transition in `BuildManager` fans out as a `WSMessage(type: "buildstatus", payload: <JSON BuildStatusInfo>)` broadcast.
- [ ] The companion app receives and displays live build log lines in real time during a build (verified on device).
- [ ] The companion app's `WebSocketClient.isConnected` flips to `true` after pairing and stays true through a full build cycle.
- [ ] Unit test coverage for the broadcast fan-out: a test double for `WebSocketManager` (or a real one with an in-process subscriber) captures the expected sequence of `buildlog` + `buildstatus` messages for a mocked build.
- [ ] Integration test extends `RemoteDeployIntegrationTests/WebSocketUpgradeTests.swift` (or a sibling file) to drive a real end-to-end: authed WS client subscribed to `buildlog`, trigger a mocked build via `BuildManager`, assert broadcast frames arrive in the correct order.
- [ ] `xcodebuild test` passes.
- [ ] No regression in the WebSocket transport itself — existing `WebSocketUpgradeTests` still pass unchanged.
- [ ] Install events (`onIPADownload` → `"install"` channel broadcast) either wired up OR explicitly carved out into another follow-up ticket with justification.
