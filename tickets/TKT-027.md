## Description

TKT-024 Commit 6 (commit `396fdb8`, shipped 2026-04-08) landed the full WebSocket transport for the `/api/v1/ws` endpoint: `NIOWebSocketServerUpgrader` in the HTTPS pipeline, bearer-token auth via the shared `AuthMiddleware`, `WebSocketChannelHandler` registered on upgrade, `WebSocketManager` managing subscriptions, and the companion's `WebSocketClient` with reconnect-with-backoff. An integration test (`RemoteDeployIntegrationTests/WebSocketUpgradeTests.swift`) exercises unauth→401, auth→101, and end-to-end subscribe+broadcast flows.

**What's missing:** no part of the server currently *calls* `WebSocketManager.broadcast(...)` with real events. The transport is plumbed end-to-end but has no emitters. The companion opens a WebSocket, subscribes to `"buildlog"` and `"buildstatus"`, and then sits there receiving nothing. The AC for TKT-024 / TKT-011 explicitly allowed this carve-out:

> If any sub-part is genuinely out of scope for this cleanup, it must be explicitly carved out into a follow-up ticket with justification, not silently dropped.

This is that follow-up.

## Scope — emitters to wire up

At least three things need to publish to the WebSocket:

1. **Build log stream** — `BuildManager.triggerBuild` already consumes `buildEngine.buildLogStream` and appends each line to `buildManager.buildLog`. Each appended line should also fan out as a `WSMessage(type: "buildlog", payload: <line>)` broadcast.

2. **Build status changes** — `BuildManager` publishes `@Published var buildStatus: BuildStatus` with cases `.idle`, `.building(progress:)`, `.success(ipaPath:)`, `.failure(error:)`. Every transition should fan out as a `WSMessage(type: "buildstatus", payload: <JSON-encoded BuildStatusInfo>)` broadcast. `BuildStatusInfo` and the conversion logic already exist in `AppStateBridge.buildStatusInfo()`.

3. **Install events** (optional) — `NIODeployServer.onIPADownload` already fires when the companion downloads an IPA. A `"install"` channel broadcast would let paired devices see install events in real time. Lower priority — could be deferred again to a smaller ticket if it bloats this one.

## Architecture question (for investigation)

Where does the broadcast call live?

- **Option A:** `BuildManager` directly calls `webSocketManager.broadcast(...)`. Requires giving `BuildManager` a reference to `WebSocketManager` (e.g., injected via `configure(...)`). Pros: direct, minimal indirection. Cons: couples `BuildManager` to the WS transport.
- **Option B:** `BuildManager` publishes to a `BuildEventPublishing` protocol, and `NIODeployServer` subscribes and forwards to its `webSocketManager`. Pros: decoupled, testable. Cons: more moving parts.
- **Option C:** Use Combine — `BuildManager`'s `@Published buildStatus` is already observable. A subscriber in `AppDelegate` or `ServiceContainer` could bridge it to `webSocketManager.broadcast`.

Recommend Option A for simplicity given this is a small internal project — but the investigation phase should confirm this is the right call.

The build log stream is the trickier piece: currently `BuildManager.triggerBuild` has a `Task { for await line in logStream { buildLog += line + "\n" } }`. That task is where the broadcast call would naturally go.

## Reproduction Steps (for the AC verification)
1. Launch the Mac host app and pair the companion app.
2. Trigger a build from the companion (or directly from the Mac menu bar).
3. Watch the companion's "live build log" UI (or the Xcode console of the companion).

Currently: no log lines appear via WebSocket.
After this ticket: each xcodebuild log line should appear on the companion as it's emitted, and build status transitions (building → success/failure) should update live.

## Investigation

### Where broadcasts need to fire

Three event streams on the host should fan out to `WebSocketManager.broadcast(...)`:

**1. Build log lines** — [RemoteDeploy/Managers/BuildManager.swift:108-113](RemoteDeploy/Managers/BuildManager.swift#L108-L113):

```swift
let logTask = Task { @MainActor in
    for await line in logStream {
        buildLog += line + "\n"
    }
}
```

One broadcast per line, at the `for await line in logStream` boundary. Channel: `"buildlog"`. Payload: the raw line.

**2. Build status transitions** — `BuildManager.buildStatus` is assigned in five places ([BuildManager.swift:102, 124, 172, 206, 211](RemoteDeploy/Managers/BuildManager.swift#L102-L211)):
- Line 102: `.building(progress: "Starting build...")`
- Line 124: `.success(ipaPath: ipaPath)` (after successful xcodebuild)
- Line 172: `.failure(error: ...)` (catch block)
- Line 206: `.success(ipaPath: ipaPath)` (from `markImportSucceeded`)
- Line 211: `.failure(error: ...)` (from `markImportFailed`)

All five should fan out as `"buildstatus"` broadcasts. Payload: JSON-encoded `BuildStatusInfo`. The existing `BuildStatus → BuildStatusInfo` conversion lives in [AppStateBridge.buildStatusInfo()](RemoteDeploy/RemoteDeployApp.swift#L245-L258) — but it takes a snapshot and only returns one `BuildStatusInfo`. For broadcasts we want to convert a specific `BuildStatus` value, not read current state. A small free function or static method on `BuildManager` is cleaner.

**3. Install events** — [RemoteDeploy/AppDelegate.swift:328](RemoteDeploy/AppDelegate.swift#L328):

```swift
server.onIPADownload = { [weak appState, serviceContainer] slug, ip, ua in
    Task {
        await serviceContainer.installTracker.recordInstall(...)
        ...
    }
}
```

Inside that closure, after `recordInstall`, fan out a `"install"` channel broadcast with the slug + source IP.

### How `WebSocketManager` reaches `BuildManager`

`WebSocketManager` is owned by `NIODeployServer` ([NIODeployServer.swift:53](RemoteDeploy/Services/NIODeployServer.swift#L53)) as a `let` property and exposed publicly. `BuildManager` already has `deployServer: (any DeployServerProtocol)?` injected via `configure(...)` at [AppDelegate.swift:153-159](RemoteDeploy/AppDelegate.swift#L153-L159).

But **`DeployServerProtocol` does not expose `webSocketManager`**. Only the concrete `NIODeployServer` does. Options:

- **Option A:** extend `DeployServerProtocol` to expose a `broadcast(type:payload:)` method. Cleanest — keeps `BuildManager` behind the protocol.
- **Option B:** inject `WebSocketManager` separately into `BuildManager.configure(...)`. Minimal protocol churn but grows `configure`'s parameter list.
- **Option C:** define a new `BuildEventPublishing` protocol with just `publishBuildLog(_:)`, `publishBuildStatus(_:)`, `publishInstall(...)` methods. `NIODeployServer` (or a thin adapter) conforms. `BuildManager` takes one of these. Cleanest separation of concerns, most indirection.

**Recommendation: Option C**, with a twist — define the protocol as `BuildEventBroadcasting` and have `NIODeployServer` conform directly. It's one extra file (the protocol) and one small `extension NIODeployServer: BuildEventBroadcasting` block. Gives us a testable seam (a mock broadcaster in `BuildManager`'s tests) without polluting `DeployServerProtocol` with WS concerns.

### Existing test coverage

- **`BuildManager`** — no dedicated tests. `XcodeBuildEngineCancelTests` exercises a tiny slice of the cancellation path through a mocked engine, but nothing covers `triggerBuild` end-to-end.
- **`WebSocketManager`** — solid unit tests in `RemoteDeployTests/WebSocketManagerTests.swift` (7 tests covering add/remove/subscribe/broadcast).
- **`WebSocketUpgradeTests`** (integration) — covers the wire protocol end-to-end. Extending it or adding a sibling file is the right place for the end-to-end broadcast test.
- **`APIRouterFactoryTests`** and handler tests — unchanged; won't touch them.

### Regression risk

- **BuildManager.triggerBuild** changes are the risky part. The log-tap is the hottest inner loop during a build (hundreds to thousands of lines per second for verbose xcodebuild output). If broadcast blocks the main actor or buffers excessively, it could slow builds or leak memory. Mitigations: `WebSocketManager.broadcast` writes to the channel with `promise: nil` (fire-and-forget) and iterates subscribers under a brief lock, so it's O(clients) and non-blocking. If no WS clients are connected, `broadcast` is nearly free.
- **Build status transitions** are one-shot per transition; fan-out overhead is negligible.
- **Install broadcasts** fire once per IPA download — also negligible.
- **Existing WebSocketUpgradeTests still pass** is the key regression signal. Also need to verify no deadlock if broadcast is called from inside a `@MainActor` context while the channel's event loop is on a different NIO thread — but `broadcast` already handles cross-thread writes correctly (NIO channels buffer + dispatch internally).

### One tricky thing: `BuildManager` is `@MainActor`, broadcasting may need cross-thread handling

`BuildManager` is `@MainActor final class`, so all its methods run on the main actor. `WebSocketManager.broadcast` takes a text string and writes to NIO channels — which are thread-safe, NIO dispatches writes to the appropriate event loop internally. So calling broadcast from the main actor is safe.

However, the `BuildEventBroadcasting` protocol should be `Sendable` so `BuildManager` can hold a reference across the main-actor boundary. Simple enough: `protocol BuildEventBroadcasting: Sendable { ... }` and `NIODeployServer` is already `@unchecked Sendable`.


## Proposed Solution

Four commits on one branch, lowest-risk first. Same ordering discipline as TKT-024: tests early, production wiring last.

### Architectural decision: `BuildEventBroadcasting` protocol

New file: `RemoteDeploy/Protocols/BuildEventBroadcasting.swift`:

```swift
import Foundation
import RemoteDeployShared

/// Fans out build-related events (log lines, status transitions, install
/// downloads) over a transport. The production implementation is
/// `NIODeployServer`, which broadcasts via its `webSocketManager`. Tests
/// pass a mock that records calls. TKT-027.
protocol BuildEventBroadcasting: Sendable {
    func broadcastBuildLog(_ line: String)
    func broadcastBuildStatus(_ status: BuildStatus)
    func broadcastInstall(slug: String, sourceIP: String)
}
```

`NIODeployServer` gets an extension that maps each method to a `webSocketManager.broadcast(type:payload:)` call with JSON-encoded payloads.

`BuildManager` gains an optional `buildEventBroadcaster: (any BuildEventBroadcasting)?` property set via `configure(...)`. All three fan-out sites become one-line calls.

### Why protocol over direct injection

- Keeps `BuildManager` decoupled from `WebSocketManager` — easier to test, easier to reason about.
- Matches the existing injection pattern (`BuildEngineProtocol`, `DeployServerProtocol`, etc.).
- The protocol makes the integration test trivial: provide a real `NIODeployServer` in one test, a mock in the unit tests.

### Tradeoffs

- One extra file and two extra indirection points. For a project this small, that's negligible. The testing clarity wins.
- `broadcastBuildStatus(_:)` takes a `BuildStatus` (the Swift enum) rather than a pre-encoded `BuildStatusInfo`. The conversion happens inside the `NIODeployServer` extension, keeping `BuildManager` ignorant of the JSON-serialization format. Symmetrical with how `AppStateBridge.buildStatusInfo()` works.
