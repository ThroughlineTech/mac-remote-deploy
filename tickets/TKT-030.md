## Description

The menu bar popover shows "Tailscale Disconnected" (red dot) even though Tailscale is clearly connected (visible in the Tailscale menu bar icon, and the server is actually running on the HTTPS port with a valid Tailscale cert). The server URL falls back to `http://192.168.29.100:8080` (local WiFi) instead of the Tailscale HTTPS URL.

Screenshot confirms: "Server Running" (green) + "Tailscale Disconnected" (red) simultaneously — and a local-WiFi HTTP URL where there should be a Tailscale HTTPS URL.

### Root causes (from code inspection)

**1. Initial check is delayed by 150ms and may miss the window.**

[AppDelegate.swift:178](RemoteDeploy/AppDelegate.swift#L178) calls `checkTailscaleStatus()` inside `performStartup()`, which itself is dispatched via `DispatchQueue.main.asyncAfter(deadline: .now() + 0.15)` (TKT-021 fix). If the Tailscale CLI is slow to respond (it spawns a subprocess via `tailscale status --self --json`), the initial check can complete AFTER the user has already opened the popover — showing the default `tailscaleConnected = false` state.

**2. Polling interval is 30 seconds — too long.**

[AppDelegate.swift:250-257](RemoteDeploy/AppDelegate.swift#L250-L257) polls every 30 seconds:

```swift
private func startStatusPolling() {
    Task { @MainActor [weak self] in
        while !Task.isCancelled {
            try? await Task.sleep(for: .seconds(30))
            await self?.checkTailscaleStatus()
        }
    }
}
```

If the initial check fails or returns false due to a race, the status stays "Disconnected" for up to 30 seconds before it self-corrects. The user has already looked at the popover and seen stale data.

**3. No check on popover open.**

The MenuBarExtra popover's content (`MenuBarView`) is re-rendered on each open, but it reads `appState.tailscaleConnected` from the cached `@Published` value — it does NOT trigger a fresh `checkTailscaleStatus()` call. The status can be stale for the entire 30-second poll window and the user has no way to force a refresh.

**4. `isConnected()` may be getting a stale CLI response.**

[CLITailscaleProvider.swift:59-72](RemoteDeploy/Services/CLITailscaleProvider.swift#L59-L72) runs `tailscale status --self --json` and parses `Self.Online`. If Tailscale's local daemon hasn't updated its state yet (e.g., the daemon is still negotiating keys), `Online` may be `false` even though the network is reachable. Also: if the CLI binary path is wrong (only checks `/usr/local/bin/tailscale` and `/Applications/Tailscale.app/Contents/MacOS/Tailscale`), the catch block silently returns `false`.

**5. Binary search paths may not cover all installations.**

Tailscale on macOS can also be installed via Homebrew to `/opt/homebrew/bin/tailscale` (Apple Silicon Macs) or via the Mac App Store (which puts the CLI at a different app bundle path). If the binary isn't at the two hardcoded paths, `isConnected()` silently returns `false` — showing "Tailscale Disconnected" even though the Tailscale app is running and connected.

## Reproduction Steps
1. Ensure Tailscale is connected and visible in the macOS menu bar.
2. Launch RemoteDeploy (or wait for the 30s poll to fire).
3. Open the RemoteDeploy menu bar popover within the first 30 seconds of launch.
4. Observe: "Tailscale Disconnected" (red dot) even though Tailscale is connected.

## Acceptance Criteria
- [ ] Tailscale status is checked immediately when the menu bar popover is opened (not only on a timer).
- [ ] Polling interval reduced from 30s to something more responsive (e.g. 10s).
- [ ] Binary search paths include `/opt/homebrew/bin/tailscale` for Homebrew on Apple Silicon.
- [ ] The status indicator in the popover updates within 1-2 seconds of opening the popover, not after a 30-second wait.
- [ ] The `send failed: Invalid argument` noise we fixed in TKT-021 does NOT regress (i.e., don't move Tailscale checking into the body-evaluation path that triggers layout recursion).
- [ ] `xcodebuild test` passes.

## Investigation

### Root cause analysis

There are four distinct issues contributing to the "Tailscale Disconnected" false negative:

**1. No on-demand check when popover opens (primary cause)**

`MenuBarView` (line 14, `RemoteDeploy/Views/MenuBarView.swift`) reads `appState.tailscaleConnected` from the cached `@Published` value. It has no `.onAppear` or `.task` modifier that triggers a fresh Tailscale status check. The popover simply renders whatever value was last cached.

`ServerStatusSection` (line 26-29, `RemoteDeploy/Views/MenuBar/ServerStatusSection.swift`) displays the red/green dot based on this cached boolean:
```swift
Circle()
    .fill(appState.tailscaleConnected ? Color.green : Color.red)
```

There is no mechanism anywhere in the view hierarchy to trigger `checkTailscaleStatus()` when the popover opens.

**2. 30-second polling interval is too long**

`AppDelegate.startStatusPolling()` (line 250-257, `RemoteDeploy/AppDelegate.swift`) polls every 30 seconds. Combined with issue #1, if the initial check at startup returns `false` (race condition, slow CLI, etc.), the user sees stale data for up to 30 seconds. The sleep-first loop pattern (`sleep` then `check`) means the first poll-driven update is 30 seconds after startup, not immediately.

**3. Initial check delayed by 150ms and may miss the window**

`performStartup()` (line 133, `RemoteDeploy/AppDelegate.swift`) is dispatched with a 150ms delay (TKT-021 fix). The `checkTailscaleStatus()` call at line 178 runs the CLI subprocess `tailscale status --self --json`, which can take hundreds of milliseconds. If the user opens the popover quickly after launch, they see the default `tailscaleConnected = false`.

**4. Missing Homebrew binary search path**

`CLITailscaleProvider.binarySearchPaths` (line 9-12, `RemoteDeploy/Services/CLITailscaleProvider.swift`) only contains:
- `/usr/local/bin/tailscale` (Intel Homebrew or manual install)
- `/Applications/Tailscale.app/Contents/MacOS/Tailscale` (App Store / direct download)

Missing: `/opt/homebrew/bin/tailscale` (Apple Silicon Homebrew). On Apple Silicon Macs with a Homebrew Tailscale install, `tailscaleBinaryPath()` throws `TailscaleError.binaryNotFound`, the `catch` at line 69 returns `false`, and the status always shows "Disconnected".

### Call chain

1. `RemoteDeployApp.body` -> `appDelegate.register()` -> `performStartup()` (150ms delay)
2. `performStartup()` -> `checkTailscaleStatus()` -> `serviceContainer.tailscaleProvider.isConnected()`
3. `CLITailscaleProvider.isConnected()` -> `tailscaleBinaryPath()` -> `runCommand(["tailscale", "status", "--self", "--json"])` -> parse `Self.Online`
4. Result stored in `appState.tailscaleConnected` (`@Published`)
5. `startStatusPolling()` repeats step 2-4 every 30 seconds (sleep-first)
6. `ServerStatusSection.body` reads `appState.tailscaleConnected` on render

### Regression risk: LOW

- Adding an `.onAppear` check to MenuBarView is safe -- it posts to `checkTailscaleStatus()` which is already `@MainActor` and idempotent.
- The TKT-021 `_NSDetectedLayoutRecursion` issue was caused by `@Published` mutations during the MenuBarExtra's initial body evaluation. An `.onAppear` modifier fires AFTER layout, so it won't re-trigger layout recursion.
- Reducing the polling interval from 30s to 10s is a trivial constant change with no architectural impact.
- Adding a binary search path is additive and the existing `for path in binarySearchPaths` loop handles it naturally.

### Key constraint (TKT-021)

The acceptance criteria explicitly warns: "don't move Tailscale checking into the body-evaluation path that triggers layout recursion." The solution must use `.onAppear` or `.task` (post-layout), NOT inline code in `body`. The `.task` modifier is slightly preferred because it's natively async, but `.onAppear` with a `Task {}` block works identically.

## Proposed Solution

A four-part fix addressing each root cause:

1. **Add `.task` to `MenuBarView`** that calls a new `refreshTailscaleStatus` notification, which `AppDelegate` observes and runs `checkTailscaleStatus()`. This ensures every popover open triggers a fresh check. Using a NotificationCenter-based approach keeps the view decoupled from the delegate (consistent with existing patterns like `.startServerRequested`).

2. **Reduce polling interval** from 30s to 10s. This is a reasonable balance between freshness and CLI subprocess overhead (each check spawns one `tailscale status --self --json` process).

3. **Add `/opt/homebrew/bin/tailscale`** to `binarySearchPaths` as the first entry (Apple Silicon is the dominant Mac platform).

4. **No changes to the 150ms startup delay** -- that's the TKT-021 fix and the other changes make it unnecessary to touch. The `.task` on popover open will catch any startup race.

**Why this approach over alternatives:**
- Alternative: expose `checkTailscaleStatus()` as a public method and call it directly from the view. Rejected because it couples the view to `AppDelegate` and breaks the existing `@EnvironmentObject`-only dependency pattern.
- Alternative: use a Combine publisher on `appState` that auto-refreshes. Over-engineered for a simple "check on open" requirement.
- The NotificationCenter pattern is already used for `.startServerRequested`, `.saveSettingsRequested`, `.openSetupAssistant`, and `.apiBuildRequested`, so this is consistent.

## Implementation Plan

- [ ] **Step 1: Add Homebrew binary path** -- In `RemoteDeploy/Services/CLITailscaleProvider.swift`, add `"/opt/homebrew/bin/tailscale"` as the first entry in `binarySearchPaths` (line 9-12). This is the highest-priority path on modern Macs.

- [ ] **Step 2: Reduce polling interval** -- In `RemoteDeploy/AppDelegate.swift`, change `Task.sleep(for: .seconds(30))` to `Task.sleep(for: .seconds(10))` in `startStatusPolling()` (line 253).

- [ ] **Step 3: Add notification name** -- In `RemoteDeploy/RemoteDeployApp.swift`, add a new `static let refreshTailscaleStatus = Notification.Name("RemoteDeploy.refreshTailscaleStatus")` to the existing `Notification.Name` extension (around line 267-277).

- [ ] **Step 4: Observe the notification in AppDelegate** -- In `RemoteDeploy/AppDelegate.swift`, inside `applicationDidFinishLaunching(_:)` (around line 97-127), add a `NotificationCenter.default.addObserver` for `.refreshTailscaleStatus` with a handler method. The handler method should call `checkTailscaleStatus()` in a `Task { @MainActor }` block, similar to the existing `handleStartServerRequested` pattern.

- [ ] **Step 5: Post the notification from MenuBarView on appear** -- In `RemoteDeploy/Views/MenuBarView.swift`, add a `.task` modifier to the outermost `VStack` (after the existing `.onReceive` modifiers, around line 36). The task body should post `NotificationCenter.default.post(name: .refreshTailscaleStatus, object: nil)`. Using `.task` (not `.onAppear`) so it's natively async and cancels on disappear.

- [ ] **Step 6: Update MockTailscaleProvider if needed** -- In `RemoteDeployTests/Mocks/MockTailscaleProvider.swift`, no changes needed -- the mock already implements `isConnected()` correctly. Verify `isConnectedCallCount` increments as expected in any new tests.

- [ ] **Step 7: Run tests** -- `xcodebuild test -project RemoteDeploy.xcodeproj -scheme RemoteDeploy -destination 'platform=macOS'` to verify no regressions, especially around TKT-019 (server starts at launch) and TKT-021 (no layout recursion).
