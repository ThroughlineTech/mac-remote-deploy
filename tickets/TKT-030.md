## Description

The menu bar popover shows "Tailscale Disconnected" (red dot) even though Tailscale is clearly connected (visible in the Tailscale menu bar icon, and the server is actually running on the HTTPS port with a valid Tailscale cert). The server URL falls back to `http://192.168.29.100:8080` (local WiFi) instead of the Tailscale HTTPS URL.

Screenshot confirms: "Server Running" (green) + "Tailscale Disconnected" (red) simultaneously — and a local-WiFi HTTP URL where there should be a Tailscale HTTPS URL.

### Root causes (from code inspection)

**1. Initial check is delayed by 150ms and may miss the window.**

[AppDelegate.swift:178](RemoteDeploy/AppDelegate.swift#L178) calls `checkTailscaleStatus()` inside `performStartup()`, which itself is dispatched via `DispatchQueue.main.asyncAfter(deadline: .now() + 0.15)` (TKT-021 fix). If the Tailscale CLI is slow to respond (it spawns a subprocess via `tailscale status --self --json`), the initial check can complete AFTER the user has already opened the popover — showing the default `tailscaleConnected = false` state.

**2. Polling interval is 30 seconds — too long.**

[AppDelegate.swift:250-257](RemoteDeploy/AppDelegate.swift#L250-L257) polls every 30 seconds:

```swift
private func startStatusPolling() {
    Task { @MainActor [weak self] in
        while !Task.isCancelled {
            try? await Task.sleep(for: .seconds(30))
            await self?.checkTailscaleStatus()
        }
    }
}
```

If the initial check fails or returns false due to a race, the status stays "Disconnected" for up to 30 seconds before it self-corrects. The user has already looked at the popover and seen stale data.

**3. No check on popover open.**

The MenuBarExtra popover's content (`MenuBarView`) is re-rendered on each open, but it reads `appState.tailscaleConnected` from the cached `@Published` value — it does NOT trigger a fresh `checkTailscaleStatus()` call. The status can be stale for the entire 30-second poll window and the user has no way to force a refresh.

**4. `isConnected()` may be getting a stale CLI response.**

[CLITailscaleProvider.swift:59-72](RemoteDeploy/Services/CLITailscaleProvider.swift#L59-L72) runs `tailscale status --self --json` and parses `Self.Online`. If Tailscale's local daemon hasn't updated its state yet (e.g., the daemon is still negotiating keys), `Online` may be `false` even though the network is reachable. Also: if the CLI binary path is wrong (only checks `/usr/local/bin/tailscale` and `/Applications/Tailscale.app/Contents/MacOS/Tailscale`), the catch block silently returns `false`.

**5. Binary search paths may not cover all installations.**

Tailscale on macOS can also be installed via Homebrew to `/opt/homebrew/bin/tailscale` (Apple Silicon Macs) or via the Mac App Store (which puts the CLI at a different app bundle path). If the binary isn't at the two hardcoded paths, `isConnected()` silently returns `false` — showing "Tailscale Disconnected" even though the Tailscale app is running and connected.

## Reproduction Steps
1. Ensure Tailscale is connected and visible in the macOS menu bar.
2. Launch RemoteDeploy (or wait for the 30s poll to fire).
3. Open the RemoteDeploy menu bar popover within the first 30 seconds of launch.
4. Observe: "Tailscale Disconnected" (red dot) even though Tailscale is connected.

## Acceptance Criteria
- [ ] Tailscale status is checked immediately when the menu bar popover is opened (not only on a timer).
- [ ] Polling interval reduced from 30s to something more responsive (e.g. 10s).
- [ ] Binary search paths include `/opt/homebrew/bin/tailscale` for Homebrew on Apple Silicon.
- [ ] The status indicator in the popover updates within 1-2 seconds of opening the popover, not after a 30-second wait.
- [ ] The `send failed: Invalid argument` noise we fixed in TKT-021 does NOT regress (i.e., don't move Tailscale checking into the body-evaluation path that triggers layout recursion).
- [ ] `xcodebuild test` passes.
