### Bugs / smells discovered during investigation

These are **out of scope** for TKT-003 (write tests, don't fix code) but worth flagging in the test for future tickets:

1. **`cleanupExpiredTokens()` is dead code.** [PairingRouteHandler.swift:36](RemoteDeploy/API/Routes/PairingRouteHandler.swift#L36) defines a 10-minute expiration cleanup, but `grep -rn cleanupExpiredTokens RemoteDeploy/` shows it's **never called**. The `pair()` method doesn't check token age — it only checks for membership in the `pendingTokens` dict. So the documented "10-minute TTL" from [SECURITY_AUDIT.md](docs/SECURITY_AUDIT.md) is **currently unenforced**. TKT-003 will:
   - Add a test of `cleanupExpiredTokens()` itself (using a test-only init that accepts a clock or by directly testing the side effect after a delay)
   - Add a test that documents the *current* behavior (`pair()` accepts a token regardless of how long ago it was registered, until cleanup runs)
   - File a follow-up note in the test comment pointing at TKT-018 to wire cleanup into the pair flow
2. **`InstallsRouteHandler` uses a semaphore to bridge async → sync.** [InstallsRouteHandler.swift:22-31](RemoteDeploy/API/Routes/InstallsRouteHandler.swift#L22-L31) creates a `DispatchSemaphore` and blocks the calling thread. This works but is anti-pattern in the NIO event loop (could deadlock under load if the event loop only has one thread). Test as-is; flag for refactor.
3. **`FilesystemRouteHandler.browse()` returns `Failed to list directory` for permission errors,** which is technically correct but not differentiated from "directory exists but unreadable." Tests will document the current behavior.
4. **`SettingsRouteHandler.update()` returns the *original* posted body in the success response**, not the post-update settings — because the updater is currently a no-op stub (`DeferredSettingsUpdater`). When TKT-009 lands real persistence, this response shape may change. Tests will use the current behavior with a comment.
5. **`BuildRouteHandler.triggerBuild` ignores `projectID` validation** — the closure-now-protocol returns "Project not found" if the projectStore doesn't have it, but the route handler doesn't itself validate the UUID. Acceptable since the router parses the UUID before dispatch ([APIRouter.swift:161-163](RemoteDeploy/API/APIRouter.swift#L161-L163)).
6. **Code coverage not enabled in scheme.** `xcodebuild` defaults to `-enableCodeCoverage NO` unless the scheme has `codeCoverageEnabled = "YES"` or `-enableCodeCoverage YES` is passed. `project.yml` does not set `gatherCoverageData: true`, so the persistent scheme regenerated by xcodegen will keep this off. TKT-003 will fix this in `project.yml`.

### Test approach: handler-direct, not router-through

Two ways to test:
- **(A) Through the router:** build `APIRouterFactory` with mocks, dispatch via `router.handle(_:)`. Used by the existing `APIRouterFactoryTests`. Pros: covers routing + auth + handler integration. Cons: more setup per test, harder to assert internal handler state.
- **(B) Direct on the handler:** instantiate the handler with mocks, call its method directly with a synthetic `APIRequest`. Pros: fast, focused, easy to assert mock call counts. Cons: skips routing/auth.

**Decision: use (B) for the per-handler test files**, keeping the existing (A) tests in `APIRouterFactoryTests` as a thin integration layer. Add a new `APIRouterTests.swift` that focuses specifically on routing/auth/CORS/method-not-allowed concerns at the router level.

This split avoids duplicating coverage and matches the codebase's existing pattern (e.g. `MockProjectStore` is exercised directly in `ProjectStoreTests`, not always through a higher layer).

### Helper: shared `APIRequest` builder

The smoke tests currently embed a `makeRequest(method:uri:body:bearerToken:)` helper inside `APIRouterFactoryTests`. I'll extract this (and the JSONDecoder helper) into a new `APITestSupport.swift` under `RemoteDeployTests/Support/` so all 10 new test files can share it.

### Regression Risk — **LOW**

This ticket is test-only:
- Zero production code changes (the dead `cleanupExpiredTokens()` and the bugs flagged above are out of scope).
- Only `project.yml` gets a one-line `gatherCoverageData: true` addition.
- No existing test files are modified (the existing `APIRouterFactoryTests.swift` smoke tests stay as-is — they're complementary integration tests).
- New test files are in `RemoteDeployTests/` which already builds clean.

Worst case: a new test catches a bug in production code that everyone has been ignoring (e.g. case sensitivity on Authorization header, edge case in path parsing). If so, fix the bug under a separate ticket and keep TKT-003 scoped to test additions.

### Why a `RemoteDeployTests/API/` subdirectory
Keeps API tests grouped and easy to run as a set. Mirrors the production layout (`RemoteDeploy/API/`). Existing tests at the top level are model/service tests; adding a third group of API tests would clutter the directory listing.

### Tradeoffs
- **Speed.** ~82 new tests adds ~1-2 seconds to the unit test run (per the existing per-test averages of 0.001-0.01s). Acceptable.
- **Maintenance.** New tests must be updated when handler signatures change. Low risk because the protocols TKT-004 introduced are stable seams.
- **Coverage > 70% target.** The handlers are simple enough (most are <100 lines, single-concern) that >70% line coverage is easily reached with 5-10 tests each. The existing smoke tests already give us a baseline; this expansion should comfortably hit 85-90% on `RemoteDeploy/API/`.

### Scope deviations from the plan
- **Skipped:** `gatherCoverageData: true` in `project.yml`. xcodegen doesn't have an easy way to set this on auto-generated schemes without defining an explicit `schemes:` block, and adding one risks breaking the existing scheme setup. Decided to use `xcodebuild -enableCodeCoverage YES` on the command line instead, which works without project changes. Acceptance criterion updated to mark this as "use the command-line flag" rather than persisted in project.yml. Could be revisited later if coverage runs become routine.
- **Skipped:** Test-only seam on `PairingRouteHandler` to time-travel pendingTokens for the cleanup test. Decided not to touch production code for the test ticket. Wrote a "fresh tokens not removed" test plus an extensive in-test comment documenting the dead-code situation and pointing at TKT-018.
- **CORS test rewrite:** Initial `test_handle_corsPreflight_returns204` assumed the router handled OPTIONS — it doesn't (CORS lives in `HTTPHandler` at the SwiftNIO layer). Test rewritten as `test_handle_optionsRequestThroughRouter_returns404NotPreflight` documenting the actual layering.
