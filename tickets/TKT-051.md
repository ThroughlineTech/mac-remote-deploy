## Description

RemoteDeploy's build engine (`XcodeBuildEngine`) currently only supports iOS targets — it runs `xcodebuild archive` + `exportArchive` and expects an `.ipa` output. When pointed at a macOS target (e.g., RemoteDeploy itself, or ClaudeDash), the export fails because macOS apps produce `.app` bundles, not IPAs.

Add macOS target support to the build engine so RemoteDeploy can build and serve macOS apps the same way it handles iOS apps. The deploy mechanism for macOS would serve a `.app` bundle (or `.zip`/`.dmg`) for download instead of triggering `itms-services://` OTA install.

This enables using RemoteDeploy + `graceful-relaunch.sh` (TKT-050) as a complete remote build-and-deploy pipeline for macOS apps.


## Acceptance Criteria

- [ ] Build engine detects whether a project targets macOS or iOS and uses the appropriate export method
- [ ] macOS builds produce a downloadable `.app` (zipped) or `.dmg` instead of an IPA
- [ ] macOS build results are served via the existing HTTPS server for download
- [ ] Companion app / web PWA can trigger macOS builds and download the result
- [ ] Existing iOS build flow is unaffected
- [ ] RemoteDeploy can successfully build and serve itself as a macOS target

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Current build pipeline (iOS-only)

The build pipeline flows: **BuildManager** -> **BuildEngineRouter** -> **XcodeBuildEngine** -> **NIODeployServer** HTTP routes.

**XcodeBuildEngine** (`RemoteDeploy/Services/XcodeBuildEngine.swift`):
- `build(project:)` (line 79): runs `xcodebuild archive` (line 110-157), then generates `ExportOptions.plist` (line 169-173), then runs `xcodebuild -exportArchive` (line 176-183), then looks for a `.ipa` file in the export directory (line 186-189), and copies it to `~/Library/Application Support/RemoteDeploy/serve/<slug>/app.ipa` (line 197-200).
- The archive destination is already platform-aware (lines 137-147): it switches on `project.platform.lowercased()` and uses `generic/platform=macOS` for macOS. So archiving already works for macOS.
- The **export step is the blocker**: `xcodebuild -exportArchive` with an ExportOptions.plist produces an IPA for iOS targets. For macOS targets, `-exportArchive` produces a `.app` bundle (or `.pkg` for Mac App Store), not an IPA. The code on line 186 specifically looks for `.ipa` files and throws `exportFailed` when none is found.

**ExportOptions.plist** (`XcodeBuildEngine.generateExportOptionsPlist`, line 522-542):
- Currently generates with `method` (ad-hoc, development) that is iOS-centric. For macOS, the valid export methods are `developer-id`, `mac-application`, or `development`. The current `ad-hoc` method is invalid for macOS and will cause the export to fail.

**BuildStatus** (`RemoteDeploy/Protocols/BuildEngineProtocol.swift`, line 8-12):
- The `.success` case carries `ipaPath: String`. This name is iOS-specific but the value is just a file path -- it works for any artifact path. Renaming would be a large refactor with low value; the name can stay.

**ProjectConfig** (`Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/ProjectConfig.swift`):
- Already has a `platform` field (line 44, defaults to `"iOS"`). The `init(from:)` decoder (line 92) falls back to `"iOS"` for older configs. This field is already sufficient -- no new field needed.
- The `exportMethod` field (line 41, defaults to `"development"`) needs to support macOS-appropriate values.

### HTTP serving layer

**HTTPHandler** (`RemoteDeploy/Services/HTTPHandler.swift`):
- Routes at lines 174-187 serve three sub-paths per slug: `/` (install page), `/manifest.plist`, and `/app.ipa`. For macOS projects, we need to add `/app.zip` (or reuse `app.ipa` path but that's confusing).

**NIOResponseGenerator** (`RemoteDeploy/Services/NIOResponseGenerator.swift`):
- `serveInstallPage` (line 59): generates an install page with an `itms-services://` link pointing at `manifest.plist`. For macOS, this needs to be a direct download link instead.
- `serveManifest` (line 88): generates the OTA manifest plist. Not applicable for macOS.
- `serveIPA` (line 102): serves the IPA file. For macOS, this needs to serve a `.zip` containing the `.app` bundle.

**sendFileResponse** (`HTTPHandler.swift`, line 267-289):
- Hardcodes `Content-Disposition: attachment; filename="app.ipa"`. For macOS, this should be `app.zip`.

**ManifestGenerator** (`RemoteDeploy/Services/ManifestGenerator.swift`):
- Entirely iOS-specific (itms-services OTA manifest). Not needed for macOS.

**InstallPageGenerator** (`RemoteDeploy/Services/InstallPageGenerator.swift`):
- The install template (line 67-131) has an `itms-services://` link. For macOS, this should be a simple `<a href>` download link to the `.zip` file.

### BuildManager and notification flow

**BuildManager** (`RemoteDeploy/Managers/BuildManager.swift`):
- `triggerBuild` (line 107): constructs `installURL` as `serverURL + "/" + project.urlSlug + "/"` (line 151). This works for both platforms -- the install page URL shape is the same.
- The push notification says "ready to install" (line 198). For macOS, "ready to download" would be more accurate, but this is cosmetic.

### BuildResult model

**BuildResult** (`Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/BuildResult.swift`):
- Field `ipaPath` (line 16) stores the artifact path. Works for any file path despite the name.

### What changes in xcodebuild for macOS vs iOS

1. **Archive**: `xcodebuild archive` works identically; just needs `generic/platform=macOS` destination (already handled).
2. **Export**: `xcodebuild -exportArchive` for macOS with method `developer-id` produces a `.app` inside the export directory (not an IPA). With method `mac-application` it produces a `.pkg`. For unsigned dev builds, the `.app` is directly inside the `.xcarchive/Products/Applications/` directory.
3. **Alternative approach**: Skip `-exportArchive` entirely for macOS. After `xcodebuild archive`, the `.app` is at `<archivePath>/Products/Applications/<AppName>.app`. We can zip this directly. This avoids the ExportOptions.plist complexity for macOS and works for development builds without code signing complications.
4. **Destination**: Already handled at lines 137-147 of XcodeBuildEngine.

### How should macOS artifacts be served?

- **Zip the .app bundle**: Simplest and most universal. `ditto -ck --keepParent <App>.app <slug>/app.zip`. Users download, double-click to extract, and drag to Applications. This is the standard distribution method for non-App-Store macOS apps.
- **DMG**: More polished (background image, Applications symlink) but significantly more complex to generate programmatically and overkill for a dev tool.
- **Decision**: Zip. It is what the acceptance criteria suggest, is trivial to generate, and works for the "build and deploy yourself" use case (TKT-050 graceful-relaunch).

### UI changes needed

- The install page for macOS projects should show a "Download" button instead of "Install" with `itms-services://`.
- The manifest.plist route should 404 for macOS projects (or simply not be linked).
- The index page listing does not need changes -- it just links to `/<slug>/`.
- The PWA and companion app trigger builds via the REST API; the API response shape does not change. The companion's "Installs" tab tracks downloads, which already works via `onIPADownload` callback.

### Regression risk assessment

**Low-Medium**. The changes are well-isolated:
- iOS build path is untouched (guarded by platform check).
- The new macOS path reuses the existing archive step and only diverges at export.
- HTTP serving changes add a new route (`app.zip`) and branch the install page template by platform; existing `app.ipa` and `manifest.plist` routes are unchanged.
- The `ProjectConfig.platform` field already exists and defaults to `"iOS"`, so all existing saved configs continue to work.
- Risk area: the `sendFileResponse` function is currently shared; the refactor to support dynamic filenames needs care to avoid breaking the iOS IPA download path.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

### Approach

Branch the build engine's post-archive step by platform: iOS continues through `exportArchive` to produce an IPA; macOS skips export and instead zips the `.app` bundle directly from the archive's `Products/Applications/` directory. The HTTP serving layer branches by platform to serve either the iOS install flow (manifest.plist + itms-services) or a direct download page for the macOS zip.

### Key design decisions

1. **Skip `-exportArchive` for macOS**: The archive already contains the fully built `.app` at `<xcarchive>/Products/Applications/`. Zipping it directly avoids ExportOptions.plist method complications for macOS (developer-id requires notarization, mac-application requires Mac App Store signing). For a dev deployment tool, the archived `.app` is sufficient.

2. **Artifact filename**: macOS artifacts are stored as `app.zip` (parallel to iOS's `app.ipa`) in the same serve directory structure: `~/Library/Application Support/RemoteDeploy/serve/<slug>/app.zip`.

3. **Platform-aware install page**: `InstallPageGenerator` gets a new method (or the existing `generatePage` gains a platform parameter) that renders a "Download" button with a direct link for macOS, vs the existing `itms-services://` button for iOS.

4. **No changes to ProjectConfig model**: The existing `platform` field is sufficient. The `exportMethod` field is ignored for macOS builds (since we skip export).

5. **No changes to BuildStatus/BuildResult models**: The `ipaPath` field name is misleading for macOS but changing it would be a large cross-cutting refactor for no functional benefit. The field stores the artifact path regardless of extension.

### Tradeoffs

- **Pro**: Minimal changes to shared models and API surface. iOS path is completely untouched behind a platform guard.
- **Pro**: Zipping is fast and simple (single `ditto` call). No external tooling needed.
- **Con**: The `.app` from the archive is not notarized or stapled. This is acceptable for a dev tool (the user controls both the build machine and the target machine), but means Gatekeeper will flag the app on first run. Users can `xattr -cr` the extracted app or right-click > Open.
- **Alternative rejected**: Using `-exportArchive` with `method=developer-id` would produce a notarized app, but requires Apple notarization service round-trips (minutes of latency), an active Developer ID certificate, and network access during the build. Too heavy for the "build and serve quickly" use case.


## Implementation Plan
<!-- Agent fills this during /ticket-investigate -->
- [ ] 1. **Branch the build pipeline by platform in XcodeBuildEngine** -- In `RemoteDeploy/Services/XcodeBuildEngine.swift`, in the `build(project:)` method, after the archive step succeeds (after line 164), add a platform check: if `project.platform.lowercased() == "macos"`, skip steps 2-3 (ExportOptions.plist generation and `-exportArchive`) and instead: (a) locate the `.app` bundle inside `<archivePath>/Products/Applications/`, (b) zip it using `ditto -ck --keepParent <App>.app <serveDir>/app.zip`, (c) set the final artifact path to `<serveDir>/app.zip`, and return that path. The existing iOS path (export + IPA copy) remains unchanged inside an `else` branch. Add a new private helper `func zipAppBundle(archivePath: String, serveDir: String) throws -> String` that finds the `.app` in the archive, runs `ditto`, and returns the zip path.

- [ ] 2. **Add `app.zip` route to HTTPHandler** -- In `RemoteDeploy/Services/HTTPHandler.swift`, in the `handleRequest` switch at line 174, add a new case `"app.zip"` that calls a new `serveAppZip(context:project:slug:head:)` method. This is parallel to the existing `"app.ipa"` case.

- [ ] 3. **Implement `serveAppZip` in NIOResponseGenerator** -- In `RemoteDeploy/Services/NIOResponseGenerator.swift`, add a new method `serveAppZip(context:project:slug:head:)` that: (a) constructs the path `<serveRoot>/<slug>/app.zip`, (b) checks the file exists (404 if not), (c) fires `server.onIPADownload` and `server.delegate?.serverDidServeIPA` callbacks (reusing the existing tracking -- rename is not worth the churn), (d) serves the file with `Content-Type: application/zip` and `Content-Disposition: attachment; filename="<project.name>.zip"`. Add a new `sendZipFileResponse(context:filePath:filename:)` method to `HTTPHandler` (or parameterize the existing `sendFileResponse` to accept a filename and content type).

- [ ] 4. **Make `sendFileResponse` support dynamic content type and filename** -- In `RemoteDeploy/Services/HTTPHandler.swift`, refactor `sendFileResponse(context:filePath:)` to accept optional `contentType` and `filename` parameters (defaulting to `"application/octet-stream"` and `"app.ipa"` for backward compatibility). Update the `Content-Disposition` header to use the provided filename. The existing `serveIPA` call site does not need changes due to the defaults.

- [ ] 5. **Add macOS download page to InstallPageGenerator** -- In `RemoteDeploy/Services/InstallPageGenerator.swift`, add a new method `generateDownloadPage(appName:version:build:buildTime:downloadURL:)` that returns HTML with a "Download" button linking directly to the zip URL (no `itms-services://`). Use the same card-style layout as the existing install template. Add a corresponding protocol method to `InstallPageGenerating` in `RemoteDeploy/Protocols/InstallPageGenerating.swift`.

- [ ] 6. **Branch `serveInstallPage` by platform** -- In `RemoteDeploy/Services/NIOResponseGenerator.swift`, in `serveInstallPage(context:project:slug:)` (line 59), check `project.platform.lowercased()`. If `"macos"`, construct a `downloadURL` pointing to `/<slug>/app.zip` and call `server.installPageGenerator.generateDownloadPage(...)`. If iOS (the default), use the existing `generatePage(...)` with `itms-services://`. Similarly, for macOS projects, the `serveManifest` method should return 404 since manifest.plist is not applicable.

- [ ] 7. **Return 404 for manifest.plist on macOS projects** -- In `RemoteDeploy/Services/NIOResponseGenerator.swift`, in `serveManifest(context:project:slug:)`, add an early return with `sendNotFound` if `project.platform.lowercased() == "macos"`.

- [ ] 8. **Add unit tests for macOS build path in XcodeBuildEngine** -- In `RemoteDeployTests/`, add a new test file `XcodeBuildEngineMacOSTests.swift` that: (a) verifies a macOS project config triggers the zip path (mock the ditto process), (b) verifies the returned artifact path ends in `.app.zip`, (c) verifies that an iOS project config still goes through the IPA export path. Use the same mock patterns as `BuildEngineRouterTests.swift` (mock engine with call counters).

- [ ] 9. **Add integration tests for macOS HTTP routes** -- In `RemoteDeployIntegrationTests/HTTPServerIntegrationTests.swift`, add tests: (a) `testMacOSProjectInstallPageShowsDownloadButton` -- register a macOS project, verify the install page contains a download link to `app.zip` and does NOT contain `itms-services://`. (b) `testMacOSManifestReturns404` -- verify `/<slug>/manifest.plist` returns 404 for a macOS project. (c) `testMacOSAppZipDownload` -- place a fake zip in the serve directory, verify `/<slug>/app.zip` returns it with correct headers.

- [ ] 10. **Update the PWA to handle macOS download** -- In `RemoteDeploy/Resources/pwa/`, update the JavaScript that handles build completion to check whether the project platform is macOS and, if so, show a "Download" link instead of an "Install" link. The API already returns `ProjectConfig` with the `platform` field, so the PWA has the data it needs.

- [ ] 11. **Verify self-build scenario** -- Manually test that RemoteDeploy can add itself as a macOS project (scheme: RemoteDeploy, platform: macOS), build itself, and serve the resulting `.app.zip` for download. Verify that the downloaded zip extracts to a working `RemoteDeploy.app`.
