## Description

The menu bar popover shows a spinning, destructive "Cancel Build" button
indefinitely even though the build already finished (the same popover reports
"Build succeeded 5 min ago" and "Installed ..."). Clicking the button does
nothing -- there is no way to clear it short of restarting the app.

Root cause: `MenuBarClient.isBuilding` preferred the live WebSocket status frame
over the authoritative REST poll:

```swift
let state = webSocket.latestStatus?.state ?? status?.buildStatus.state
return state == "building"
```

`WebSocketClient.latestStatus` is only ever set when a `buildstatus` frame
arrives, and is never cleared on disconnect/reconnect. Status broadcasts are
fire-and-forget with no replay on (re)subscribe (see TKT-068), so a client that
is briefly disconnected when the terminal `success`/`failure` frame is sent
never learns the build ended -- `latestStatus.state` stays `"building"`. Because
of the `??`, that stale frame masks the correct polled state (`GET /build/status`
reads `BuildManager` in-process and correctly returns `"success"`) forever.

Why Cancel can't rescue it: clicking it calls `BuildCoordinator.cancelBuild()`,
which reads `buildEngine.status` -- now `.success` -- so it is a no-op and
cannot change the stale WebSocket frame. Nothing the user clicks clears it.

Contrast: the iOS companion does NOT have this bug. Its BuildManager uses a
*stored* `isBuilding` flag that the REST poll resets to false on any terminal
state, with the WebSocket as a pure accelerator -- so a missed frame self-heals
within a poll. The menu bar (rebuilt as an API client in TKT-056 Phase 3)
regressed this by making `isBuilding` a computed property that prefers the WS
with no recovery path.

This is the client-side fix. TKT-068 is the paired server-side hardening that
stops the WebSocket frame from going stale in the first place; either fix alone
unsticks the spinner, but both together close the whole class.

## Reproduction Steps (bugs only)
1. Trigger a build from the menu bar (or any client).
2. While it runs, cause the menu bar's WebSocket to drop and reconnect (server
   HTTPS restart, Tailscale hiccup, or any blip -- a build runs for minutes, and
   reconnect backoff is up to 16s, so the window is wide), such that the terminal
   `success` frame is broadcast while the menu bar socket is down.
3. After the build succeeds, open the popover.
4. Observe: "Cancel Build" spins forever; "Build succeeded N min ago" shows at
   the same time; clicking Cancel does nothing.

## Implementation Plan
- [x] Extract the poll-vs-WebSocket precedence into a pure
      `BuildStateReconciler.isBuilding(polled:live:)` in `RemoteDeployShared`
      (testable from `RemoteDeployTests`, which cannot link `MenuBarClient` --
      `project.yml` deliberately excludes it from the `RemoteDeployServer` test
      host; the companion shares the same reconciliation need).
- [x] Rule: a terminal poll (`success`/`failure`) returns not-building; else
      `(live ?? polled) == "building"`.
- [x] Delegate `MenuBarClient.isBuilding` to it.
- [x] Unit test every poll/live combination, including the exact regression
      (`polled: "success", live: "building"` -> false).

## Regression Report
- Only `isBuilding`'s precedence changed; trigger/cancel paths untouched.
- Minor, self-healing behavior change: at the very start of a *subsequent*
  build, if the poll still reports the previous build's terminal state while the
  WS already reports `"building"`, the button can show "Build & Deploy" for up to
  one poll (<=3s) before catching up. Masked in practice because `triggerBuild`
  force-refreshes the poll immediately, and eliminated entirely once TKT-068
  keeps the WS frame fresh.
