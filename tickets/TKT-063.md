## Description

Clicking "View Build Log" in the menu bar opens the window, but it is
permanently empty and never shows live build output. The web PWA and iOS
companion show the live log fine; only the menu bar is affected.

Root cause: the menu bar connects its WebSocket to the loopback plain-HTTP
listener (`ws://127.0.0.1:8080/api/v1/ws`), but TKT-056 (Phase 3) split the
single listener into HTTPS (:8443) + loopback (:8080) and only installed the
WebSocket upgrader on the HTTPS pipeline. The loopback pipeline used
`configureHTTPServerPipeline()` with no `withServerUpgrade:`, so the upgrade
request fell through to `HTTPHandler`, which returns 401 for `/api/v1/ws`. The
menu bar's `WebSocketClient` then reconnect-looped forever and `buildLogLines`
was never populated. Web/iOS were unaffected because they connect over HTTPS
:8443, which had the upgrader.

Secondary issues found and fixed at the same time:
- `WebSocketClient.openSocket()` reset `buildLogLines = []` on every (re)connect,
  so a transient drop in the backoff loop wiped accumulated output.
- The build-log window only ever showed the live stream, so opening it between
  builds (or on fresh launch) was blank even though the last build's log is
  already persisted in build history.

## Reproduction Steps (bugs only)
1. Launch the menu bar app with the server running.
2. Open the menu bar popover and click "View Build Log".
3. The window opens but stays empty -- even during a live build (no streaming),
   while the same build's log streams correctly to the web PWA.

## Investigation

Two listeners in `NIODeployServer`:
- `bindHTTPSListener` (:8443) installed `NIOWebSocketServerUpgrader` via
  `configureHTTPServerPipeline(withServerUpgrade:)`.
- `bindHTTPListener` (:8080, the loopback surface the menu bar uses) used plain
  `configureHTTPServerPipeline()` -- no upgrader. The upgrade fell through to
  `HTTPHandler` -> 401, which `WebSocketClient.receiveMessages()` treats as a
  handshake failure and retries forever (backoff capped at 16s).

The existing `WebSocketUpgradeTests` only ever connected over `wss://` (the
HTTPS listener), which is why the loopback gap went undetected.

## Proposed Solution

Extract the WebSocket upgrader construction into a shared
`makeWebSocketUpgradeConfig(...)` helper and install it on BOTH listeners so the
pipelines cannot drift again. Auth stays enforced via the same
`webSocketAuthenticator`. Plus: stop wiping the log on reconnect (reset on the
`buildstatus -> building` transition instead), and backfill the window from the
last persisted build when nothing is streaming.

## Implementation Plan
- [x] Add `NIODeployServer.makeWebSocketUpgradeConfig(...)`; call it from both
      `bindHTTPSListener` and `bindHTTPListener`.
- [x] Remove the `buildLogLines = []` reset from `WebSocketClient.openSocket()`;
      reset on the `buildstatus -> building` transition in `handleMessage`.
- [x] Backfill `MenuBarClient.buildLogLines` from `lastBuildResult.buildLog`
      when the live stream is empty and no build is in progress.
- [x] Add a loopback (plain-HTTP) upgrade + broadcast regression test.
