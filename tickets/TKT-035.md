## Description

The Notifications tab in Settings works functionally but looks rough compared to native macOS settings panels. Screenshot shows:

- API key displayed as raw hex in a plain text field with no masking or visual treatment — looks like debug output, not a credential field
- Provider sections (Prowl, Pushover, ntfy) have inconsistent spacing — too much vertical whitespace between the toggle and the key field
- "Send Test" buttons are right-aligned but visually disconnected from the provider they belong to
- The `?` help icon (info popover trigger) floats without clear association to its provider
- "Notify On" checkboxes at the bottom are functionally fine but visually plain — no grouping box or section header treatment matching the provider cards above
- Disabled providers (Pushover, ntfy) show "Send Test" buttons that are grayed out — would be cleaner to hide them entirely when the provider is disabled
- No visual distinction between enabled (Prowl, blue checkbox) and disabled (Pushover/ntfy, gray checkbox) providers — the whole row should look different, not just the checkbox color

### Design direction

Match the style of macOS System Settings (Ventura+):
- Providers as rounded-corner cards with a subtle background tint when enabled
- API key in a `SecureField` (or a toggleable show/hide field) — don't display raw keys by default
- "Send Test" inline within the card, only visible when the provider is enabled
- "Notify On" section as a proper `Form` / `GroupBox` with a header
- Tighter vertical rhythm overall — less dead space between providers

This is a cosmetic enhancement, not a functional fix. Current UI works; it just looks unpolished for a v2.1 app.

## Acceptance Criteria
- [ ] API keys are masked by default (SecureField or similar) with a show/hide toggle
- [ ] Provider sections have a card-like visual treatment (rounded corners, subtle background)
- [ ] Disabled providers are visually muted (not just a gray checkbox — the whole row is de-emphasized)
- [ ] "Send Test" only shows when the provider is enabled
- [ ] "Notify On" section has a GroupBox or equivalent visual grouping
- [ ] Vertical spacing is tightened — no excessive dead space between providers
- [ ] No functional regression: enabling/disabling providers, editing keys, sending tests, and notification triggers all still work
- [ ] `xcodebuild test` passes

## Investigation

### Current layout analysis

The Notifications tab is `PushSettingsTab` in `RemoteDeploy/Views/SettingsView.swift` (lines 275-389). It uses a `ScrollView > VStack` with three `PushProviderSection` instances (Prowl, Pushover, ntfy) separated by `Divider()`, followed by a `GroupBox("Notify On")` section and an error label.

**`PushProviderSection`** (lines 394-438, same file) is a generic reusable view that renders:
- An `HStack` with a `Toggle`, a `?` help button, and a `Send Test` button
- When enabled, the injected `content` (text fields) appears below with `.textFieldStyle(.roundedBorder)`

**Key issues confirmed by code review:**

1. **API keys displayed as plain `TextField`** (lines 295, 307-308, 318-319) — no masking whatsoever. Raw hex/tokens are fully visible.
2. **`Send Test` always visible** (line 425-427) — button is shown even when provider is disabled, just `.disabled(!enabled)`. The ticket wants it hidden entirely.
3. **No card-like visual treatment** — `PushProviderSection` is a plain `VStack` with no background, border, or rounded corners. The only visual separation is `Divider()`.
4. **No visual muting of disabled providers** — only the checkbox color changes; the provider name, help button, and fields all look the same.
5. **`GroupBox("Notify On")` already exists** (line 327) — this acceptance criterion is already partially met. The grouping is there but could use tighter styling.
6. **Spacing** — the outer VStack uses `spacing: 16` which, combined with `Divider()` lines, creates excessive dead space.

**Related file:** `PushNotifSetupStep` (setup wizard, `RemoteDeploy/Views/SetupSteps/PushNotifSetupStep.swift`) uses a *different* pattern — `GroupBox` with `DisclosureGroup`-style expand/collapse. It also uses plain `TextField` for keys. Changes here are out of scope for this ticket (setup wizard is a separate flow), but the setup step already has better visual grouping to reference.

**Model:** `PushNotificationConfig` in `Packages/RemoteDeployShared/Sources/RemoteDeployShared/Models/PushNotificationConfig.swift` — no changes needed. All fields are plain `String`; masking is purely a view concern.

### Regression risk: LOW

All changes are cosmetic within `PushSettingsTab` and `PushProviderSection`. The `syncConfig()` plumbing, `sendTest()` logic, and `PushNotificationConfig` model are untouched. The `.onChange` bindings that drive persistence remain identical. No new state, no new data flow.

## Proposed Solution

Restyle the `PushSettingsTab` and `PushProviderSection` views in `SettingsView.swift` to match macOS Ventura System Settings aesthetics:

1. **Wrap each provider in a styled card** — use `GroupBox` or a custom `RoundedRectangle` background with `.fill(.quaternary)` when enabled, `.fill(.clear)` when disabled.
2. **Replace `TextField` with `SecureField` + show/hide toggle** for API keys / tokens. Use a small eye icon button to toggle between `SecureField` and `TextField`.
3. **Conditionally show `Send Test`** — only render when `enabled` is true (move inside the `if enabled` block).
4. **Visually mute disabled providers** — apply `.opacity(0.5)` to the entire card content (except the toggle itself) when disabled.
5. **Remove `Divider()` separators** — the card backgrounds provide visual separation; dividers become redundant.
6. **Tighten spacing** — reduce outer VStack spacing from 16 to 12.
7. **Keep `GroupBox("Notify On")` as-is** — it already meets the acceptance criterion. Minor tightening of internal spacing if needed.
