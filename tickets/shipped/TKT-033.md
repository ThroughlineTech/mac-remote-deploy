## Description

Two related window-management issues in the menu bar app:

### 1. "Setup Guide" vs "Setup Assistant" — pick one, be consistent

The same window is called different things depending on where you look:

| Location | Name used |
|---|---|
| Menu bar popover button label | "Setup Guide" ([UtilitiesSection.swift:23](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L23)) |
| Window title bar | "Setup Assistant" ([RemoteDeployApp.swift:63](RemoteDeploy/RemoteDeployApp.swift#L63)) |
| Code comments, struct names, source files | "Setup Assistant" (SetupAssistantView.swift, ProjectSetupValidators comments, etc.) |
| AppDelegate log/launch message | "setup assistant" ([AppDelegate.swift:183](RemoteDeploy/AppDelegate.swift#L183)) |

The button says "Setup Guide" but clicking it opens a window titled "Setup Assistant." This is jarring — the user wonders if they opened the right thing. The codebase overwhelmingly uses "Setup Assistant" (it's the struct name, the window ID, the AppDelegate message). The button label is the outlier.

**Fix:** rename the button label from "Setup Guide" to "Setup Assistant" in [UtilitiesSection.swift:23](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L23). One line.

### 2. Windows open behind VS Code (and other apps)

When clicking "Setup Guide" or "Settings..." from the menu bar popover, the window opens but does not come to the front. It appears behind the frontmost app (VS Code, Safari, etc.). The user has to manually click on the RemoteDeploy window in the dock-less app switcher or use Mission Control to find it.

**Root cause:** `LSUIElement = true` apps (no dock icon) don't get automatic window-ordering treatment from AppKit. When a window opens, AppKit doesn't call `NSApp.orderFrontRegardless()` or `NSApp.activate(ignoringOtherApps:)` on the new window — the app stays in the background and the window renders behind whatever's currently focused.

The "Settings..." button has a partial fix: `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })` at [UtilitiesSection.swift:37](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L37). But `NSApp.activate()` alone doesn't always bring windows to front in modern macOS (14+) — Apple deprecated the `ignoringOtherApps:` variant, and the default behavior respects the current activation policy.

The "Setup Guide" button has NO `NSApp.activate()` at all — which is why TKT-032 exists (it needs Settings opened first to "warm up" the activation state).

**Fix options:**
- **A)** Add `NSApp.activate()` to both button actions. Use `NSApp.activate()` (no deprecated flags) + `NSApp.windows.first(where: { $0.title == "Setup Assistant" })?.orderFrontRegardless()` after the window opens.
- **B)** Use `NSApp.setActivationPolicy(.regular)` temporarily when opening a window, then switch back to `.accessory` after the window closes. This makes the app behave like a normal app (dock icon appears) while a window is open, which gets AppKit's full window-ordering treatment.
- **C)** For the Setup Assistant specifically: instead of `openWindow(id:)`, use `NSApp.sendAction(Selector(("showSettingsWindow:")), to: nil, from: nil)` — no wait, that's for Settings only. For custom windows, force-ordering is the right call.

## Investigation

### Issue 1: Naming inconsistency — "Setup Guide" vs "Setup Assistant"

The only user-facing occurrence of "Setup Guide" in Swift code is:

- **`RemoteDeploy/Views/MenuBar/UtilitiesSection.swift:23`** — `Label("Setup Guide", systemImage: "questionmark.circle")`

Every other reference uses "Setup Assistant":

- **`RemoteDeploy/RemoteDeployApp.swift:63`** — `Window("Setup Assistant", id: "setup-assistant")` (the window title)
- **`RemoteDeploy/Views/SetupAssistantView.swift`** — struct/file name
- **`RemoteDeploy/AppDelegate.swift:185`** — notification post to open setup assistant
- **`RemoteDeploy/Views/MenuBar/UtilitiesSection.swift:2`** — code comment says "Setup Guide" (should also be updated for consistency)

Root cause: the button label was written with a different name than the window scene. Simple oversight.

### Issue 2: Windows open behind other apps

There are **four** window-opening paths from the menu bar popover:

1. **"Setup Assistant" button** (`UtilitiesSection.swift:22`) — calls `openWindow(id: "setup-assistant")` with **no** `NSApp.activate()`. This is also the root cause of TKT-032 (window doesn't open at all on cold start).
2. **"Settings..." button** (`UtilitiesSection.swift:30-37`) — uses `SettingsLink` with `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })`. Has activation but the window still opens behind other apps on macOS 14+ because `NSApp.activate()` alone is insufficient for `LSUIElement` apps.
3. **"View Build Log" button** (`BuildControlsSection.swift:59`) — calls `openWindow(id: "build-log")` with **no** `NSApp.activate()`. Same problem as Setup Assistant.
4. **Auto-open at launch** (`MenuBarView.swift:33-34`) — receives `.openSetupAssistant` notification and calls `openWindow(id: "setup-assistant")`. No activation call. The window opens behind whatever app was focused at launch.

Root cause: `LSUIElement = true` apps (no dock icon) do not receive AppKit's standard window-ordering behavior. `openWindow(id:)` creates the window but doesn't bring the app to the foreground. `NSApp.activate()` alone (without `ignoringOtherApps:`) is unreliable on macOS 14+ for accessory-policy apps.

**The reliable fix** is to call `NSApp.activate()` immediately before opening the window, then after a brief async delay, find the window by title and call `orderFrontRegardless()` on it. This two-step approach works because:
- `NSApp.activate()` requests activation from the window server
- `orderFrontRegardless()` forces the specific window above all others regardless of activation state

### Overlap with TKT-032

TKT-032 addresses the Setup Guide button not opening at all on cold start (missing `NSApp.activate()`). This ticket (TKT-033) subsumes that fix — adding `NSApp.activate()` to the Setup Assistant button action fixes both TKT-032's "doesn't open" bug and TKT-033's "opens behind other apps" bug. However, TKT-033 goes further by also adding `orderFrontRegardless()` to ensure the window comes to front, and applies the same fix to the Build Log button.

**If TKT-032 ships first**, its `NSApp.activate()` addition to the Setup Assistant button is a subset of what TKT-033 needs. TKT-033 would then only need to add `orderFrontRegardless()` and fix the Build Log button. Either ordering works.

### Regression risk: Low

- The naming change is a single string literal.
- `NSApp.activate()` is already used for the Settings button; extending it to other buttons is consistent.
- `orderFrontRegardless()` is a standard AppKit call with no side effects beyond window ordering.
- `LSUIElement = true` is preserved; no dock icon appears.

## Proposed Solution

1. **Rename** "Setup Guide" to "Setup Assistant" in the button label and code comment in `UtilitiesSection.swift`.
2. **Extract a helper function** `activateAndOpenWindow(id:)` that calls `NSApp.activate()` then `openWindow(id:)`, then on the next runloop tick finds the window by ID/title and calls `orderFrontRegardless()`.
3. **Apply the helper** to the Setup Assistant button, the Build Log button, and the auto-open-at-launch path in `MenuBarView`.
4. **Settings button** already has `NSApp.activate()` via `simultaneousGesture`; enhance it with an `orderFrontRegardless()` call on the settings window for consistency.
