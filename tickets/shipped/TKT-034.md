## Description

When opening the Setup Assistant from the menu bar popover (via "Setup Guide" / "Setup Assistant"), it opens directly to Step 2 (HTTPS Certificate) with Step 1 (Tailscale) already marked as completed (green checkmark). This happens even on subsequent opens — the assistant never resets to Step 1.

Screenshot confirms: step indicator shows Step 1 with a checkmark, Step 2 highlighted as the current step, and the "HTTPS Certificate" content displayed.

### Likely cause

The `SetupAssistantView` probably persists its `currentStep` state (via `@State`, `@AppStorage`, or `@SceneStorage`) and auto-advances past Step 1 when it detects Tailscale is connected (`appState.tailscaleConnected == true`). So if Tailscale is already connected at the time the window opens, Step 1 is auto-completed and the view jumps to Step 2.

This may be intentional behavior ("skip steps that are already done"), but it's confusing:
- A user clicking "Setup Guide" expects to see the beginning, not a mid-flow state.
- If they want to review or re-do Step 1 (e.g., switch Tailscale accounts), they can't get there without clicking "Back."
- The step indicator shows Step 1 as "completed" even though the user didn't actively complete it — the assistant did it silently.

### Options

**A) Always open to Step 1.** Reset `currentStep = 1` when the Setup Assistant window appears. Let the user see all steps from the beginning and advance manually. Simplest, most predictable. The checkmark on Step 1 can still show "already configured" status without auto-advancing.

**B) Keep auto-advance but explain it.** If the assistant auto-skips completed steps, show a brief message like "Steps 1–2 already configured. Showing the next incomplete step." This preserves the current behavior but makes it transparent.

**C) Open to a summary/overview page** that shows all steps with their status (configured/not configured) and lets the user click into any one. More work but the most user-friendly long-term.

## Reproduction Steps
1. Ensure Tailscale is connected.
2. Launch RemoteDeploy.
3. Open the menu bar popover → click "Setup Guide."
4. Observe: Step 2 (HTTPS Certificate) is displayed, Step 1 has a green checkmark.
5. Expected: Step 1 is displayed, or a summary showing all steps.

## Investigation

### Root cause

The bug is in `SetupAssistantView.swift`, lines 43-50. The `currentStep` `@State` property is initialized from `UserDefaults`:

```swift
private static let savedStepKey = "RemoteDeploy.setupAssistant.currentStep"

@State private var currentStep: SetupStep = {
    let raw = UserDefaults.standard.integer(forKey: SetupAssistantView.savedStepKey)
    return SetupStep(rawValue: raw) ?? .tailscale
}()
```

When the user completes Step 1 and advances to Step 2, `currentStep` is persisted to `UserDefaults` via the `.onChange(of: currentStep)` handler at line 73-76. On the next open of the Setup Assistant, the `@State` initializer reads the saved value and restores directly to Step 2 (`.certificate`, rawValue 1).

This is **not** an auto-advance based on `appState.tailscaleConnected` — it is simpler than the ticket description suspected. The step is persisted in `UserDefaults` and never reset when the window reopens. The `UserDefaults` key is only cleared when the user clicks "Done" on the final step (line 179).

The `Window("Setup Assistant", id: "setup-assistant")` scene in `RemoteDeployApp.swift` (line 63) creates a SwiftUI `Window` whose content is `SetupAssistantView`. SwiftUI `Window` scenes reuse the same view instance across opens/closes, so `@State` retains its value between opens. Even if the view were recreated, the `@State` initializer reads from `UserDefaults`, so the same stale step would be restored.

There is no `onAppear` or `.task` modifier on `SetupAssistantView` that resets `currentStep` when the window appears.

### Why checkmarks show on past steps

The step indicator at lines 82-121 shows a checkmark for any step with `step.rawValue < currentStep.rawValue`. This is purely visual based on the current step index — it does not check whether Tailscale is actually connected or certificates are configured. So if `currentStep` restores to `.certificate` (rawValue 1), Step 1 shows a checkmark regardless of actual state.

### Files examined

- `RemoteDeploy/Views/SetupAssistantView.swift` — `@State currentStep` initialization (line 47-50), UserDefaults persistence (line 73-76), step indicator checkmarks (line 94), UserDefaults removal on Done (line 179)
- `RemoteDeploy/RemoteDeployApp.swift` — Window scene definition (line 63-77)
- `RemoteDeploy/Views/MenuBar/UtilitiesSection.swift` — Setup Guide button (line 22)
- `RemoteDeploy/AppDelegate.swift` — auto-open on empty projects (line 184-186)
- `RemoteDeploy/Views/SetupSteps/TailscaleSetupStep.swift` — no auto-advance logic here

### Regression risk: Low

The fix only changes the step initialization logic in a single view. The step indicator, navigation buttons, and step content routing are all untouched. The UserDefaults persistence mechanism can be simplified without affecting other features.

## Proposed Solution

**Option A from the ticket: always open to Step 1.** This is the simplest and most predictable approach.

Remove the UserDefaults persistence of `currentStep` entirely. The setup assistant is a short 5-step wizard that takes ~2 minutes; there is no real benefit to resuming mid-flow. Always initialize `currentStep = .tailscale` so every open starts from the beginning.

The step indicator checkmarks will still accurately reflect progress as the user advances through the wizard in a single session. Steps that are "already done" (e.g., Tailscale connected) will show their status within the step content (the `TailscaleSetupStep` already shows "Tailscale detected" with a green checkmark when connected), but the wizard will not auto-skip them.

## Implementation Plan
- [ ] 1. In `RemoteDeploy/Views/SetupAssistantView.swift`, remove the `savedStepKey` constant (line 43) and replace the `@State private var currentStep` initializer (lines 47-50) with a simple `@State private var currentStep: SetupStep = .tailscale`.
- [ ] 2. In `RemoteDeploy/Views/SetupAssistantView.swift`, remove the `.onChange(of: currentStep)` modifier (lines 73-76) that persists to UserDefaults.
- [ ] 3. In `RemoteDeploy/Views/SetupAssistantView.swift`, in the "Done" button action (line 179), remove the `UserDefaults.removeObject(forKey: Self.savedStepKey)` call since the key no longer exists.
