## Description

Clicking "Import IPA..." in the menu bar popover opens an `NSOpenPanel` file picker. If the user clicks Cancel (or presses Escape), the file picker closes but sometimes the Settings window or Setup Guide window opens instead of simply returning to the idle state. The wrong window appears intermittently, not every time.

### Suspected root cause (from code inspection)

[RemoteDeploy/Views/MenuBar/UtilitiesSection.swift:52-72](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L52-L72):

```swift
private func importIPA() {
    let panel = NSOpenPanel()
    ...
    guard panel.runModal() == .OK, let url = panel.url else { return }
    ...
}
```

`NSOpenPanel.runModal()` runs a synchronous modal event loop **inside** a SwiftUI `MenuBarExtra` popover button action. This is a known problematic combination:

1. The user clicks "Import IPA..." → the button action fires synchronously.
2. `runModal()` opens the file picker as a modal window. The `MenuBarExtra` popover loses key-window status and likely **dismisses** (standard popover behavior when it loses focus).
3. The modal run loop now owns the event queue. While running, AppKit may deliver stale or misrouted events.
4. The user clicks Cancel → `runModal()` returns `.cancel` → `guard` exits early.
5. AppKit re-activates the RemoteDeploy app (the modal is dismissed, the app comes to front).
6. On re-activation, one of two things can happen:
   - **`SettingsLink`'s `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })`** at [UtilitiesSection.swift:37](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L37) has a gesture recognizer that may fire on the re-activation event, opening the Settings window.
   - **SwiftUI's re-evaluation of the view tree** during app re-activation may re-trigger a button action from the stale touch-down event, hitting either "Setup Guide" (`openWindow(id: "setup-assistant")`) or "Settings..." depending on where the touch coordinates land relative to the now-ghosted popover layout.

The `SettingsLink` with `.simultaneousGesture(TapGesture())` is particularly suspicious — `.simultaneousGesture` doesn't replace the default gesture but runs alongside it, so any activation event that SwiftUI interprets as a tap on the SettingsLink area could open Settings.

### Two likely fixes

**Fix A (minimal):** Defer the `NSOpenPanel` call to the next runloop turn via `DispatchQueue.main.async { ... }`. This lets the button's click handler return synchronously, the popover dismiss cleanly, and the modal open on a fresh event-loop iteration with no stale gesture state.

**Fix B (SwiftUI-native):** Replace the `NSOpenPanel.runModal()` call with SwiftUI's `.fileImporter(isPresented:allowedContentTypes:onCompletion:)` modifier, which manages the file-picker lifecycle through SwiftUI's own presentation system rather than a modal run loop. This avoids the popover-dismiss/re-activate race entirely. More invasive but more correct long-term.

## Reproduction Steps
1. Launch RemoteDeploy.
2. Open the menu bar popover.
3. Click "Import IPA..."
4. In the file picker, click Cancel.
5. Observe: sometimes the Settings window or Setup Guide opens instead of returning to idle.
6. Repeat steps 2–5 a few times — the bug is intermittent but reproducible within ~5 attempts.

## Investigation

### Root cause confirmed

The bug originates in `RemoteDeploy/Views/MenuBar/UtilitiesSection.swift:52-59`. The `importIPA()` function calls `NSOpenPanel.runModal()` synchronously inside a SwiftUI button action within a `MenuBarExtra` popover (style: `.window`).

**The interaction chain:**

1. User clicks "Import IPA..." button (line 14) — the button's `action: importIPA` fires synchronously.
2. `importIPA()` at line 53 creates an `NSOpenPanel` and calls `panel.runModal()` at line 59.
3. `runModal()` opens a system-level modal dialog. This causes the `MenuBarExtra` popover to lose key-window status and dismiss (standard AppKit behavior — popovers dismiss when they lose focus).
4. The modal run loop now owns the event queue. The SwiftUI view tree for the popover is still technically alive but its window is gone.
5. User clicks Cancel → `runModal()` returns `.cancel` → the `guard` at line 59 exits early.
6. AppKit re-activates the RemoteDeploy app (the modal dismissed, app comes to front).

**Why Settings or Setup Guide opens:**

There are two `SettingsLink` views with `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })`:
- `UtilitiesSection.swift:37` — the "Settings..." button, directly below "Import IPA..."
- `ProjectsListSection.swift:40` — the "Add Project..." button

The `.simultaneousGesture(TapGesture())` modifier installs a gesture recognizer that runs **alongside** the default `SettingsLink` gesture. When `runModal()` returns and AppKit re-activates the app, SwiftUI may re-evaluate the view tree. If the activation event is interpreted as a tap within the `SettingsLink` hit area, the Settings window opens. Similarly, the "Setup Guide" button at line 22 (`openWindow(id: "setup-assistant")`) can fire if the stale touch-down coordinates from the original "Import IPA..." click land within its hit region after the popover layout is invalidated.

The bug is intermittent because it depends on exact timing of:
- Whether the popover dismisses before or after the modal captures the event queue
- Whether AppKit's re-activation event lands on a gesture recognizer
- The position of the click relative to the popover's (now-invalidated) layout

**Other NSOpenPanel.runModal() usages in the codebase** (not affected):
- `ProjectFormView.swift:150` — runs inside a Settings window, not a popover. No dismiss race.
- `CertificateSetupStep.swift:210` — runs inside the Setup Assistant window. No dismiss race.
- `ProjectSetupStep.swift:201` — runs inside the Setup Assistant window. No dismiss race.

These are safe because they're in regular windows that don't auto-dismiss on focus loss.

### Regression risk: **Low**

The fix is isolated to a single function in a single file. The import flow itself (IPA copying, slug resolution, build manager state update) is untouched. The only change is *when* the NSOpenPanel runs relative to the button action's return.

## Proposed Solution

**Fix A (minimal, recommended): Defer `NSOpenPanel` to the next runloop turn.**

Wrap the entire `importIPA()` body in `DispatchQueue.main.async { ... }`. This lets the button action return synchronously, the popover dismiss cleanly on its own schedule, and the file picker open on a fresh event-loop iteration with no stale gesture state or active popover layout.

**Why Fix A over Fix B (.fileImporter):**
- `.fileImporter` must be attached as a modifier to a view in the hierarchy. In a `MenuBarExtra` popover that dismisses when focus changes, this creates the same class of lifecycle problems — the modifier's presentation state may outlive the popover.
- Fix A is a one-line wrapper around existing, working code. Zero API surface change.
- Fix A matches the pattern already used elsewhere in the codebase (e.g., `AppDelegate` uses `DispatchQueue.main.async` to defer startup work past SwiftUI's first layout pass — see `RemoteDeployApp.swift:35-37` comment).

**Tradeoffs:**
- Fix A introduces a one-frame delay before the file picker appears. Imperceptible to users.
- The `NSOpenPanel.runModal()` pattern remains (not SwiftUI-native), but it's the same pattern used in three other places in the codebase and is correct when not running inside a popover's synchronous button action.
