## Description

Phase 5 of the backend-decoupling plan (docs/backend-decoupling-plan.md). The web
PWA reaches feature parity with the menu bar for projects and settings: it can
create / edit / delete projects (picking a path + scheme via the filesystem
endpoints) and edit server settings -- not just browse and build. The REST API
already exposes everything needed; this is frontend buildout on the PWA.

Two server-visible correctness issues surfaced while building the write surface
and are fixed here (see Design):
1. Settings GET redacts secrets but PUT validated them literally, so any
   write client (incl. the menu bar after Phase 3) breaks settings on save.
2. The PWA's inline event handlers are blocked by the document CSP, so the
   browser was non-interactive (Phase 4's browser ship gate was still pending).

## Design

### Server: settings sentinel preservation (SettingsRouteHandler)
GET /api/v1/settings redacts certPath/keyPath to "[configured]" and the push
secrets to "[redacted]". PUT decodes a full SettingsData and validates cert/key
paths exist on disk. A client that round-trips settings (the menu bar's
`applySettings` read-modify-write, and now the PWA editor) therefore sends the
sentinels back, which either 500s ("[configured]" not found) or stores
"[redacted]" as a real secret.

Fix: before validating/applying a PUT, the handler reads the current settings and
replaces any incoming field still equal to its sentinel with the stored real
value. The sentinel constants are shared between get() (redaction) and update()
(un-redaction), and the PUT response is re-redacted so it matches GET's contract
and never echoes a real secret back over the wire. This is the correct central
fix: it makes GET/PUT round-trippable for every client and removes a latent
menu-bar settings bug from Phase 3.

### PWA: CSP-clean event delegation
The document CSP is `default-src 'self'; style-src 'unsafe-inline'; connect-src
'self' wss: ws:`. With no `script-src 'unsafe-inline'`, inline event handler
attributes (onclick/onsubmit/onchange) are blocked. The PWA is refactored to a
single delegated listener set on the persistent #app element using
data-click / data-change / data-input / data-submit attributes mapped to an
ACTIONS table. No inline handlers remain and the CSP is unchanged (not loosened),
which keeps us aligned with the Phase 7 hardening direction.

### PWA: projects + settings buildout
The PWA is intentionally dependency-free vanilla JS. New UI is split into two
extra global scripts (projectform.js, settingsform.js) to keep each file small;
all share the global `state`, `api()`, and helpers from app.js.

- Project form mirrors macOS ProjectFormView: path (filesystem browser),
  name, scheme (detect via /filesystem/schemes on the chosen .xcodeproj),
  bundle ID, team ID, provisioning profile, configuration, export method,
  platform, macOS local-deploy toggle/path, URL slug. Validators mirror
  ProjectSetupValidators / BundleIDValidator (reverse-DNS bundle ID, 10-char
  uppercase team ID, required name + scheme).
- Filesystem browser navigates under /Users/, selecting a .xcodeproj sets the
  project path + project file and triggers scheme detection.
- Settings editor edits port, hostname, cert/key paths, and the full push
  config; redacted fields are shown as their sentinel and preserved on save
  unless the user types a new value.

## Gotchas
- crypto.randomUUID() supplies the new project's id (PWA is HTTPS-only after
  pairing, so it is available); Swift's UUID decoder accepts the standard form.
- /filesystem/schemes expects the full `.xcodeproj` path, not the directory.
- Settings PUT requires a complete SettingsData; the editor always sends the
  whole object (with sentinels preserved server-side).
- Resources/pwa/** is a bundled folder reference, so new .js files are served
  after a rebuild without xcodeproj surgery.

  constants; GET/PUT now use `redacted(_:)`; PUT restores placeholder-valued
