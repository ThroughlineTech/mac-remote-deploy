## Description

TKT-043 fixed the companion Build tab getting stuck in the "BuildingÔÇª" state for successful/failed builds that actually reached the compile phase ÔÇö by making `BuildControlView` / `BuildLogStreamView` observe `BuildManager` and `WebSocketClient` as `@EnvironmentObject` directly, so terminal `buildstatus` WebSocket frames re-render the view in-place.

But there's a second, related failure mode that TKT-043's fix may not cover: **xcodebuild preflight errors that fail the archive before any compile work begins.**

Concrete example observed 2026-04-11: a project was misconfigured in the companion's project store with `scheme=RemoteDeploy` + `destination=generic/platform=iOS`. The RemoteDeploy scheme is macOS-only, so `xcodebuild archive` failed immediately with:

```
xcodebuild: error: Unable to find a destination matching the provided destination specifier:
    { generic:1, platform:iOS }
```

This happens *before* xcodebuild starts the "Command line invocation" log streaming that the companion observes. In the companion Build tab during this failure, the symptom matches the original TKT-043 bug: "BuildingÔÇª 63s" with the elapsed counter still ticking, log area showing only the initial line (or nothing), button stuck disabled. The Prowl notification *may or may not* fire depending on how the host's `BuildEngine` and notification plumbing handle pre-build failures.

The question this ticket needs to answer: **does the host emit a terminal `buildstatus` frame (state=`failure`) when xcodebuild exits non-zero at the destination-specifier / scheme-validation stage, before any output is produced?** If yes, TKT-043's fix should already handle it and this ticket can be closed as already-fixed. If no, there's a gap in the host's `XcodeBuildEngine` error-path signaling, and the companion's Build tab will still get stuck in the `BuildingÔÇª` state forever for this class of failure.

## Reproduction Steps (bugs only)

1. In RemoteDeploy (Mac app) ÔåÆ Settings ÔåÆ Projects, create or edit a project entry with a deliberately invalid scheme/destination combination:
   - Scheme: `RemoteDeploy` (a macOS-only scheme)
   - Platform/Destination: iOS
2. Open the companion app, go to the **Build** tab, select this project
3. Tap **Build & Deploy**
4. Observe the xcodebuild invocation fail at the destination check within a second or two (no log output, no compile phase)
5. Check the companion Build tab:
   - Expected: transitions to a failure state with the error message, button re-enabled, timer stopped
   - Observed (hypothesis): stays in "BuildingÔÇª Ns" indefinitely, log area empty, button stuck disabled ÔÇö same symptom as the original TKT-043 bug

## Acceptance Criteria

- [ ] When `xcodebuild archive` fails at preflight (destination mismatch, invalid scheme, missing DEVELOPMENT_TEAM, etc.), the companion Build tab transitions out of the "BuildingÔÇª" state
- [ ] The failure is visible to the user on the companion ÔÇö either a specific error message from xcodebuild's stderr, or a generic "Build failed to start" indicator with the exit code
- [ ] The elapsed-time ticker stops at the final duration
- [ ] The Build & Deploy button becomes re-enabled
- [ ] The fix handles at least these preflight failure modes:
  - Invalid destination (scheme/platform mismatch)
  - Scheme does not exist in the project
  - Project file cannot be opened
- [ ] The same preflight failure also produces a Prowl (and any other configured push) notification that matches the on-screen error message, so the UI and notification channels stay consistent

## Investigation

### End-to-end chain for a preflight failure (by code inspection)

When `xcodebuild archive` is invoked with a scheme/destination mismatch, it exits non-zero almost immediately (typically within 1ÔÇô2 seconds, before any compile work). Tracing how that exit propagates through the stack:

1. **[RemoteDeploy/Services/XcodeBuildEngine.swift:273](RemoteDeploy/Services/XcodeBuildEngine.swift#L273) ÔÇö `runXcodebuild`.** The process is launched with stdout/stderr pipes and readability handlers that call `emitLog` for every incoming line. On termination ([XcodeBuildEngine.swift:321-329](RemoteDeploy/Services/XcodeBuildEngine.swift#L321-L329)):
   ```swift
   } else if proc.terminationStatus != 0 {
       let msg = "xcodebuild exited with code \(proc.terminationStatus)"
       self?.setStatus(.failure(error: msg))
       continuation.resume(throwing: BuildError.xcodebuildFailed(proc.terminationStatus, msg))
   }
   ```
   The engine **does** detect the non-zero exit, sets its local `.failure` status, and throws `BuildError.xcodebuildFailed`. The throw propagates up through [`XcodeBuildEngine.build(project:)`](RemoteDeploy/Services/XcodeBuildEngine.swift#L146).

2. **[RemoteDeploy/Managers/BuildManager.swift:196-226](RemoteDeploy/Managers/BuildManager.swift#L196-L226) ÔÇö `triggerBuild` `catch` block.** The thrown error is caught:
   ```swift
   } catch {
       let endTime = Date()
       logTask.cancel()
       setBuildStatus(.failure(error: error.localizedDescription))
       ...
       notificationManager.notifyBuildFailure(...)
       await sendPushNotification?("Build Failed", ..., .high, nil)
   }
   ```
   `setBuildStatus` at [BuildManager.swift:89-92](RemoteDeploy/Managers/BuildManager.swift#L89-L92) both assigns `@Published var buildStatus` **and** calls `buildEventBroadcaster?.broadcastBuildStatus(new)`. So a terminal `.failure` frame is broadcast automatically on every error-path transition.

3. **[RemoteDeploy/Services/NIODeployServer+BuildEventBroadcasting.swift:21-28](RemoteDeploy/Services/NIODeployServer+BuildEventBroadcasting.swift#L21-L28) ÔÇö `broadcastBuildStatus`.** Maps the internal `BuildStatus.failure(error:)` to `BuildStatusInfo(state: "failure", message: error)`, JSON-encodes it, and calls `webSocketManager.broadcast(type: "buildstatus", payload: json)`. All subscribed WebSocket clients receive the terminal frame.

4. **[RemoteDeployCompanion/Services/BuildManager.swift:109-134](RemoteDeployCompanion/Services/BuildManager.swift#L109-L134) ÔÇö `observeWebSocketStatus`.** The companion's sink explicitly handles both terminal states:
   ```swift
   if status.state == "success" || status.state == "failure" {
       self.isBuilding = false
       self.pollTask?.cancel()
   }
   ```

5. **[RemoteDeployCompanion/Views/Build/BuildControlView.swift](RemoteDeployCompanion/Views/Build/BuildControlView.swift) ÔÇö TKT-043's fix.** With `@EnvironmentObject var buildManager: BuildManager`, the Build tab now reacts directly to the `isBuilding = false` assignment above and re-renders in-place.

### Conclusion from code inspection

**The chain is correct.** Every link from `xcodebuild` exiting non-zero at preflight to the companion Build tab re-rendering exists and has a sensible implementation. TKT-043's fix should, in theory, cover this failure mode: the same `buildstatus` WebSocket frame that unsticks the UI for compile-phase failures will also unstick it for preflight failures, because it's the same broadcast path on the host side.

**Existing test coverage confirms the BuildManager side.** [RemoteDeployTests/BuildManagerBroadcastTests.swift:152](RemoteDeployTests/BuildManagerBroadcastTests.swift#L152) ÔÇö `test_triggerBuild_failurePath_broadcastsBuildingThenFailure` ÔÇö stubs `MockBuildEngine.buildShouldThrow` with a synthetic `LocalizedError` and verifies the broadcaster receives a terminal `.failure(...)` call. This proves the BuildManager catch ÔåÆ broadcast path works for *any* thrown error, regardless of source. A real `XcodeBuildEngine.xcodebuildFailed` throw is just another `LocalizedError` through the same codepath.

### What's NOT currently tested

- No integration test launches a **real** `XcodeBuildEngine` against a bad scheme/destination and asserts that `build(project:)` throws `BuildError.xcodebuildFailed` with a non-zero exit code. The cancellation tests ([RemoteDeployTests/XcodeBuildEngineCancelTests.swift](RemoteDeployTests/XcodeBuildEngineCancelTests.swift)) cover the signal/kill path but not the exit-status propagation path for a naturally-failing invocation.
- No end-to-end test reproduces: "launch real xcodebuild with bogus args ÔåÆ engine throws ÔåÆ BuildManager broadcasts `.failure` on the live `NIODeployServer` ÔåÆ subscribed WS client sees the terminal frame." [RemoteDeployIntegrationTests/BuildManagerWebSocketIntegrationTests.swift](RemoteDeployIntegrationTests/BuildManagerWebSocketIntegrationTests.swift) (extended by TKT-043) covers the WS broadcast shape using a `server.broadcastBuildStatus(.failure(error: "code sign error"))` call directly, not a real build.

### Secondary issues found while tracing the chain

1. **Error message is not useful for the user.** When xcodebuild exits non-zero, the only information reaching the companion is `BuildError.xcodebuildFailed(code, "xcodebuild exited with code N")`. The actual xcodebuild stderr lines (`xcodebuild: error: Unable to find a destination matching the provided destination specifier: { generic:1, platform:iOS }`) are streamed to the log via `emitLog("[stderr] ...")` and will appear in `buildLog`, but the short error string that ends up in the `.failure` status / push notification / Prowl title is generic. From a UX standpoint, the push notification will say "xcodebuild failed (exit 70): xcodebuild exited with code 70" ÔÇö unhelpful. The engine should capture the last handful of stderr lines and include them in the thrown error's message.

2. **Potential stderr-ordering race.** At [XcodeBuildEngine.swift:295-301](RemoteDeploy/Services/XcodeBuildEngine.swift#L295-L301), the stderr `readabilityHandler` is a closure that fires on an unspecified queue. The `terminationHandler` ([line 304](RemoteDeploy/Services/XcodeBuildEngine.swift#L304)) clears the handlers ([lines 306-307](RemoteDeploy/Services/XcodeBuildEngine.swift#L306-L307)) ÔÇö which could theoretically race a final stderr read-out still in flight and lose the last stderr line. For a preflight failure where *all* the interesting stderr comes out right before the exit, this is a plausible source of a "missing error message" symptom. Not a stuck-UI bug per se, but worth fixing at the same time.

3. **Log stream finish timing.** `BuildManager.triggerBuild` at [line 150/198](RemoteDeploy/Managers/BuildManager.swift#L198) calls `logTask.cancel()` before `setBuildStatus(.failure(...))`. Since `logTask` runs on `@MainActor` same as the catch block, the cancel happens synchronously, so in-flight `broadcaster.broadcastBuildLog(line)` calls that were already executing in the task's iteration are NOT stopped mid-call. The cancel just stops *further* iterations. The `for await` loop sees `Task.isCancelled` at the next iteration boundary and exits. For preflight failures where the last stderr line is the informative error, there's a small window where it might be emitted but not consumed before `logTask.cancel()` fires. Low risk, worth a defensive `continuation.finish()` on the engine's log continuation in the error termination path to flush pending lines deterministically.

### Regression Risk

**Low-to-medium.** The "fix" for the primary bug (if it manifests) is additive ÔÇö better error messages, explicit log-stream draining on error, and test coverage. None of it changes the happy path or the existing broadcast shape. The only subtle area is stderr capture: a regression here would affect error messaging for all xcodebuild failures, not just preflight ones, so a misstep would be broadly noticeable and caught quickly.


## Proposed Solution

**Phase A ÔÇö verify the primary bug exists.** Before writing any code, reproduce the exact preflight scenario on a real device with the post-TKT-043 companion. If the Build tab transitions correctly, Phase A is the whole fix: close this ticket as verified and only land the Phase B improvements as their own small ticket (or skip entirely).

**Phase B ÔÇö regardless of A, ship the secondary improvements.** These are independent of whether the primary bug manifests:

1. **Capture stderr tail in the thrown error message.** Give `XcodeBuildEngine` a small ring-buffer (last 8 lines) that the stderr readability handler appends to as it streams. When the termination handler sees a non-zero exit, include the ring-buffer contents in `BuildError.xcodebuildFailed`'s message. The downstream companion and Prowl notification will then show the actual xcodebuild error line ("Unable to find a destination matchingÔÇª") instead of "exit code 70".

2. **Deterministic log-stream drain on error.** In the error branch of [BuildManager.swift:196-226](RemoteDeploy/Managers/BuildManager.swift#L196-L226), before `logTask.cancel()`, explicitly finish the engine's log stream so any in-flight stderr lines flush to the broadcaster before the task exits.

3. **Integration-test coverage for real xcodebuild preflight failure.** Add one test under `RemoteDeployTests/XcodeBuildEngineCancelTests.swift` (renaming the file is out of scope) or a new `XcodeBuildEngineFailureTests.swift` that runs a real `xcodebuild archive` against `/Applications/Xcode.app/Contents/Developer/../not-a-real-project.xcodeproj` (or a scheme name that doesn't exist in the current project) and asserts:
   - The call throws `BuildError.xcodebuildFailed`
   - The thrown error's message contains a substring of actual xcodebuild stderr output (proving the stderr ring-buffer is populated)

4. **Companion acceptance test (optional but valuable).** Add a UI-style test in `RemoteDeployCompanionUITests` that exercises the stuck-preflight scenario end-to-end. Requires the `--auto-pair` launch hook TKT-044 introduced. Worth considering but not strictly necessary for this ticket ÔÇö the host-side test coverage plus TKT-043's wiring fix is enough to guarantee the behavior in principle.

**Why this order:** the biggest risk here is spending implementation effort on a bug that doesn't exist. Phase A is a 10-minute device check that either closes the ticket or converts it into a much more specific bug report. Phase B is worth doing either way because it improves observability and fills a genuine test gap ÔÇö but it's cheap, scoped, and uncontroversial.

**Alternatives considered:**

- **Rewrite the engine to expose stderr separately from the log stream.** Could make stderr handling more explicit and testable. Rejected as overkill ÔÇö the existing single-stream + "[stderr] " prefix approach is good enough, and changing it would touch every existing test.
- **Have `XcodeBuildEngine` emit `.failure` proactively on throw (vs. letting BuildManager do it).** Rejected ÔÇö the engine's `setStatus(.failure(...))` is internal; only BuildManager's call to `buildEventBroadcaster.broadcastBuildStatus` fans out to subscribers. Keeping a single broadcast site (BuildManager) is the right single responsibility boundary.

**Phase B implementation (host-side):**

- `RemoteDeploy/Services/XcodeBuildEngine.swift`
  - Added `lockedRecentStderrLines` ring buffer (`OSAllocatedUnfairLock<[String]>`, capacity 8) and `stderrRingCapacity` constant (B1).
