## Description

The General tab in Settings has a single "Launch at Login" checkbox centered in a large empty window. It looks unfinished — like a placeholder page that never got filled in. Screenshot confirms: just the checkbox, massive whitespace above and below.

### Options

**A) Add more settings to General.** Candidates that could live here:
- HTTP port (currently hardcoded to 8080, now configurable per TKT-028 — could be user-facing)
- HTTPS port (currently in the Server tab — could consolidate)
- Tailscale polling interval (currently hardcoded to 30s in AppDelegate — TKT-030 will reduce it, but making it user-configurable is an option)
- Auto-start server on launch (currently always starts if certs are configured)
- Theme / appearance (if ever relevant)
- Reset / clear all data button

**B) Merge General into another tab.** "Launch at Login" could move to the Server tab (it's about server lifecycle). Then delete the General tab entirely. Fewer tabs = less clutter.

**C) Redesign the layout.** Keep General but use a `Form` with grouped sections and descriptive text so it doesn't look empty even with one setting. Add a version/build number display at the bottom (currently only on the About tab). Even one setting can look intentional if the layout gives it visual weight.

**Recommendation:** Option B (merge into Server) or Option C (redesign with visual weight). Option A risks bloating a tab with settings that don't belong together just to fill space.

## Reproduction Steps
1. Open Settings.
2. Click the "General" tab.
3. Observe: a single checkbox in a vast empty window.

## Acceptance Criteria
- [ ] The General tab either (a) has enough content to justify its existence with a professional layout, or (b) its sole setting is merged into another tab and General is removed.
- [ ] "Launch at Login" functionality is preserved regardless of where it lives.
- [ ] Settings window doesn't feel like it has a placeholder tab.
- [ ] `xcodebuild test` passes.

## Investigation

### Current state

All settings tabs live in a single file: `RemoteDeploy/Views/SettingsView.swift`.

The `GeneralSettingsTab` struct (lines 443-482) contains exactly one control: a `Toggle("Launch at Login")` inside a `Form` with `.padding()`. The form's frame is inherited from the parent `TabView` which is `.frame(width: 560, height: 450)` (line 52). Result: a single checkbox floating in 560x450 of whitespace.

### Tab inventory (SettingsView, lines 16-53)

| Tab | Content | Visual density |
|-----|---------|---------------|
| Server | Port, Hostname, Cert, Key fields in a `Form` | Medium (TKT-037 plans to polish) |
| Projects | List + add/edit/delete | Good |
| Notifications | 3 provider sections + event toggles in `ScrollView` | Dense (TKT-035 plans to polish) |
| Devices | Paired devices list | Good |
| **General** | **Single toggle** | **Empty** |
| About | Icon, version, links, credits | Good |

### Option analysis

**Option A (add more settings):** The ticket lists candidates (HTTP port, polling interval, auto-start server, reset data). But HTTP port and auto-start are server-lifecycle concerns that belong on the Server tab. Polling interval is an implementation detail. Adding unrelated settings just to fill space would create a junk-drawer tab.

**Option B (merge into Server tab, delete General):** "Launch at Login" is arguably a server-lifecycle setting (the app is a server). Moving it to the Server tab eliminates the empty-tab problem entirely. However, TKT-037 is already planning a full redesign of the Server tab layout with grouped sections. Adding the toggle there is natural — it fits a "Startup" or "Server Lifecycle" section. But this creates a dependency: if TKT-037 is implemented first, we know exactly where the toggle lands. If TKT-036 goes first, we just append it to the existing Server form and TKT-037 will reorganize it.

**Option C (redesign layout):** Keep General but add visual weight — grouped `Form` sections, descriptive text, version info. This is lipstick on a pig. One setting doesn't warrant its own tab no matter how pretty the layout is.

**Recommendation: Option B.** Merge "Launch at Login" into the Server tab and remove the General tab entirely. This is the cleanest solution: fewer tabs, no empty-feeling pages, and the toggle lands in a conceptually appropriate place.

### Regression risk: Low

- The `GeneralSettingsTab` struct is self-contained (lines 443-482). It has no external callers besides the `TabView` in `SettingsView.body`.
- The `SMAppService` login-item logic moves verbatim; no behavioral change.
- Removing a tab from `TabView` cannot break other tabs.
- TKT-039 deals with tab selection (opening Settings to a specific tab). If TKT-039 references the General tab by index, removing it would shift indices. But TKT-039 is about opening to Projects, not General, so the risk is minimal. We should verify TKT-039's approach doesn't use a hardcoded tab index that assumes General exists.

## Proposed Solution

**Merge the "Launch at Login" toggle from GeneralSettingsTab into ServerSettingsTab, then delete GeneralSettingsTab and its tab entry.**

The toggle will be placed at the bottom of the Server tab's `Form`, separated by a `Divider()`, as a standalone "Startup" control. This is the minimal change. If TKT-037 later redesigns the Server tab into grouped sections, it can move the toggle into a "Startup" `GroupBox`.
