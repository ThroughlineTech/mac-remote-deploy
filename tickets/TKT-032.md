## Description

Clicking "Setup Guide" in the menu bar popover does nothing on first attempt. However, if the user clicks "Settings..." first (which opens the Settings window), and THEN clicks "Setup Guide," the Setup Guide window opens normally. After that first successful open, subsequent clicks on "Setup Guide" also work — until the app is relaunched, at which point the cycle repeats.

This suggests the Setup Guide window requires some activation state that "Settings..." establishes as a side effect.

### Suspected root cause (from code inspection)

"Setup Guide" uses `openWindow(id: "setup-assistant")` at [UtilitiesSection.swift:22](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L22):

```swift
Button(action: { openWindow(id: "setup-assistant") }) {
    Label("Setup Guide", systemImage: "questionmark.circle")
}
```

`openWindow(id:)` is a SwiftUI environment action (`@Environment(\.openWindow)`) that tells the `WindowGroup` or `Window` scene with that ID to open. In a `MenuBarExtra` app with `LSUIElement = true` (no dock icon), SwiftUI windows don't render until the app has been "activated" at least once via `NSApp.activate()` or equivalent.

"Settings..." uses `SettingsLink` with `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })` at [UtilitiesSection.swift:30-37](RemoteDeploy/Views/MenuBar/UtilitiesSection.swift#L30-L37):

```swift
SettingsLink {
    ...
}
.buttonStyle(.plain)
.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })
```

The `NSApp.activate()` call is what makes the app "activated" in AppKit's window lifecycle. Once activated, subsequent `openWindow(id:)` calls work because the app's window server connection is now warm.

**The fix**: the "Setup Guide" button also needs `NSApp.activate()` before calling `openWindow(id:)`, same as the `SettingsLink` does — OR the button action should call `NSApp.activate()` then dispatch `openWindow` to the next runloop turn:

```swift
Button(action: {
    NSApp.activate()
    openWindow(id: "setup-assistant")
}) { ... }
```

This is the same pattern documented in Apple's `MenuBarExtra` → `Window` interop guidance for `LSUIElement` apps.

## Reproduction Steps
1. Launch RemoteDeploy fresh (cold start, not just re-click the menu bar icon).
2. Open the menu bar popover.
3. Click "Setup Guide" — nothing happens.
4. Click "Settings..." — the Settings window opens.
5. Close the Settings window.
6. Click "Setup Guide" again — it now works.
7. Force-quit and relaunch → "Setup Guide" is broken again until Settings is opened first.

## Investigation

### Root cause confirmed

The bug is exactly as described in the ticket description. In `RemoteDeploy/Views/MenuBar/UtilitiesSection.swift`:

- **Line 22**: The "Setup Guide" button calls `openWindow(id: "setup-assistant")` with no `NSApp.activate()`.
- **Lines 30-37**: The "Settings..." `SettingsLink` includes `.simultaneousGesture(TapGesture().onEnded { NSApp.activate() })`.

In an `LSUIElement = true` app (no dock icon), SwiftUI's `openWindow(id:)` environment action requires the app to have been "activated" at least once via `NSApp.activate()` for the window server to render windows. Until that first activation, `openWindow` silently does nothing.

The Settings button's `.simultaneousGesture` call to `NSApp.activate()` is what "warms up" the activation state. Once activated, subsequent `openWindow` calls work for the remainder of the app's lifecycle — explaining why Setup Guide works after Settings has been opened once, but fails on a cold launch.

### Secondary code path

`RemoteDeploy/Views/MenuBarView.swift:33-34` has an `.onReceive` handler for `.openSetupAssistant` that also calls `openWindow(id: "setup-assistant")` without `NSApp.activate()`. This fires at launch when no projects are configured (triggered from `AppDelegate.swift:185`). This path has the same bug but is a separate concern — at launch, the app may already be activating, and this auto-open path is less user-visible. Fixing it here would be a bonus but is not required by the acceptance criteria.

### Window scene declaration

`RemoteDeploy/RemoteDeployApp.swift:63-77` declares `Window("Setup Assistant", id: "setup-assistant")` — this is correctly configured. The scene exists; it just doesn't render because `openWindow` is a no-op when the app isn't activated.

### Regression risk: Low

The change is adding a single `NSApp.activate()` call before an existing `openWindow` call — the exact same pattern already used by the Settings button on line 37. This is a well-tested AppKit API with no side effects beyond bringing the app to the foreground (which is the desired behavior). No layout changes, no new state, no new dependencies.

**Note on TKT-033 overlap:** TKT-033 also touches `UtilitiesSection.swift` (renaming the button label from "Setup Guide" to "Setup Assistant" and adding window-ordering logic). Both tickets modify the same button action on line 22. However, the changes are independent and non-conflicting — TKT-032 adds `NSApp.activate()` to the action closure, TKT-033 renames the label text and may add `orderFrontRegardless()`. Either can land first; the second will just need to account for the slightly modified line.

## Proposed Solution

Add `NSApp.activate()` to the Setup Guide button's action closure, immediately before the `openWindow(id:)` call. This mirrors the pattern already used by the SettingsLink on lines 36-37 of the same file.

The simplest correct fix is:

```swift
Button(action: {
    NSApp.activate()
    openWindow(id: "setup-assistant")
}) {
    Label("Setup Guide", systemImage: "questionmark.circle")
        ...
}
```

No `DispatchQueue` or runloop deferral is needed — `NSApp.activate()` is synchronous and sets the activation state immediately, so the subsequent `openWindow` call benefits from it in the same runloop turn. This matches the SettingsLink pattern already in production.
