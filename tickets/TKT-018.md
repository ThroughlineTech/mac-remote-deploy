## Description
Pairing tokens were intentionally shortened from 256 bits → 32 bits (8 hex chars) to support typeable short codes in the TokenEntryView flow and to work around iOS simulator limitations (no camera for QR scanning, no paste into XCUITests). 32 bits = ~4 billion possibilities, which is only defensible with rate limiting.

This ticket adds the compensating controls and revisits whether we can restore higher entropy for the simulator path.

### Context
- [JSONPairedDeviceStore.swift:generateToken()](RemoteDeploy/Services/JSONPairedDeviceStore.swift) now produces 8-char hex tokens
- [TokenEntryView in ServerDiscoveryView.swift](RemoteDeployCompanion/Views/Discovery/ServerDiscoveryView.swift) asks users to type this code
- [SECURITY_AUDIT.md](docs/SECURITY_AUDIT.md) accepted risk #10 ("no rate limiting") specifically *because* the token was high-entropy; that assumption no longer holds
- Pairing is still HTTPS-only (`rejectPairingOverHTTP = true` restored in [NIODeployServer.swift](RemoteDeploy/Services/NIODeployServer.swift))
- Pairing tokens still expire 10 minutes after generation

### Work items
1. **Rate limit `/api/v1/pair`** — target: 1 request per second per source IP, 10 failed attempts within 10 minutes → 1-hour lockout for that IP. Applies to POST only. Add to `PairingRouteHandler` or as middleware in `APIRouter`.
2. **Revisit simulator screenshot pairing** — the real reason for the short token is that XCUITest can't easily inject a long token. But the `--auto-pair` launch-argument path added in [ConnectionManager.swift](RemoteDeployCompanion/Services/ConnectionManager.swift) already bypasses the pair endpoint entirely by constructing `APIClient` directly. Can we lean into that and keep long tokens for real pairing? Investigate:
   - Making `--auto-pair` the sanctioned screenshot path (documented in `scripts/capture-screenshots.sh`)
   - Restoring 256-bit tokens for real user pairing flows
   - If short codes are still wanted for the "type the code from your Mac" UX even without the simulator constraint, treat that as a separate UX decision and document it
3. **Update [SECURITY_AUDIT.md](docs/SECURITY_AUDIT.md)** — document whichever posture we land on: either "restored 256-bit + rate limiting as defense in depth" or "accepted 32-bit + rate limiting as primary defense." Either way, update risk #10 to reflect the current state.
4. **Restore the non-Tailscale LAN TokenEntryView flow if desired** — with `rejectPairingOverHTTP` restored, tapping a Bonjour-discovered server and entering a code over plain HTTP now 403s. Options:
   - Use `https://<localIP>:8443` with a trust exception for local IPs in the iOS `CertValidatingSessionDelegate`
   - Accept that non-Tailscale LAN pairing is not supported; remove or disable the TokenEntryView path when only HTTP is available
   - Use a temporary HTTPS listener bound to the local IP
